Automated Phishing Triage with Bit-Flag Scoring

A production N8N workflow that ingests Microsoft 365 phishing alerts from Wazuh, enriches IPs, domains, and file hashes against VirusTotal and AbuseIPDB in parallel, scores each alert with a 9-signal bit-flag system, and notifies only when the score actually warrants it.

The Use Case

Microsoft 365 Defender catches a phishing email. It records the verdict, the authentication failures, the sender IP, the attachment hashes, the recipient. Wazuh pulls those audit logs in and fires rule 91556. That fires a webhook to our N8N instance. So far, so good.

The problem starts at the next step. The raw Wazuh alert is dense — five hundred lines of JSON with most of it irrelevant. A human analyst handling this manually would:

  1. Open the Microsoft Defender portal to see the email details
  2. Copy the sender IP and paste into VirusTotal
  3. Copy the sender domain and paste into VirusTotal
  4. If there's an attachment, copy the SHA-256 into VirusTotal too
  5. Check AbuseIPDB for the IP
  6. Read the SPF/DKIM/DMARC results in the alert
  7. Decide whether to notify the user, the SOC team, both, or neither
  8. If notifying, write an email with all the relevant context

That's 5-8 minutes per alert for something that fires dozens of times a day. The work is mechanical — the analyst is just shuttling data between four browser tabs — but the judgement call at step 7 is the bottleneck. Most of these phishing alerts are already handled by M365 (the email went to the junk folder), so notifying the user is noise. Some, though, slipped through. Knowing which is which without doing the lookup work is what makes triage slow.

I wanted a system that did all the mechanical work automatically, scored the result against a clear rubric, and only bothered an analyst when the score said it was worth bothering them.

The architectural decisions worth calling out: filtering happens at every stage, the four enrichment APIs run in parallel, and scoring uses a bit-flag system so the resulting number is a unique fingerprint of which signals fired. We'll come back to each.

Stage 1 — Filter at the Source

Two filter nodes drop alerts that aren't worth processing. The first checks the Wazuh rule ID; the second checks the Microsoft 365 DeliveryAction field.

Rule 91556 is the specific Wazuh rule for "Office 365: Phishing and malware events from Exchange Online Protection and Microsoft Defender for Office 365." We only care about that — not Exchange admin events, not mailbox login events. Drop everything else before it touches the rest of the pipeline.

The DeliveryAction filter is more interesting. The relevant values are:

  • Delivered — the email landed in the recipient's inbox
  • DeliveredAsSpam — it landed in the junk folder
  • Blocked — M365 dropped it entirely
  • Quarantined — held for admin review

I process the first two and ignore the rest. If M365 has already blocked or quarantined the email, the user can't interact with it — no notification needed. But if it reached either the inbox or the junk folder, the user might still see it and click something. Those alerts deserve enrichment and scoring.

The filter accepts either path — webhook delivery ($json.body.all_fields…) or a Wazuh re-indexed shape ($json._source.data…) — because the same workflow gets triggered from both sources. That dual-path handling shows up everywhere; it's the next stage's whole job.

Stage 2 — Artifact Extraction (Two JSON Shapes, One Code Block)

The Wazuh alert is shaped differently depending on whether the webhook fires from the live integration or from a re-played message in the Wazuh indexer. Same fields, different parent paths. Rather than build two parsing nodes and a merge, one Code node handles both shapes.

📄 Extract Artifacts — Code node 
const item = $input.first().json;

// Dynamic path selection — handles both webhook and re-index shapes
const rawData = (item.body?.all_fields?.data)
              ? item.body.all_fields.data.office365
              : (item._source?.data)
              ? item._source.data.office365
              : null;

if (!rawData) {
    throw new Error("Could not find Office365 data in expected JSON paths.");
}

// Defensive extraction — never crash on missing fields
const ip_address = rawData.SenderIp || "N/A";
const senderEmail = rawData.P2Sender || rawData.SenderAddress;
let domain_name = "N/A";
if (senderEmail?.includes("@")) {
    domain_name = senderEmail.split("@")[1];
}

// Attachments come as an array — flatten what we need
const attachments = (rawData.AttachmentData || []).map(file => ({
    name:      file.FileName || "Unknown",
    extension: file.FileType || "Unknown",
    hash:      file.SHA256   || null
}));

// AuthDetails is an array of {Name, Value} — turn it into a lookup
const authList = rawData.AuthDetails || [];
const getAuthStatus = (name) => {
    const found = authList.find(i => i.Name === name);
    return found ? found.Value : "N/A";
};

return {
    source:            "MS365",
    target_ip:         ip_address,
    target_domain:     domain_name,
    sender_mail:       senderEmail || "Unknown Sender",
    receiver_mail:     Array.isArray(rawData.Recipients)
                         ? rawData.Recipients.join(", ")
                         : rawData.Recipients || "Unknown",
    email_subject:     rawData.Subject || "No Subject",
    delivery_location: rawData.LatestDeliveryLocation || "Unknown",
    delivery_action:   rawData.DeliveryAction || "Unknown",
    message_timestamp: rawData.MessageTime || "N/A",
    detection_method:  rawData.DetectionMethod || "Unknown",
    m365_verdict:      rawData.Verdict || "Unknown",
    attachments:       attachments,
    auth_details: {
        spf:   getAuthStatus("SPF"),
        dkim:  getAuthStatus("DKIM"),
        dmarc: getAuthStatus("DMARC")
    }
};

Three details that earn their keep:

  • Optional chaining everywhere. Wazuh alerts don't always carry every field, and the workflow has to survive partial data. Every property access uses ?. and every value has an || fallback. The node throws exactly one error — when neither root path exists — and every other missing field becomes "N/A" downstream.
  • The AuthDetails transformation. M365 returns SPF/DKIM/DMARC as an array of {Name, Value} objects, not a dictionary. The getAuthStatus helper turns array lookups into property accesses for the rest of the pipeline.
  • Output shape is intentionally flat. Every downstream node references $json.target_ip, not $json.body.all_fields.data.office365.SenderIp. The Code node is also the schema-flattening node.

Stage 3 — Parallel Enrichment Across Four APIs

The extracted artifacts fan out to four HTTP nodes that run simultaneously:

  • VirusTotal IP lookup — returns last-analysis stats (how many engines flagged it), the AS owner, and country
  • VirusTotal Domain lookup — returns last-analysis stats on the sender domain
  • VirusTotal File Hash lookup — for the first attachment (if any), returns engine detection counts
  • AbuseIPDB lookup — returns a 0-100 confidence score for the IP, plus its usage type

These all happen in parallel because N8N fans out arrays of outgoing connections to nodes that run concurrently. Four sequential API calls would take about 1.6 seconds in total. Four parallel calls take ~400 ms — the time of the slowest single call. That difference matters when the SOC receives 60 alerts in a burst.

A 4-input Merge node downstream waits for all four to complete, then passes a single combined item to the scorer. If any of the enrichment calls fails (VT rate limit, AbuseIPDB hiccup), the workflow uses onError: continueRegularOutput so the rest of the pipeline still runs with a missing enrichment instead of dying. A missing VT result just means that signal scores zero — which is the safe default.

Free-tier API rate limits matter here. VirusTotal's free public API allows 4 requests per minute. If the SOC gets a flood of phishing alerts, we'd burn through that quota immediately. In production we use the paid VT tier (1,000 req/min) — but the workflow's error handling means a hit limit doesn't break the pipeline, it just means that one alert is scored without VT data.

Stage 4 — Bit-Flag Threat Scoring

This is the core of the workflow. Every interesting design decision lives here.

The naive approach to "score a phishing alert" is to add up weighted signals — SPF failure is worth 2 points, DKIM failure is worth 5, malicious IP is worth 10, and so on. The problem with that is the resulting score is lossy: a score of 17 could mean "DKIM fail + malicious IP" or "SPF fail + something else + something else." You can't reconstruct which signals fired from the score alone.

The bit-flag approach fixes that. Every signal gets a unique power-of-two value:

▸ The 9 bit flags 
const BIT = {
    M365_SPAM:        1,    //  2^0
    DMARC_FAIL:       2,    //  2^1
    SPF_FAIL:         4,    //  2^2
    DKIM_FAIL:        8,    //  2^3
    IP_MALICIOUS:     16,   //  2^4
    DOMAIN_MALICIOUS: 32,   //  2^5
    ABUSEIPDB:        64,   //  2^6
    EXEC_ATTACHMENT:  128,  //  2^7
    M365_MALWARE:     256   //  2^8
};

Each signal sets its bit if triggered. Final threat_score is the sum. Because the values are powers of two, the resulting score uniquely encodes which signals fired. A score of 25 (= 16 + 8 + 1) tells me unambiguously that this alert had a malicious IP, a DKIM failure, and an M365 spam verdict.

Here's the full signal grid, ordered by weight:

Bit 01
M365 Spam Verdict
Microsoft tagged the email as Spam. The lowest-weight signal — M365 already handled it.
Bit 12
DMARC Fail
DMARC check failed, "none", or missing. Strong spoofing signal.
Bit 24
SPF Fail
Sending server not in the domain's authorised list. Spoofing indicator.
Bit 38
DKIM Fail
Message signature missing or invalid. Content tampering or impersonation.
Bit 416
IP Malicious
VirusTotal ≥3 engines flagged the sender IP, or AbuseIPDB ≥50.
Bit 532
Domain Malicious
VirusTotal ≥3 engines flagged the sender domain, or urlscan returned malicious.
Bit 664
AbuseIPDB ≥50
High-confidence abuse report independent of VirusTotal. Belt-and-suspenders.
Bit 7128

An .exe / .bat / .ps1 / .js / .vbs etc. attached. Massive escalator on its own.
Bit 8256
M365 Malware Verdict
Microsoft itself called it malware. Highest weight — independent expert system agrees.

The weighting is deliberately non-linear. An executable attachment alone (128) is worth more than DMARC + SPF + DKIM combined (2 + 4 + 8 = 14). That matches reality: a single failing auth check is common and not always alarming, but an unprompted .exe in an inbound email is almost never legitimate.

The scoring code

▸ Threat score calculation 
let threat_score = 0;
const flags_triggered = [];

if (isM365Spam)         { threat_score += BIT.M365_SPAM;        flags_triggered.push('M365 Spam'); }
if (dmarc_failed)        { threat_score += BIT.DMARC_FAIL;       flags_triggered.push('DMARC Fail'); }
if (spf_failed)          { threat_score += BIT.SPF_FAIL;         flags_triggered.push('SPF Fail'); }
if (dkim_failed)         { threat_score += BIT.DKIM_FAIL;        flags_triggered.push('DKIM Fail'); }
if (ipMalicious)         { threat_score += BIT.IP_MALICIOUS;     flags_triggered.push('IP Malicious'); }
if (domMalicious)        { threat_score += BIT.DOMAIN_MALICIOUS; flags_triggered.push('Domain Malicious'); }
if (abuseTriggered)      { threat_score += BIT.ABUSEIPDB;        flags_triggered.push('AbuseIPDB ≥50'); }
if (isExecAttachment)    { threat_score += BIT.EXEC_ATTACHMENT;  flags_triggered.push('Executable Attachment'); }
if (isM365Malware)       { threat_score += BIT.M365_MALWARE;     flags_triggered.push('M365 Malware'); }

The flags_triggered array is also produced — handy for auditing, for sending in the analyst email, and for any future ML training data that wants to learn which signal combinations actually correlated with real incidents.

Risk Tiers and the Tier-Based Percentage

The raw score lives in the range 0-511 (sum of all 9 bits). That's not a human-friendly scale. Risk tiers map ranges of scores to human labels, and a tier-based percentage maps them to 0-100% for the visual progress bar in the email.

Why tier-based instead of linear? Because the relationship between threat signals and actual risk is non-linear. A score of 16 (just a malicious IP) is genuinely much worse than a score of 14 (all three auth checks failing on a benign sender). A linear mapping would put them within 1% of each other; the tier system puts the 16 into "High Risk" (≥50%) and the 14 into "Medium Risk" (~46%) — and that's the right call.

▸ Tier-based percentage function 
function calcRiskPercent(score) {
    if (score === 0)  return 0;
    if (score >= 128) return Math.min(Math.round(75 + ((score - 128) / (511 - 128)) * 25), 100);
    if (score >= 16)  return Math.round(50 + ((score - 16)  / (127 - 16)) * 24);
    if (score >= 8)   return Math.round(25 + ((score - 8)   / (15  - 8))  * 24);
    /* 1-7 */         return Math.round(1  + ((score - 1)   / (7   - 1))  * 23);
}

Within each tier, the percentage scales linearly. Between tiers, it jumps. The result looks natural in the email's progress bar without flattening important distinctions.

The Notify Decision

An If node downstream of the scorer checks should_notify === true, which the scorer sets when threat_score >= 8. Anything below that — a phishing email M365 already handled cleanly, with no auth failures or bad reputation — never reaches a human. That's the whole point.

This is where the noise reduction happens. In a typical week, about 60% of M365 phishing alerts score below 8 — meaning M365 caught them, the auth checks passed, and the sender IP/domain were clean. Those don't deserve a notification. The remaining 40% — anything M365 caught but with a real signal of compromise — do. The platform sends those, and only those.

Stage 5 — The Styled Email

When the scorer says "notify," an HTML node renders a styled email and an Outlook node sends it. The template is intentionally Outlook-compatible — table-based layout, inline styles, no flexbox or grid. Outlook for Windows still renders email like it's 2007, and "looks fine in Gmail" is not the same thing as "looks fine in the wild."

Three details from the template are worth pulling out.

Defanged IPs and domains

Email clients aggressively auto-link any URL or IP they see. Showing a known-malicious IP as a clickable link in a security warning email is a bad idea. The template displays them defanged:

▸ Defang helpers 
function defangIP(ip) {
    if (!ip || ip === 'N/A') return ip;
    const lastDot = ip.lastIndexOf('.');
    return ip.substring(0, lastDot) + '[.]' + ip.substring(lastDot + 1);
}

function defangDomain(domain) {
    if (!domain || domain === 'N/A') return domain;
    const lastDot = domain.lastIndexOf('.');
    return domain.substring(0, lastDot) + '[.]' + domain.substring(lastDot + 1);
}

So 198.51.100.4 becomes 198.51.100[.]4, and example.org becomes example[.]org. Outlook leaves those alone. The reader sees the value clearly but can't accidentally click into it.

Plain-English authentication statements

Instead of just showing "SPF: Fail / DKIM: Fail", the scorer also generates a human-readable explanation of what the combination means:

▸ Authentication statement builder 
let auth_statement = '';

if (spf_failed && dkim_failed) {
    auth_statement = 'SPF & DKIM both FAILED — this email cannot be trusted. '
                   + 'The sending server is not authorised for this domain and the message '
                   + 'signature is invalid, making spoofing or impersonation highly likely.';
} else if (dkim_failed) {
    auth_statement = 'DKIM FAILED — sender identity unverified. The message\'s digital '
                   + 'signature is missing or invalid, which means the content may have been '
                   + 'tampered with or the domain is being impersonated.';
} else if (spf_failed) {
    auth_statement = 'SPF FAILED — unauthorised sending server. The mail server that sent '
                   + 'this email is not listed as an approved sender for this domain — '
                   + 'a strong indicator of spoofing.';
} else if (allAuthPass_dmarc_none) {
    auth_statement = 'SPF & DKIM passed and sender reputation is currently clean — however, '
                   + 'this domain has no DMARC policy published (DMARC: None). Without DMARC '
                   + 'enforcement, spoofing from this domain cannot be blocked even if SPF/DKIM pass.';
} else {
    auth_statement = 'SPF & DKIM passed — but this does not confirm legitimacy. Other threat '
                   + 'signals are present; treat this email as high risk and verify the sender '
                   + 'independently.';
}

Non-technical recipients won't know what DKIM is. They will understand "this email's identity cannot be verified, treat it as suspicious." The statement is rendered inline in the email so the explanation always travels with the alert.

Conditional attachment section

The template only renders the attachment block if there's actually an attachment, and the styling adapts based on whether the attachment was flagged by VirusTotal (red), is a suspicious executable type (amber), or just an ordinary document (neutral). One template, three visual states.

A Real Alert, Scored End to End

Here's an alert that came through last week. A spoofed-domain phishing email targeting a finance user, delivered to the junk folder by M365.

Alert #1 — Phish to Junk

Sender IP: 198.51.100.4 (Mailchimp ASN) Sender domain: titan-awards[.]example Subject: "Deadline Tonight — Full Results Friday" DeliveryAction: DeliveredAsSpam (in junk folder) M365 Verdict: Phish DetectionMethod: Spoof external domain SPF: Fail → flag SPF_FAIL (+4) DKIM: Fail → flag DKIM_FAIL (+8) DMARC: Fail → flag DMARC_FAIL (+2) VT IP detections: 0 → no flag (Mailchimp infrastructure) VT domain detections: 0 → no flag (new domain, not yet scanned) AbuseIPDB confidence: 2 → no flag (well below 50) Attachments: none → no flag M365 Verdict: Phish (not "spam", not "malware") → no flag ───────────────────────────────── threat_score = 4 + 8 + 2 = 14 tier = Medium Risk risk_percent = 25 + ((14-8) / 7) * 24 ≈ 46% should_notify = TRUE

An analyst receives a Medium Risk notification with the email details, the failed-auth explanation, and the recommendation to verify through a separate channel. The recipient gets a heads-up that something landed in their junk folder that's actively impersonating a domain — which is exactly the signal a phishing-aware user needs to keep clicking "Mark as spam" instead of "Mark as not spam".

Contrast that with another alert from the same batch: M365 caught a legitimate marketing email and mistakenly tagged it as phishing. SPF passed, DKIM passed, DMARC was set to "none," the sender domain was clean across VT and AbuseIPDB, no attachment, M365 verdict was "Spam" (not "Phish"). Score: 1 + 2 = 3. Below threshold, no notification — exactly right.

Lessons from Six Months of Running This

1. Bit-flag scoring beat weighted-sum scoring

I tried weighted-sum scoring first. It worked, but every time I wanted to tune thresholds or audit a decision, I had to rebuild the logic from the score backwards. Bit-flags removed that entire problem. The score is the audit trail.

2. The dual-path JSON parser saved me a refactor

About a month in, we added a second source (Wazuh re-indexing for back-fill) and I expected to need a new pipeline. Because the extraction was written for two paths from day one, all I had to do was add the new source's webhook URL. No code changes.

3. Outlook compatibility is not optional

The first version of the email template used flexbox. It looked perfect in Gmail and in the Outlook preview. In actual Outlook for Windows it rendered as a vertical list of broken boxes. I rewrote the template using HTML tables (1990s style) and never had a layout bug since.

4. Parallel enrichment was the easiest win

Switching the four lookups from sequential to parallel was a 30-second change in N8N (drag four output connections from one node instead of chaining them) and shaved over a second off every alert's processing time. I should have done it on day one instead of day forty.

Beyond Phishing — Where the Pattern Applies

Bit-flag scoring + parallel enrichment isn't a phishing-specific pattern

  • Endpoint alert prioritization. EDR alerts have similar shape: a verdict from the tool, a hash, a process tree, network connections. Score them with bit-flags (signed binary? known LOLBin? unsigned + network call? credential dumping API?) and you get the same auditable triage system.
  • Authentication anomaly scoring. Failed MFA challenge, IP not seen in 90 days, impossible travel, geolocation mismatch, unusual user-agent. Each one a bit. Score and tier.
  • Customer support ticket triage. Bit-flags for paying customer, mentioned a competitor, mentioned "lawyer", attached a screenshot, in the founder's circle. Different domain, same engineering.
  • Fraud detection on transactions. Different currency than usual, IP from anonymizing proxy, unusual amount, brand-new account. Score them and route the medium tier to humans, the very-high tier to auto-block.
  • Any multi-signal decision system where you want a single number to convey the verdict and stay reversible into the input signals that produced it.

A good threat score isn't just a number. It's a fingerprint of which signals fired — visible at a glance, auditable in retrospect.
