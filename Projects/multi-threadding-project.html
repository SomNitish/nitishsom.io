How It Works — End to End

Four containers, one network, two background services, and a single-file Python API serving a vanilla-JS dashboard. Runs on one Ubuntu host. Designed so any one piece can be understood (and debugged) on its own.

Highlights — Engineering Decisions That Shaped It

The technical choices that made the difference between a script and a system. Each one is explained in depth in its blog part, but here's the summary.

🔢

Atomic incident IDs via Postgres trigger

Lifetime per-customer counter using ON CONFLICT DO UPDATE. Zero duplicates under any concurrency, with row-level locking measured in microseconds.

📈

60-line bash autoscaler

Polls Redis queue depth, scales workers 10→50, asymmetric cooldown to prevent thrashing. Beats Kubernetes HPA for single-host operational simplicity.

🛡️

Deny-only permission overrides

Roles define the ceiling. Per-user permission overrides can only restrict, never elevate. Privilege escalation through misconfiguration is structurally impossible.

🔐

Single-session, same-browser refresh

One session per user, but logging in from the same browser extends the existing session instead of replacing it. Same security as enterprise SSO, in 30 lines of Python.

🚨

Queue-stuck overlay with auto-drafted email

When workers stop processing, the dashboard goes red and pre-fills an emergency email for the on-call engineer to copy. Cut response time from 3.5 hours to 2 minutes.

🔁

Fire-and-forget Jira sync

Endpoints return in 50ms while Jira calls happen in a background thread. Nightly reconciliation script catches anything that fell out of sync during Atlassian outages.

🪵

Append-only audit log by convention

Every state change recorded. No UPDATE or DELETE on the table — anywhere in the codebase. The trail survives any UI bug or admin mistake.

📦

Multi-tenant isolation from day one

Per-customer incident sequences, per-customer Jira projects, per-customer dashboards. Adding a third tenant takes one row in incident_seq and one config block.
