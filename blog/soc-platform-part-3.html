The Database That Stops Duplicate IDs

Part 2 gave us 20 parallel workers — and a brand-new bug: two workers inserting at the same millisecond produced the same incident ID. This part fixes it the right way: in the database, with a trigger, in code older and more tested than anything I could write.

The Bug That Showed Up After Part 2

End of Part 2: queue mode running, 10-50 workers auto-scaling, alerts flowing through in 3 seconds. The throughput problem was solved. So I shipped it.

Three days later, an analyst pinged me on Slack:

"Hey, I just got two emails about CLIENT-A-IR-062026-487 — different alerts, different IPs, same incident ID. Is that supposed to happen?"

It was not supposed to happen. I opened Postgres and ran a count:

▸ peek at the damage 
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c "
  SELECT incident_id, COUNT(*) FROM incidents
  GROUP BY incident_id
  HAVING COUNT(*) > 1
  ORDER BY COUNT(*) DESC;
"

Eleven duplicates over three days. About 5% of the alerts that arrived in close pairs. The bug had been there since day one — single-threaded N8N just hid it by serializing everything. Add parallel workers and the cracks appeared.

How the Race Window Opens

My first ID generator was a few lines of JavaScript inside an N8N Function node. Read the last sequence number, increment, insert. The pattern looks innocent:

▸ the broken approach (do NOT do this) 
// Inside an N8N Function node
const ym  = new Date().toISOString().slice(0,7).replace('-','');  // 062026
const cust = $input.first().json.customer || 'CLIENT-A';

// Step 1: read the last sequence
const last = await pg.query(
  "SELECT last_seq FROM incident_seq WHERE customer=$1",
  [cust]
).rows[0]?.last_seq || 0;

const next = last + 1;

// Step 2: write it back
await pg.query(
  "UPDATE incident_seq SET last_seq=$1 WHERE customer=$2",
  [next, cust]
);

// Step 3: format the ID
const incidentId = `${cust}-IR-${ym}-${String(next).padStart(3,'0')}`;

The bug lives between Step 1 and Step 2. Two workers running this code at the same millisecond:

T+0 ms Worker A: reads last_seq = 486 T+1 ms Worker B: reads last_seq = 486 (Worker A hasn't written yet!) T+5 ms Worker A: UPDATE last_seq = 487 T+6 ms Worker B: UPDATE last_seq = 487 (overwrites with same value) T+7 ms Worker A: generates ID CLIENT-A-IR-062026-487 T+8 ms Worker B: generates ID CLIENT-A-IR-062026-487 DUPLICATE

Any "read, compute, write" pattern in application code has this exact bug. You can wrap the JS in a transaction, you can add a Redis lock, you can write a distributed coordinator. All of those are inventions on top of a problem the database already solved fifty years ago.

The right fix isn't smarter application code. The right fix is to let the database be the source of truth.

The ID Format I Wanted

Before fixing the trigger, I need to be precise about what the IDs should look like. The format is four parts:

CLIENT-A-IR-062026-487 tenant   type   month-year (MMYYYY)   sequence

One detail that took me a while to get right: the sequence number is continuous per customer — it never resets. Only the month-year in the middle changes. So if June ends at 509, the first incident of July is:

CLIENT-A-IR-072026-510 July's first incident continues the count

Why continuous instead of monthly reset? Two reasons:

  • Customer reports. Customers reference incident IDs in tickets and audits. A monthly reset means IR-062026-001 and IR-072026-001 exist as separate incidents — confusing. Continuous numbering means every ID is unique forever.
  • Forensics. The sequence number alone tells you the chronological order across the customer's history. "Incident 487 happened before incident 510" is obvious. Compare that to "incident 062026-23 and 072026-04" — you have to mentally translate.

The Schema, Two Tables

Slim. We're solving exactly one problem: atomic ID generation. Just two tables.

The incidents table

This is the table the trigger fires on. Each row is one security incident.

📄 schema.sql — incidents 
CREATE TABLE IF NOT EXISTS incidents (
  seq                    SERIAL PRIMARY KEY,
  incident_id            TEXT,                  -- generated by trigger
  customer               TEXT NOT NULL DEFAULT 'CLIENT-A',
  alert_name             TEXT NOT NULL,
  severity               TEXT NOT NULL DEFAULT 'MEDIUM',
  category               TEXT NOT NULL DEFAULT 'Alert',
  log_source             TEXT DEFAULT '',
  status                 TEXT NOT NULL DEFAULT 'open'
                         CHECK (status IN ('open','closed','reopened','resolved')),
  assigned_to            TEXT DEFAULT '',
  logged_at              TEXT DEFAULT '',

  -- email notification state
  email_sent             BOOLEAN NOT NULL DEFAULT FALSE,
  email_sent_at          TIMESTAMPTZ,

  -- jira link
  jira_id                TEXT DEFAULT '',
  jira_key               TEXT DEFAULT '',
  jira_url               TEXT DEFAULT '',

  -- timeline
  created_at             TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  resolved_at            TIMESTAMPTZ,
  assigned_at            TIMESTAMPTZ,
  alert_generated_at     TIMESTAMPTZ,

  -- pre-computed metrics (seconds)
  mttd_seconds           INTEGER,
  mtta_seconds           INTEGER,
  mttr_seconds           INTEGER,

  remarks                TEXT DEFAULT '',
  is_manual              BOOLEAN NOT NULL DEFAULT FALSE,
  created_by             TEXT DEFAULT ''
);

Two things worth pointing out:

  • seq is the real primary key, not incident_id. seq is an integer the database manages — fast to index, what foreign keys will point to. incident_id is the human-readable string that appears in emails and Jira tickets. Some designs collapse these into one. That's a mistake — the day you want to change the ID format, you'd be rewriting every foreign key in the database.
  • incident_id TEXT with no NOT NULL. The trigger populates it on insert. The application code inserts the row with incident_id blank, and Postgres fills it in before the row is actually written.

The counter table

One row per customer. That's it. No year_month column. The counter is continuous and never resets.

📄 schema.sql — incident_seq 
CREATE TABLE IF NOT EXISTS incident_seq (
  customer   TEXT NOT NULL PRIMARY KEY,
  last_seq   INTEGER NOT NULL DEFAULT 0
);

Three columns... wait, two. That's all the state we need. Each customer gets exactly one row holding their lifetime incident counter. The primary key on customer is what makes the locking work — but we'll get to that.

The Trigger That Makes IDs Atomic

This is the heart of the data layer. About 20 lines of PL/pgSQL, runs before every insert into incidents, and is bulletproof under concurrency.

📄 schema.sql — set_incident_id() trigger 
CREATE OR REPLACE FUNCTION set_incident_id()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_my   TEXT;
  v_seq  INTEGER;
  v_cust TEXT;
BEGIN
  -- MMYYYY in Asia/Kolkata (matches our analyst clocks)
  v_my   := TO_CHAR(NEW.created_at AT TIME ZONE 'Asia/Kolkata', 'MMYYYY');
  v_cust := COALESCE(NEW.customer, 'CLIENT-A');

  -- The atomic upsert. This single statement is the magic.
  INSERT INTO incident_seq (customer, last_seq)
    VALUES (v_cust, 1)
  ON CONFLICT (customer)
    DO UPDATE SET last_seq = incident_seq.last_seq + 1
  RETURNING last_seq INTO v_seq;

  -- Compose the final ID
  NEW.incident_id := v_cust || '-IR-' || v_my || '-' || LPAD(v_seq::TEXT, 3, '0');

  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_set_incident_id ON incidents;
CREATE TRIGGER trg_set_incident_id
  BEFORE INSERT ON incidents
  FOR EACH ROW EXECUTE FUNCTION set_incident_id();

Two parts worth understanding deeply.

1. The format line

v_my := TO_CHAR(NEW.created_at AT TIME ZONE 'Asia/Kolkata', 'MMYYYY');

This pulls the month-year from the row's timestamp in our local timezone. MMYYYY is the format mask — gives you 062026 for June 2026, 072026 for July, and so on. Different months produce different strings; the counter doesn't care.

2. The atomic upsert

INSERT INTO incident_seq (customer, last_seq)
  VALUES (v_cust, 1)
ON CONFLICT (customer)
  DO UPDATE SET last_seq = incident_seq.last_seq + 1
RETURNING last_seq INTO v_seq;

This is the part that makes the whole thing safe. Read it as: "Insert a new row with sequence 1. If a row already exists for this customer, instead increment that row's last_seq by one. Either way, give me the resulting last_seq value."

The locking property: Postgres takes a row-level lock on (customer) for the duration of this statement. If 10 workers try to run it at the same millisecond, they serialize on that one row. Worker 1 finishes (last_seq = 488), lock releases. Worker 2 enters (last_seq = 489). Worker 3 (last_seq = 490). And so on.

Each worker gets a unique number. The serialization happens on a single 4-byte row in microseconds. Even at 100 concurrent inserts, the total overhead is < 10 ms. Zero throughput cost, absolute correctness.

That's it. That's the bug fixed. From a 5% duplicate rate to zero, in 20 lines of PL/pgSQL.

Applying the Schema

Save the table definitions and trigger to a file, then pipe it into the Postgres container:

▸ apply schema.sql from the host 
# schema.sql lives next to docker-compose.yml
cd /opt/security-automation/

# Pipe it into psql inside the container
docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue < schema.sql

Or apply changes inline using a heredoc — useful for one-off tweaks:

▸ inline tweak via heredoc 
docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue <<'EOF'
ALTER TABLE incidents ADD COLUMN log_source TEXT DEFAULT '';
EOF

And for quick read-only queries, use the -c flag:

▸ one-off query 
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c \
  "SELECT customer, last_seq FROM incident_seq ORDER BY customer;"

Proving It Works

I don't trust correctness arguments. I trust tests. Here's the concurrency test I ran against the new schema — ten parallel inserts firing at the same time, then a query to see if any IDs collided:

▸ fire 10 inserts in parallel and check for duplicates 
# Launch 10 inserts in background
for i in {1..10}; do
  docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c \
    "INSERT INTO incidents (customer, alert_name, severity, category)
     VALUES ('CLIENT-A','concurrency-test-$i','LOW','Test')
     RETURNING incident_id;" &
done
wait

# Now check for duplicates
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c "
  SELECT incident_id, COUNT(*) FROM incidents
  WHERE alert_name LIKE 'concurrency-test-%'
  GROUP BY incident_id
  HAVING COUNT(*) > 1;
"

Cleanup after testing:

▸ clean up test rows 
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c "
  DELETE FROM incidents WHERE alert_name LIKE 'concurrency-test-%';
"

Note: the deleted rows leave gaps in the sequence (488 through 497 are gone but the counter is now at 497). That's fine. The counter is for uniqueness, not for "always count up by one." Skipped numbers aren't a bug.

How July's First Incident Becomes 510

This is the design property the rest of the platform depends on. Let's trace it.

On June 30 at 23:55 IST, an alert fires:

▸ last incident of June 
# Trigger sees created_at = 2026-06-30 23:55:00 IST
# v_my  = '062026'
# upsert returns last_seq = 509
# v_seq = 509
# incident_id = 'CLIENT-A-IR-062026-509'

Then at July 1 at 00:02 IST, the next alert arrives:

▸ first incident of July 
# Trigger sees created_at = 2026-07-01 00:02:00 IST
# v_my  = '072026'                    -- changed!
# upsert returns last_seq = 510       -- continues from 509
# v_seq = 510
# incident_id = 'CLIENT-A-IR-072026-510'

The month-year in the ID rolls forward. The sequence number does not. Exactly the behavior we wanted.

What Else Lives in the Data Layer (For Now)

Two more tables earn their place at this stage. Both are small.

incident_comments

Investigation notes attached to an incident. One incident, many comments. We'll surface these in the dashboard in Part 5.

📄 schema.sql — incident_comments 
CREATE TABLE IF NOT EXISTS incident_comments (
  id           SERIAL PRIMARY KEY,
  incident_seq INTEGER NOT NULL REFERENCES incidents(seq) ON DELETE CASCADE,
  author       TEXT NOT NULL,
  body         TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_comments_incident ON incident_comments(incident_seq);

The incident_seq foreign key points at the integer primary key on incidents, not at the human-readable incident_id. This was the whole point of keeping the integer key separate — foreign keys stay stable even if we someday change the ID format.

audit_log

Every state-changing action writes a row here. Status changes, comment additions, ticket closures — everything. The table is small, the schema is simple, and the value is enormous when something goes wrong.

📄 schema.sql — audit_log 
CREATE TABLE IF NOT EXISTS audit_log (
  id           SERIAL PRIMARY KEY,
  incident_seq INTEGER NOT NULL DEFAULT 0,
  action       TEXT NOT NULL,
  actor        TEXT NOT NULL,
  old_value    TEXT DEFAULT '',
  new_value    TEXT DEFAULT '',
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_audit_incident ON audit_log(incident_seq);
CREATE INDEX IF NOT EXISTS idx_audit_time ON audit_log(created_at DESC);

The table is append-only by convention. The application code never updates or deletes rows here. There's no audit-log-editor in the UI. If you really wanted to mutate it, you'd need direct database access — at which point you've already bypassed every other control on the system.

What gets logged at this stage:

  • Incident created (trigger or manual)
  • Status changed (open → resolved, etc.)
  • Comment added
  • Assignee changed

More categories will appear as the platform grows (logins, config changes), but they all follow the same shape: who did it, when, what changed.

The Migration I Broke

I promised in Part 1 to be honest about failures. Here's one from this layer.

When I needed to add the email_sent tracking column to incidents, I ran what felt like an innocent ALTER TABLE:

▸ what I ran (don't do this on a live table) 
docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue <<'EOF'
ALTER TABLE incidents ADD COLUMN email_sent BOOLEAN NOT NULL DEFAULT FALSE;
EOF

What I didn't realize: on older Postgres versions, ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT rewrites every row in the table. With ~4,000 incidents at the time, this locked the table for about 4 minutes. During those 4 minutes, every workflow trying to insert an incident hung. Workers piled up. The dashboard showed "connection error." The autoscaler ramped to max workers trying to drain the queue.

The fix: split the migration into three statements that don't rewrite rows:

▸ the safe pattern 
docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue <<'EOF'
-- Step 1: add nullable column (instant, no table rewrite)
ALTER TABLE incidents ADD COLUMN email_sent BOOLEAN;

-- Step 2: backfill in batches (no lock held long)
UPDATE incidents SET email_sent = FALSE WHERE seq BETWEEN 1 AND 1000;
UPDATE incidents SET email_sent = FALSE WHERE seq BETWEEN 1001 AND 2000;
UPDATE incidents SET email_sent = FALSE WHERE seq BETWEEN 2001 AND 3000;
UPDATE incidents SET email_sent = FALSE WHERE seq BETWEEN 3001 AND 4000;

-- Step 3: now add NOT NULL and DEFAULT (fast, no rewrite)
ALTER TABLE incidents ALTER COLUMN email_sent SET NOT NULL;
ALTER TABLE incidents ALTER COLUMN email_sent SET DEFAULT FALSE;
EOF

Three statements, each one fast. Total downtime: zero. The autoscaler log from the bad night, by the way, was visible right next to the dashboard's connection-error spike — exactly the pattern from Part 2 where the burst handler ramps up trying to keep up. That was the whole platform telling me I'd done something dumb.

The takeaway: production migrations are not the same as local schema edits. Even small-looking changes can lock a table for minutes if you don't think about what rewrites are happening underneath.

Beyond the SOC: Where This Pattern Applies

Atomic counters via Postgres triggers — broader uses

  • Invoice numbers. A finance app generating sequential per-customer invoice IDs needs the same guarantee — no duplicates, no gaps caused by application bugs.
  • Order IDs. E-commerce backends with multiple workers processing checkouts in parallel hit the same race condition I did.
  • Audit / case numbers. Compliance systems, legal case tracking, HR ticketing — anywhere a human-readable monotonic identifier matters.
  • Tenant-scoped sequences. Any SaaS multi-tenant app that wants per-tenant sequential IDs (rather than global UUIDs) can use this exact pattern: one row per tenant in a counter table, ON CONFLICT DO UPDATE in the trigger.
  • Receipt / confirmation codes. Booking systems, ticketing platforms — wherever a short numeric code is friendlier than a UUID.

The right place to enforce uniqueness isn't application code. It's the database — code older and more battle-tested than anything you'll write.

What's Next

The infrastructure is humming, IDs are atomic, and the database is the source of truth. What's missing is everything between the database and the analyst.

Right now, when an analyst wants to see open tickets, they SSH in and run a SQL query. When they want to add a comment, they hand-write an UPDATE. There's no login, no permissions, no API. Part 4 fixes that — a FastAPI backend that exposes the data layer to the dashboard, handles authentication, manages sessions, and syncs state back to Jira.

The backend is where Postgres becomes a product.
