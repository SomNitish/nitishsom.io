📕 PART 3 / 6 — Data Layer

The Database That Defines the Platform

A 12-table PostgreSQL schema, an atomic incident ID generator that handles concurrent inserts without race conditions, a permission model that can only restrict (never elevate), and an immutable audit log. The data layer isn't storage — it's the contract.

Why the Database Is the Source of Truth

In Part 2 we got the infrastructure running — main process, queue, workers, autoscaler. Workflows can now execute in parallel. But that immediately raises a question: how do you guarantee correctness when 27 workers are inserting into the same table?

The instinct from application engineering says: "use a mutex, use a Redis lock, use distributed coordination." That's all wrong here. The instinct that works is older and simpler: let the database be the source of truth.

The database has been solving concurrent-writes-to-shared-state problems for fifty years. The right architecture isn't to invent your own coordination layer on top — it's to push the invariants down into the database itself, where they're enforced by code that's been battle-tested across millions of production systems.

Everything I built into this schema follows that principle. Incident IDs aren't generated in application code — they're generated by a Postgres trigger. Audit entries can't be modified — the table is append-only by convention. Permissions can't be elevated through misconfiguration — the logic in the database enforces a ceiling. The schema doesn't just store data. It defines how the platform behaves.

The 12 Tables at a Glance

Here's the full catalog. Each one solves a specific problem.

users

Dashboard login accounts. Three roles (admin / analyst / readonly). Per-user permission overrides via JSONB.

The core table. One row per security incident. Auto-generated IDs. Soft-delete flag. MTTD/MTTA/MTTR fields.

Ticket assignment dropdown. Decoupled from user accounts on purpose (covered below).

Key-value store for platform settings — Jira credentials, status mappings, integration tokens.

The Big One: Atomic Incident ID Generation

This is the part I'm proudest of, and it's also the part that most often gets done wrong. Let me walk through the problem first, then the solution.

The Problem

Each incident needs a human-readable ID like CLIENT-A-IR-202606-001. The format is:

  • Tenant prefix — CLIENT-A or CLIENT-B
  • Type tag — IR for "Incident Response"
  • Year-month — 202606
  • Sequence number — 001, 002, ... reset every month, separate per tenant

So at the start of June 2026, Client-A's first incident gets CLIENT-A-IR-202606-001 and Client-B's first incident gets CLIENT-B-IR-202606-001 — independently. On July 1st, both reset to 001.

Sounds simple. Now imagine the implementation in naive application code:

# DON'T DO THIS — race condition guaranteed
def generate_incident_id(customer, conn):
    ym = datetime.now().strftime("%Y%m")
    last = conn.execute(
        "SELECT MAX(seq) FROM incident_month_seq "
        "WHERE customer=%s AND year_month=%s",
        [customer, ym]
    ).fetchone()[0] or 0
    new_seq = last + 1
    conn.execute(
        "INSERT INTO incident_month_seq VALUES (%s, %s, %s)",
        [customer, ym, new_seq]
    )
    return f"{customer}-IR-{ym}-{new_seq:03d}"

What happens when two workers call this function at the same time?

  1. Worker A reads last = 7 at T+0ms
  2. Worker B reads last = 7 at T+1ms (Worker A hasn't inserted yet)
  3. Worker A inserts seq=8 at T+5ms
  4. Worker B inserts seq=8 at T+6ms
  5. Both workflows generate the ID CLIENT-A-IR-202606-008
  6. You wake up the next morning to two incidents with the same ID

This is exactly the bug we shipped in the early days. We had 3 workers running, and during an alert burst, ~5% of incidents came out with duplicate IDs. We spent two days debugging it before realizing the issue wasn't in N8N — it was in our Python ID generator.

The fix isn't a mutex. It isn't a Redis lock. The fix is to let Postgres do what it was built to do.

The Counter Table

First, the table that holds the per-tenant monthly counter:

📄 schema.sql — incident_month_seq
CREATE TABLE IF NOT EXISTS incident_month_seq (
  customer   TEXT NOT NULL,
  year_month TEXT NOT NULL,
  last_seq   INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (customer, year_month)
);

That's it. Four columns. The composite primary key (customer, year_month) is everything — it guarantees that for any tenant+month combination there is exactly one row holding the counter.

The trick is in how we update it.

The Trigger Function

Postgres triggers fire automatically when rows are inserted, updated, or deleted. Our trigger fires before any incident insert and replaces the empty incident_id field with a freshly-generated value:

📄 schema.sql — set_incident_id() trigger
CREATE OR REPLACE FUNCTION set_incident_id()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_ym   TEXT;
  v_seq  INTEGER;
  v_cust TEXT;
BEGIN
  -- Get YYYYMM in Asia/Kolkata timezone (matches our analyst clocks)
  v_ym   := TO_CHAR(NEW.created_at AT TIME ZONE 'Asia/Kolkata', 'YYYYMM');
  v_cust := COALESCE(NEW.customer, 'CLIENT-A');

  -- The atomic upsert — this is the magic
  INSERT INTO incident_month_seq (customer, year_month, last_seq)
    VALUES (v_cust, v_ym, 1)
  ON CONFLICT (customer, year_month)
    DO UPDATE SET last_seq = incident_month_seq.last_seq + 1
  RETURNING last_seq INTO v_seq;

  -- Compose the final ID
  NEW.incident_id := v_cust || '-IR-' || v_ym || '-' || LPAD(v_seq::TEXT, 3, '0');

  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_set_incident_id ON incidents;
CREATE TRIGGER trg_set_incident_id
  BEFORE INSERT ON incidents
  FOR EACH ROW EXECUTE FUNCTION set_incident_id();

Why This Works Under Concurrency

The critical lines are these:

INSERT INTO incident_month_seq (customer, year_month, last_seq)
  VALUES (v_cust, v_ym, 1)
ON CONFLICT (customer, year_month)
  DO UPDATE SET last_seq = incident_month_seq.last_seq + 1
RETURNING last_seq INTO v_seq;

This is called an upsert, and it's atomic at the row level in Postgres. Here's what happens when 5 workers try to generate IDs simultaneously:

  1. All 5 workers run INSERT INTO incidents (...) at the same millisecond.
  2. The trigger fires for each one — 5 concurrent calls to set_incident_id().
  3. Each call tries to INSERT INTO incident_month_seq.
  4. Postgres takes a row-level lock on the (customer, year_month) row. Only one trigger can be inside the upsert at a time.
  5. Worker 1 finishes its upsert, gets last_seq=8, lock releases.
  6. Worker 2 enters, increments to 9, releases.
  7. Workers 3, 4, 5 follow in turn — 10, 11, 12.
  8. Each worker's trigger returns a unique v_seq, and the parent INSERT into incidents proceeds with a unique incident_id.

The key insight: the locking is happening on a single 4-byte counter row. It's measured in microseconds. Even at 100 concurrent inserts, the total serialization overhead is well under 10 milliseconds. The throughput cost is effectively zero, and the correctness guarantee is absolute.

Verifying It Works

Here's how I verified the trigger handles concurrency correctly. This is a real test I ran against the production schema using a test database:

📄 test_concurrent_inserts.sql
-- Set up two parallel sessions and have them race
-- Session A
BEGIN;
INSERT INTO incidents (customer, alert_name, severity, category)
  VALUES ('CLIENT-A', 'concurrent-test-A', 'LOW', 'Test');

-- Session B (started while A is uncommitted)
BEGIN;
INSERT INTO incidents (customer, alert_name, severity, category)
  VALUES ('CLIENT-A', 'concurrent-test-B', 'LOW', 'Test');
-- ^ Session B will BLOCK here until Session A commits or rolls back

-- Now commit Session A
COMMIT;
-- Session B unblocks, gets the next sequence number

-- Commit Session B
COMMIT;

-- Verify
SELECT incident_id, alert_name FROM incidents
WHERE alert_name LIKE 'concurrent-test-%'
ORDER BY incident_id;
-- Result:
-- CLIENT-A-IR-202606-008 | concurrent-test-A
-- CLIENT-A-IR-202606-009 | concurrent-test-B
-- Two distinct IDs, even though both sessions tried at the same time.

The blocking behavior in Session B is exactly what we want — it serializes the counter increment without any application-level coordination.

The incidents Table

This is the biggest table in the schema. It carries 40+ columns because incidents have a lot of state. Let me show it and then explain the design decisions.

  wazuh_received_at      TIMESTAMPTZ,
  n8n_received_at        TIMESTAMPTZ,

  mtta_seconds           INTEGER,
  mttr_seconds           INTEGER,

  deleted                BOOLEAN NOT NULL DEFAULT FALSE,
  deleted_at             TIMESTAMPTZ,
  deleted_by             TEXT,
  
  -- Deletion request workflow
  deletion_requested     BOOLEAN NOT NULL DEFAULT FALSE,
  deletion_reason        TEXT,
  deletion_requested_by  TEXT,
  deletion_requested_at  TIMESTAMPTZ,
  
  created_by             TEXT DEFAULT ''
);

Two Primary Keys (Sort Of)

Notice we have both seq SERIAL PRIMARY KEY and incident_id TEXT. Why both?

  • seq is the internal database identifier. It's an integer, it's fast to index, and it's what all foreign keys point to. incident_comments.incident_seq references this column.
  • incident_id is the human-readable identifier shown in the UI, Jira, emails, and reports. It's a TEXT field because it has structure (CLIENT-A-IR-202606-001).

Some designs collapse these into one — they use the human-readable string as the primary key. That's a mistake. The day you decide to change the ID format (e.g., add a region prefix, switch from 3-digit to 4-digit sequence), you're rewriting every foreign key in your database. Keep the internal integer key separate, and treat the display ID as derived data.

The Soft Delete Pattern

Look at the deletion fields:

deleted                BOOLEAN NOT NULL DEFAULT FALSE,
deleted_at             TIMESTAMPTZ,
deleted_by             TEXT,

deletion_requested     BOOLEAN NOT NULL DEFAULT FALSE,
deletion_reason        TEXT,
deletion_requested_by  TEXT,
deletion_requested_at  TIMESTAMPTZ,

This is the soft-delete-with-approval-workflow pattern. Two stages:

  1. Request: An analyst sets deletion_requested=TRUE with a reason. The incident stays visible but shows a "Deletion Pending" badge in the dashboard.
  2. Approval: An admin reviews the reason and either approves (sets deleted=TRUE) or rejects (clears the request fields).

Why this matters in a SOC context: incidents are evidence. They're tied to compliance audits, customer reports, and (occasionally) legal proceedings. If an analyst deletes an incident accidentally — or maliciously — there's no undo. With approval workflow, every deletion has a paper trail and a second pair of eyes.

Even after approval, deleted=TRUE is a soft delete. The row stays in the database. It just disappears from every query because every read uses WHERE NOT COALESCE(deleted, FALSE). If we need to recover an incident, we can. If we need to prove what was there for a customer audit, we can.

Pre-Computed Metric Columns

The three metric columns (mttd_seconds, mtta_seconds, mttr_seconds) are pre-computed and stored. Some database purists would say "no, compute these on read." But:

  • The dashboard's performance KPI section queries metrics for hundreds of incidents at a time. Computing 1,500 timestamp diffs on every page load adds 200ms+ latency.
  • The metrics depend on timestamps that, once set, never change. There's no "stale value" risk.
  • The backend can recompute these on demand if a corruption is detected (and it has — covered in Part 6).

This is the classic "denormalize for performance" tradeoff. Worth it here because reads vastly outnumber writes.

The Audit Log

Every state-changing action writes a row here. The table is small, the schema is simple, the value is immense:

Why "Append-Only by Convention"

Postgres doesn't have a built-in "append-only table" type. You enforce it through application code: every INSERT is allowed; UPDATE and DELETE on this table never happen. Anywhere.

I considered using a Postgres trigger to actively block updates and deletes. But that adds complexity, and the convention is enforced at the only layer that matters: the FastAPI code. There's no API endpoint that updates or deletes an audit row. There's no admin tool that lets you edit one. If you SSH into the database server and manually run UPDATE audit_log SET actor='...', you're already an attacker who's bypassed every other control.

For higher-assurance environments (PCI-DSS, HIPAA), you'd want database-level enforcement plus a separate write-only log shipped to a remote system. We didn't need that. Our audit log is for operational visibility and customer-facing compliance reports, not anti-fraud.

The Permission System

RBAC is one of those things that sounds simple but gets weird fast. Three roles? Five roles? Per-resource permissions? Per-action permissions? Hierarchical inheritance? You can spend months building one.

I built a model that I think hits the sweet spot for a SOC platform: roles define the ceiling; per-user settings can only restrict, never elevate. Here's how it works.

The users Table

📄 schema.sql — users
CREATE TABLE IF NOT EXISTS users (
  id                    SERIAL PRIMARY KEY,
  username              TEXT NOT NULL UNIQUE,
  full_name             TEXT NOT NULL,
  email                 TEXT NOT NULL,
  password_hash         TEXT NOT NULL,
  role                  TEXT NOT NULL DEFAULT 'analyst'
                        CHECK (role IN ('admin','analyst','readonly')),
  active                BOOLEAN NOT NULL DEFAULT TRUE,
  last_login            TIMESTAMPTZ,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  username_change_count INTEGER NOT NULL DEFAULT 0,
  username_changed_at   TIMESTAMPTZ,
  session_hours         INTEGER DEFAULT NULL,
  permissions           JSONB NOT NULL DEFAULT '{}'::jsonb
);
CREATE UNIQUE INDEX IF NOT EXISTS users_email_unique ON users(email);

Two things to notice:

  • role with a CHECK constraint — Database-level enforcement. You can't accidentally insert a role like 'superuser' via a typo. The constraint rejects the insert.
  • permissions JSONB — Per-user permission overrides. JSONB is structured, queryable, and far more flexible than 9 separate boolean columns.

The Three Roles and Their Defaults

The role defaults live in the application code (not the database) because they're often updated and the values are checked dynamically. Here's the dictionary that the backend uses:

📄 api.py — ROLE_DEFAULTS
ROLE_DEFAULTS = {
    "admin": {
        "view_incidents":  True,
        "add_comment":     True,
        "close_ticket":    True,
        "assign_analyst":  True,
        "create_manual":   True,
        "export_csv":      True,
        "delete_ticket":   True,
        "flush_queue":     True,
        "view_admin":      True,
    },
    "analyst": {
        "view_incidents":  True,
        "add_comment":     True,
        "close_ticket":    True,
        "assign_analyst":  True,
        "create_manual":   True,
        "export_csv":      True,
        "delete_ticket":   False,   # Can request, can't perform
        "flush_queue":     False,
        "view_admin":      False,
    },
    "readonly": {
        "view_incidents":  True,
        "add_comment":     False,
        "close_ticket":    False,
        "assign_analyst":  False,
        "create_manual":   False,
        "export_csv":      True,    # Export is read-only
        "delete_ticket":   False,
        "flush_queue":     False,
        "view_admin":      False,
    },
}

Each role gets a set of defaults. Admins get everything. Analysts can manage tickets but can't delete them or touch infrastructure. Readonly accounts can see everything and export to CSV but cannot modify anything.

The export-CSV-for-readonly choice is deliberate. Readonly is the role we use for the "TV monitor" account that lives on the wall display in the SOC. That account never touches data, but the SOC manager occasionally needs to grab a CSV from it without logging in to a different account. CSV export is read-only by definition — it can't damage anything.

The Permission Resolution Function

The key idea: per-user permissions can only restrict. Even if you set {"delete_ticket": true} on a readonly user, that true is ignored because the role doesn't grant it in the first place. Here's how it's enforced:

📄 api.py — get_user_permissions()
def get_user_permissions(user_id: int, role: str) -> dict:
    """Return effective permissions = role defaults merged with per-user overrides.
    Only False overrides are applied — permissions can only be restricted, not elevated."""
    import json
    defaults = ROLE_DEFAULTS.get(role, ROLE_DEFAULTS["readonly"]).copy()
    try:
        row = query("SELECT permissions FROM users WHERE id=%s", [user_id], fetch="one")
        if row and row.get("permissions"):
            overrides = row["permissions"]
            if isinstance(overrides, str):
                overrides = json.loads(overrides)
            for k, v in overrides.items():
                if k in defaults and not v:
                    defaults[k] = False    # Only the False values are applied
    except Exception:
        pass
    return defaults

Read line 11 carefully: if k in defaults and not v: defaults[k] = False.

  • if k in defaults — Ignore override keys that aren't in the role defaults. No way to invent new permissions.
  • and not v — Only apply the override if it's setting the value to False. True overrides are silently ignored.

The result: even if someone tampers with the permissions JSONB column to write {"flush_queue": true} on an analyst account, the analyst still can't flush the queue. The role default is False, and the override is silently dropped because not v is False.

This pattern is called "deny-overrides-allow" or "negative permissions only" — and it's the single most important security property in the whole RBAC system.

It means privilege escalation through misconfiguration is structurally impossible. A bug in the admin UI that lets you set arbitrary permissions on a user account can't grant that user access they didn't already have. The worst it can do is take away permissions — which is recoverable.

Real Use Case: Restricting an Analyst

An example from production. We hired a new junior analyst — let's call him User-X. We wanted him to be a full analyst but not have the ability to close tickets until he completed onboarding. The override:

UPDATE users
SET permissions = '{"close_ticket": false}'
WHERE username = 'user_x';

User-X can do everything else an analyst can do (view, comment, assign, create manual tickets), but the "Close" button doesn't appear in the UI for him. Two weeks later, after onboarding, we removed the override:

UPDATE users
SET permissions = '{}'
WHERE username = 'user_x';

One database change, no role reassignment. Clean and reversible.

Sessions and Browser Fingerprinting

Authentication generates a session token. The session table holds them:

📄 schema.sql — sessions
CREATE TABLE IF NOT EXISTS sessions (
  id         SERIAL PRIMARY KEY,
  token      TEXT NOT NULL UNIQUE,
  user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  expires_at TIMESTAMPTZ NOT NULL,
  user_agent TEXT DEFAULT ''
);

Five columns. The interesting ones are expires_at and user_agent.

Why Store the User-Agent?

It enables a feature I call "single-session-with-same-browser-refresh." The rule:

  • If a user logs in from the same browser as their existing session, the existing session is refreshed (expiration extended). They don't get a new token.
  • If a user logs in from a different browser, the old session is killed. Only one session per user, ever.

Why? Because the alternative is worse. If you allow multiple sessions, an attacker who steals a token can use it concurrently with the legitimate user, and no one notices. If you forbid duplicate sessions entirely, every page refresh logs the user out. The "same browser refreshes" middle ground is what enterprise SSO platforms do — and it's about 50 lines of Python here. (Full implementation in Part 4.)

Per-User Session Duration

Look at users.session_hours INTEGER DEFAULT NULL. When it's NULL, the system uses the global default of 1 hour. For specific accounts, we can set a longer session:

UPDATE users SET session_hours = 8 WHERE username = 'soc_tv_monitor';

The TV monitor account that lives on the wall display logs in once and stays logged in for 8 hours. The standard analyst accounts re-auth every hour. Granular, per-account, configurable.

Notifications: The Pub-Sub Inside the Database

Modern web apps reach for WebSockets or Server-Sent Events for real-time UI updates. I built notifications using nothing but a Postgres table and polling. Here's why and how.

📄 schema.sql — notifications
CREATE TABLE IF NOT EXISTS notifications (
  id           SERIAL PRIMARY KEY,
  user_id      INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type         TEXT NOT NULL DEFAULT 'system',
  incident_seq INTEGER,
  incident_id  TEXT DEFAULT '',
  title        TEXT NOT NULL,
  message      TEXT NOT NULL DEFAULT '',
  from_user    TEXT DEFAULT '',
  read         BOOLEAN NOT NULL DEFAULT FALSE,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Sources of notifications in the platform:

  • Ticket assignments — "Analyst X assigned CLIENT-A-IR-202606-042 to you."
  • Ticket closures — "Analyst Y closed CLIENT-A-IR-202606-042." Sent to whoever was assigned + the ticket creator.
  • Deletion requests — Sent to all admins when an analyst requests deletion.
  • Worker health alerts — Broadcast to all users when workers go down (and again when they recover).
  • Admin broadcasts — One-off messages from the admin UI.

Polling vs WebSockets

The dashboard polls /api/notifications/unread_count every 10 seconds. WebSockets would be more elegant. But:

  • The dashboard already polls every 3-5 seconds for queue metrics. Adding notification polling to that loop is essentially free.
  • WebSockets through reverse proxies are operationally fragile — they require sticky sessions, careful timeout tuning, and reconnect logic.
  • For ~20 active users, polling has zero performance impact. The endpoint runs a COUNT(*) FROM notifications WHERE user_id=? AND read=FALSE — one indexed query.
  • A 10-second latency on "you have a new notification" is fine for this use case.

Polling won on operational simplicity. The notification system is dead reliable and easy to debug. If a user says "I'm not getting notifications," I run one query against the table and either see the rows or don't.

The Decoupled Analyst Dropdown

This is a small but interesting design choice. There are two ways to handle the "Assign Analyst" dropdown:

  1. Show every user whose role is "analyst" in the dropdown.
  2. Maintain a separate list of analyst names that can be assigned.

I chose option 2. Here's the table:

📄 schema.sql — analysts
CREATE TABLE IF NOT EXISTS analysts (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL UNIQUE,
  email      TEXT DEFAULT '',
  active     BOOLEAN NOT NULL DEFAULT TRUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

The dropdown is fed from this table, not from the users table. Why? Three real reasons:

  1. External analysts. We sometimes need to assign tickets to people who don't have dashboard logins — outsourced incident responders, vendor SOC team members. They need to appear in the dropdown without getting a login account.
  2. Active vs. inactive. When an analyst leaves the team, we want them to disappear from the dropdown immediately. But their historical assignments should still display their name correctly. Disabling them in analysts removes them from new dropdowns; the assigned_to field on old incidents still shows their name.
  3. Display name vs. login name. The login name might be j.smith; the analyst's display name is Jane Smith. The dropdown should show the display name.

The analysts.name column is matched against incidents.assigned_to by string equality. They have to match exactly. The admin UI enforces this — when you add an analyst to the dropdown, the form validates against the spelling of their full_name in the users table.

The Migration I Broke (A Postmortem)

I promised in Part 1 to talk about my failures. Here's one.

When I introduced the soft-delete feature in v5.6, I needed to add four new columns to incidents: deleted, deleted_at, deleted_by, deletion_requested. I ran:

ALTER TABLE incidents ADD COLUMN deleted BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE incidents ADD COLUMN deleted_at TIMESTAMPTZ;
ALTER TABLE incidents ADD COLUMN deleted_by TEXT;
ALTER TABLE incidents ADD COLUMN deletion_requested BOOLEAN NOT NULL DEFAULT FALSE;

What I didn't realize: Postgres ALTER TABLE ... ADD COLUMN ... DEFAULT on a non-NULL boolean rewrites every row in the table. With ~4,000 rows at the time, this locked the table for about 4 minutes. During those 4 minutes, every workflow trying to insert an incident hung. The N8N queue backed up. The dashboard showed "connection error."

The fix would have been to split the migration:

-- Step 1: Add nullable column (instant, no rewrite)
ALTER TABLE incidents ADD COLUMN deleted BOOLEAN;

-- Step 2: Backfill in batches (no lock)
UPDATE incidents SET deleted = FALSE WHERE seq BETWEEN 1 AND 1000;
UPDATE incidents SET deleted = FALSE WHERE seq BETWEEN 1001 AND 2000;
-- ... etc

-- Step 3: Add the NOT NULL constraint (fast, no rewrite)
ALTER TABLE incidents ALTER COLUMN deleted SET NOT NULL;
ALTER TABLE incidents ALTER COLUMN deleted SET DEFAULT FALSE;

Three statements, each one instant. Total downtime: zero.

The 4-minute outage was visible in the audit log, in the autoscaler logs (which showed queue depth spiking and then dropping back), and in user_activity_log entries from confused analysts trying to refresh the page. I rolled out the fix the next morning and wrote up the postmortem.

The takeaway: production database migrations need to be planned like distributed system changes, not like local schema edits. The official Postgres docs have a great section on this — I should have read it before, not after.

Indexes

I'll keep this short because most of the queries hit primary keys, but two indexes were worth adding:

📄 schema.sql — indexes
-- Email uniqueness for login (case-insensitive matching done in application)
CREATE UNIQUE INDEX IF NOT EXISTS users_email_unique ON users(email);

-- Queue snapshots are time-series — newest first is the most common access
CREATE INDEX IF NOT EXISTS idx_qs_time ON queue_snapshots(captured_at DESC);

Why not more indexes? Because:

  • The incidents table is small. 5,000-10,000 rows. Sequential scans are fast at this size. Adding indexes costs writes and disk space for no measurable read benefit.
  • Most queries filter by seq or incident_id — both already indexed (one is the PK, the other is unique by trigger logic).
  • The dashboard's "list incidents" query has tab-style filters (customer, status, time range). Each filter narrows the result set. Postgres's query planner handles this efficiently with just the PK.

If the table grows past 100,000 rows, I'll revisit. Maybe a composite index on (customer, status, created_at). But premature indexing is its own performance bug.
