Multi-Threaded N8N with Auto-Scaling Workers

Docker Compose architecture, queue mode configuration, and the bash daemon that scales worker containers from 10 to 50 based on Redis queue depth — keeping latency low during alert storms without burning resources at idle.

Where We Are

In Part 1 I described the problem: a single-threaded N8N container choking on incoming alerts, race conditions producing duplicate incident IDs, and no way to scale without restarting everything. This part covers how I solved it at the infrastructure layer.

The whole platform runs on a single Ubuntu 24.04 server in /opt/security-automation/. No Kubernetes, no cloud-managed services, no fancy orchestration. Just Docker Compose, a couple of bash scripts, and systemd.

That's deliberate. Complexity has a cost. The simpler the operations layer, the easier it is to debug at 2 AM when something breaks.

Every layer of the stack should be something a new engineer can fully understand in an afternoon. If they can't, you're going to lose them at the worst possible moment — when the platform is on fire and they're the only person around to fix it.

Why Queue Mode (and Not Something Else)

Before diving into the configs, I want to address the obvious question: why N8N queue mode? Why not Celery? Why not just a different tool entirely?

The honest answer is that I evaluated several options:

Option Pros Cons
Apache Airflow Mature, distributed, scheduled Heavy, DAG model is wrong for event-driven SOAR
Temporal Best-in-class workflow engine Steep learning curve, code-first (no GUI for analysts)
Custom Celery Full control, Python-native 3-6 months of dev time, no visual workflow editor
StackStorm Built for SOAR, mature YAML hell, declining community, hard to debug
N8N queue mode Visual editor, queue mode is production-ready, huge integration library Some quirks around concurrency, less battle-tested at scale

N8N won on the dimension that matters most for a SOC: an analyst can look at a workflow and understand it. When I'm on vacation and a junior analyst needs to figure out why an alert didn't generate a ticket, they can open the N8N editor and visually trace the flow. That's not possible with Celery or Temporal.

The "less battle-tested at scale" concern was solved by understanding the queue mode architecture deeply. Which is what we'll do next.

How Queue Mode Actually Works

In default mode, N8N is a single process. Webhook arrives → workflow executes in the same process → result is written to the database. If that process is busy executing a 90-second workflow, every new webhook waits.

Queue mode separates the responsibilities:

  1. The main process only handles the editor UI, webhook ingestion, and scheduling. It doesn't execute workflows.
  2. Redis becomes the queue. The main process pushes jobs onto Redis. Workers pull them off.
  3. Worker processes are completely separate. They connect to Redis, claim jobs, execute them, write results to Postgres. You can have 1 worker or 50 — N8N doesn't care.

The magic property of this design: workers are stateless. Any worker can execute any workflow. If a worker crashes mid-execution, the job goes back to the queue and another worker picks it up. No coordinator, no leader election, no consensus protocol. Just Redis and BullMQ doing what they do best.

The implication: scaling becomes a matter of running more workers. There's no code change. There's no config change. There's no coordination. docker compose up -d --scale n8nq-worker=20 and you've gone from 10 to 20 workers. Five seconds later they're processing jobs.

The PostgreSQL Service

Three things worth pointing out:

  • Port 5433 on host, 5432 in container. The host server already had a Postgres instance running for an old project on the default port. Rather than fight that, I mapped the new instance to 5433. This is a one-character change that saves you hours of "why won't Postgres start" debugging.
  • The healthcheck uses pg_isready, not a generic nc -z. pg_isready actually verifies the database is accepting connections and can authenticate. nc -z just checks if the port is open — which it would be even if Postgres was still loading.
  • start_period: 10s. Postgres takes a few seconds to initialize on first boot. Without start_period, the healthcheck fails immediately and Docker marks the container unhealthy. The first 10 seconds are a grace period.

The Redis Service

This one is worth understanding because Redis misconfiguration is the #1 cause of "my queue mode setup just disappears jobs."

  • --requirepass — Always set a password. Even on a private network. The number of unauthenticated Redis instances exposed on the internet is staggering. Defense in depth.
  • --appendonly yes — This enables AOF (Append-Only File) persistence. If the Redis container restarts, the queue survives. Without this, every restart drops pending jobs on the floor.
  • --maxmemory 256mb — Caps Redis memory at 256MB. Plenty for queue mode (we've never come close).
  • --maxmemory-policy noeviction — This is critical. The default is noeviction but I set it explicitly because the alternatives are dangerous. With allkeys-lru (the default for caching), Redis would silently delete old queued jobs to make room for new ones. You'd lose alerts and not know about it. With noeviction, if memory fills up, new writes fail loudly, and we get an alert. Loud failure beats silent data loss every time.

The N8N Main Service

The main process runs the editor UI and ingests webhooks. It's the entry point. Three details matter:

  • Reads from .env via env_file: .env. All config lives in one place. Workers read from the same file, ensuring main and workers agree on every setting.
  • SSL certs mounted read-only at /home/node/.n8n/certs. The :ro flag means even if N8N got compromised, it can't modify the certs. Small detail, real security value.
  • The healthcheck uses nc -z localhost 5678. We're just checking the port is bound. A full HTTP healthcheck would be better, but N8N's /healthz endpoint sometimes responds before the queue subsystem is actually ready. The TCP check is more reliable in practice.

The N8N Worker Service

This is where the magic happens. Note these subtle but important details:

  • command: worker — This single word turns an N8N container into a worker. Same image, same env vars, completely different role. That's elegant.
  • No container_name — When you have multiple replicas, you can't name them all the same thing. Docker auto-generates names like n8nq-worker-1, n8nq-worker-2, etc.
  • No ports exposed — Workers don't serve HTTP. They only pull from Redis and write to Postgres. There's nothing to expose.
  • deploy.replicas: 2 — This is the default at boot. The autoscaler (next section) adjusts this dynamically.
  • The healthcheck uses pgrep — We check that the n8n worker process is actually running. If the process died but the container is still alive (rare, but it happens), the healthcheck catches it.

The Settings That Matter Most

Let me call out the ones that aren't obvious:

EXECUTIONS_MODE=queue

This is the master switch. Without it, N8N ignores Redis entirely and runs in default single-threaded mode. The setting must be present in both main and worker containers. If main has it but workers don't, workers won't start. If workers have it but main doesn't, main will execute workflows itself and ignore the workers. The env_file: .env approach guarantees both read the same value.

N8N_CONCURRENCY_PRODUCTION_LIMIT=5

Each worker can execute up to 5 workflows in parallel before it stops claiming new jobs. With 10 workers, the platform can run 50 workflows simultaneously. With 50 workers, 250. This number is the lever you pull when you want each worker to do more vs. when you want to scale horizontally.

I settled on 5 after testing values from 1 to 20. Below 5, workers were underutilized. Above 8, individual workflows started timing out because the worker was thrashing. Five is the sweet spot for our workload (Wazuh whitelist checks + Postgres inserts + HTTP calls to Jira and Microsoft Graph).

N8N_ENCRYPTION_KEY

This is the most important value in the entire .env. N8N encrypts all credentials (Jira API tokens, OAuth refresh tokens, database passwords stored in workflows) using this key.

⚠ Lose this key and every credential in your N8N instance becomes unrecoverable. You'll have to re-enter every API token, every OAuth setup, every database connection. For us, that meant ~30 credentials and about 4 hours of work.

The first day I had this set up, I rotated the key without saving the old one. Spent an entire Saturday afternoon rebuilding credentials. Don't do that.

OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true

By default, when you click "Execute Workflow" in the N8N editor, the workflow runs in the main process. That blocks the UI thread and defeats the purpose of queue mode for manual testing. This setting routes manual executions through workers too. Always set this in queue mode.

EXECUTIONS_DATA_MAX_AGE=168

168 hours = 7 days. N8N stores every execution's full payload by default. At 5,000 executions per month, this database grows fast. We retain 7 days for debugging and discard older data. The pruner runs automatically on the main process. If you don't set this, expect Postgres disk usage to grow by 5-10GB per week.

N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0

N8N has two execution engines. Version 0 is the older, more stable one. Version 1 has edge cases when sub-workflows pass data between each other — exactly the pattern our SOAR workflows use. Pinning to version 0 saved us from a class of weird bugs. If you start fresh on a newer N8N, test version 1 carefully before turning it on.

The Auto-Scaler

Two workers handle our baseline load with room to spare. But baseline isn't the problem — burst traffic is.

An "alert storm" in our environment looks like this: a misconfigured app deploys, and 200 alerts hit the system in 90 seconds. With 2 workers running 5-concurrency each, that's 10 parallel executions. The remaining 190 alerts queue up. With a workflow average of 2-3 seconds, the queue clears in about a minute. Not terrible — but not great.

The right answer is: scale up automatically during the storm, scale back down when it's over. This is what the autoscaler does.

📄 autoscale.sh
#!/bin/bash
# ─────────────────────────────────────────────────────────────
# N8N Worker Auto-Scaler
# 
# Polls Redis queue depth every 5 seconds and adjusts worker
# count between MIN_WORKERS and MAX_WORKERS based on load.
# 
# Cooldown prevents thrashing during transient spikes.
# ─────────────────────────────────────────────────────────────

MIN_WORKERS=10
MAX_WORKERS=50
SCALE_UP_THRESHOLD=1        # ≥ this many queued → scale up
SCALE_DOWN_THRESHOLD=2      # ≤ this many queued → scale down
COOLDOWN=10                 # seconds between scaling actions
COMPOSE_DIR="/opt/security-automation"

REDIS_PASS=$(grep ^REDIS_PASSWORD "$COMPOSE_DIR/.env" | cut -d= -f2)
LAST_SCALE=0

# Total queue depth = pending jobs + active jobs
get_queue_depth() {
    local wait active
    wait=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN "bull:jobs:wait" 2>/dev/null | tr -cd '0-9')
    active=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN "bull:jobs:active" 2>/dev/null | tr -cd '0-9')
    echo $(( ${wait:-0} + ${active:-0} ))
}

# Count running worker containers
get_current_workers() {
    local count
    count=$(docker compose -f "$COMPOSE_DIR/docker-compose.yml" \
        ps --format json 2>/dev/null | grep -c "n8nq-worker" || echo 2)
    echo "${count:-2}"
}

# Scale the worker service, respecting cooldown
scale_workers() {
    local target=$1
    local now current elapsed
    now=$(date +%s)
    elapsed=$((now - LAST_SCALE))

    # Skip if we're still in cooldown
    [ "$elapsed" -lt "$COOLDOWN" ] && return

    current=$(get_current_workers)
    # Skip if already at target
    [ "$current" -eq "$target" ] && return

    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Scaling workers: $current -> $target"
    cd "$COMPOSE_DIR"
    docker compose up -d --scale n8nq-worker="$target" --no-recreate 2>/dev/null
    LAST_SCALE=$now
}

echo "[$(date '+%Y-%m-%d %H:%M:%S')] Auto-scaler started | Min=$MIN_WORKERS Max=$MAX_WORKERS"

# Main loop
while true; do
    QUEUE=$(get_queue_depth)
    CURRENT=$(get_current_workers)

    echo "[$(date '+%H:%M:%S')] Queue depth: $QUEUE | Workers: $CURRENT"

    # Scale up if queue is busy
    if [ "$QUEUE" -ge "$SCALE_UP_THRESHOLD" ] && [ "$CURRENT" -lt "$MAX_WORKERS" ]; then
        NEW=$((CURRENT + 1))
        [ "$NEW" -gt "$MAX_WORKERS" ] && NEW=$MAX_WORKERS
        scale_workers "$NEW"

    # Scale down if queue is empty and we have spare workers
    elif [ "$QUEUE" -le "$SCALE_DOWN_THRESHOLD" ] && [ "$CURRENT" -gt "$MIN_WORKERS" ]; then
        NEW=$((CURRENT - 1))
        [ "$NEW" -lt "$MIN_WORKERS" ] && NEW=$MIN_WORKERS
        scale_workers "$NEW"
    fi

    sleep 5
done

Design Decisions in the Autoscaler

Several things in this script are worth talking about because they're not obvious.

Why MIN_WORKERS=10 and not 2?

Baseline capacity. With 2 workers, a sudden burst means jobs queue up for ~30 seconds before the autoscaler reacts and a new worker becomes ready. With 10 workers running 5-concurrency, we can absorb a burst of 50 simultaneous workflows with zero queue depth. The 8 extra idle workers cost us about 200MB of RAM. That's a trivial cost for never having a queue backlog under normal conditions.

Why scale up at queue depth = 1?

Because by the time you see queue depth = 5, you're already late. The whole point of autoscaling is to react before the user notices. Setting the threshold at 1 means we're scaling up the instant anything starts queueing — and during a real burst, we'll be at MAX_WORKERS within 30-40 seconds.

Why scale down at queue depth ≤ 2 (not 0)?

To avoid flapping. If we scale down at 0, then a single arriving job triggers scale-up. The system thrashes. Scale-down at 2 means there's some buffer — we only release workers when the queue is genuinely quiet.

The COOLDOWN of 10 seconds

This is the key anti-flapping mechanism. After any scaling action, the script ignores further scale signals for 10 seconds. That gives Docker time to actually start the new worker and for it to start claiming jobs. Without the cooldown, you'd see scale up → scale down → scale up patterns when the queue is hovering around the threshold.

The --no-recreate flag

This is critical. Without it, every docker compose up -d --scale would tear down and recreate all worker containers. With it, only the delta is applied — going from 10 to 11 workers starts 1 new container, doesn't touch the existing 10. This matters because jobs in flight on existing workers would be killed during recreation. Setting this saved me from a class of subtle bugs.

Reading Redis password from .env

REDIS_PASS=$(grep ^REDIS_PASSWORD "$COMPOSE_DIR/.env" | cut -d= -f2) — The autoscaler reads the Redis password from the same .env file the containers use. No duplication, no risk of drift. When you rotate the Redis password, you change it in one place.

Running the Autoscaler as a Systemd Service

You don't want to leave the autoscaler running in a screen session. You want it to start on boot, restart on failure, and log to journald. That's what systemd is for.

📄 /etc/systemd/system/n8n-autoscale.service
[Unit]
Description=N8N Worker Auto-Scaler
After=docker.service
Requires=docker.service

[Service]
Type=simple
WorkingDirectory=/opt/security-automation
ExecStart=/bin/bash /opt/security-automation/autoscale.sh
Restart=always
RestartSec=5
StandardOutput=append:/opt/security-automation/autoscale.log
StandardError=append:/opt/security-automation/autoscale.log
User=root

# Don't allow runaway resource use
LimitNOFILE=4096

[Install]
WantedBy=multi-user.target

Enable and start:

sudo systemctl daemon-reload
sudo systemctl enable n8n-autoscale.service
sudo systemctl start n8n-autoscale.service

# Check status
sudo systemctl status n8n-autoscale.service

# Watch the logs
tail -f /opt/security-automation/autoscale.log

Why I Use Bash and Not Python

The autoscaler is 60 lines of bash. I could have written it in Python with a proper logging library, type hints, and a config schema. I chose bash because:

  • Zero dependencies. Bash and docker are already on the box. No pip install, no virtualenv, no Python version drift.
  • One file, end-to-end readable. Anyone on the team can read 60 lines of bash and understand what's happening. A Python equivalent would be 200+ lines once you add proper structure.
  • No moving parts. The script either runs or it doesn't. Bash's failure modes are simpler than Python's.
  • Composes with shell tools. docker compose, redis-cli, grep, cut — these are all command-line tools. Calling them from Python adds friction.

If the script grew past 200 lines or needed structured config, I'd port it to Python. At 60 lines it would be over-engineered.

What Happens During Worker Container Failure

This is the design question that determines whether queue mode is "production-grade" or "neat demo": what happens when a worker dies mid-execution?

BullMQ (the queue library N8N uses) handles this with job ownership leases. When a worker claims a job, it's marked as "active" with a timestamp. If the worker doesn't complete or refresh the lease within a timeout, BullMQ marks the job as stalled and re-queues it. Another worker picks it up.

The exact behavior:

  1. Worker-3 claims job exec_id=4521 at T+0s
  2. Worker-3 starts executing — runs Whitelist Check node, runs Postgres insert
  3. Worker-3's container crashes at T+1.2s (OOM kill, host reboot, whatever)
  4. BullMQ's stalled-job checker runs (default every 30 seconds)
  5. Job exec_id=4521 is marked as stalled and re-queued
  6. Worker-5 claims it, executes it from the start

⚠ The catch: workflows must be idempotent for this to work safely.

If the original execution wrote a row to Postgres before crashing, the retry will write a second row. We solved this in two ways:

  1. The incident ID generator (covered in Part 3) is wrapped in a Postgres trigger so duplicate inserts get unique IDs — they don't collide.
  2. External API calls (Jira ticket creation) are at the end of the workflow. If a worker crashes before that, the only side effect is a half-written database row, which we clean up via a maintenance script.

What I'd Do Differently

In the spirit of honesty I promised in Part 1, here are the parts of this setup I'd revisit:

1. The 5-second polling interval is too tight

Every 5 seconds, the autoscaler runs docker exec against the Redis container. That's about 17,000 docker exec calls per day. It works, but it generates a lot of unnecessary Docker daemon traffic. I'd switch to a Python script using redis-py connecting directly to Redis (port 6380 is exposed) — same effect, no docker exec overhead.

2. Cooldown should be asymmetric

Scaling up should be fast (no cooldown, or 5 seconds). Scaling down should be slow (60+ seconds) to absorb transient quiet periods. Currently both use the same 10-second cooldown. I noticed during testing that the system sometimes scales down a worker, then immediately scales it back up when the next burst arrives — wasting the container teardown cycle.

3. Should publish metrics to Prometheus

The autoscaler logs everything to a flat file. That's fine for grep-and-investigate. But for time-series analysis — "what's our peak worker count this quarter?" — I'd want Prometheus scraping the autoscaler. Maybe Part 6 will cover this.

4. The healthchecks could be smarter

The worker healthcheck just checks pgrep -f 'n8n worker'. If the worker process is alive but has a stuck job (e.g., infinite loop in a custom node), the healthcheck passes. A smarter check would query the worker's local state and verify it's actively processing.
