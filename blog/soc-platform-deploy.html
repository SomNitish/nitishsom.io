Deploy the SOC Platform

The no-narrative version of the 6-part series. Copy the three files, follow eight steps, and have the full N8N queue-mode automation stack running on your own host in about twenty minutes.

📚

This page is just the deploy. If you want the why — the use case, the design decisions, the postmortems — read the 6-part blog series. It walks through the same architecture but explains every choice. Links to all parts are at the bottom of this page.

What You're Deploying

A self-hosted, multi-tenant security automation platform with:

  • N8N in queue mode — main process for webhooks + UI, separate worker containers for parallel job execution
  • PostgreSQL 16 — N8N state + the SOC dashboard schema (12 tables, including atomic incident ID generation via a trigger)
  • Redis 7 — the BullMQ queue between main and workers
  • SSL by default — self-signed certs for internal use; swap for a real cert in production

What's not included on this page: the FastAPI dashboard backend and the vanilla-JS frontend. Those are separate components (see Parts 4 and 5). This SOP gets the workflow engine + database running — which is the platform's load-bearing layer.

Prerequisites

🐧

Linux host

Ubuntu 22.04+ recommended. Any modern Linux with systemd works.

🐳

Docker Engine

Version 24+ with Docker Compose v2 plugin.

💾

Resources

4 GB RAM minimum, 20 GB disk. SSD strongly preferred.

🔓

Root or sudo

For writing to /opt/ and managing the docker daemon.

Ports we use

PortServiceDirection
5678N8N main (HTTPS) — editor + webhooksInbound from SIEM/SOAR sources
5433PostgreSQL (host-mapped)Local only — for backups/restore
6380Redis (host-mapped)Local only — for monitoring/autoscaler

Make sure 5678 is reachable from your SIEM. The other two are accessed via docker exec from the host and shouldn't be exposed externally.

Step-by-Step Deploy

1

Create the folder structure

All platform files live under /opt/security-automation/. One directory, easy to back up, easy to find.

▸ run on the host 
sudo mkdir -p /opt/security-automation/{certs,dashboard,backups}
sudo chown -R $USER:$USER /opt/security-automation
cd /opt/security-automation
2

Generate a self-signed SSL cert

N8N needs HTTPS for webhooks. A self-signed cert is fine for internal SOC use — replace with a real cert from Let's Encrypt or your CA if exposing externally.

▸ generate the cert 
cd /opt/security-automation/certs
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
    -sha256 -days 365 -nodes \
    -subj "/CN=soc-platform-host"
chmod 600 key.pem
chmod 644 cert.pem
cd ..
3

Create the .env file

This holds every secret and config value. Same file is read by all three services (main, workers, postgres init). Save the template below as /opt/security-automation/.env, then replace the four [REPLACE_*] placeholders.

Generate the encryption key:

▸ generate N8N_ENCRYPTION_KEY 
openssl rand -hex 32

Copy the output into the N8N_ENCRYPTION_KEY= line in .env, then lock down the file:

▸ secure the .env file 
chmod 600 /opt/security-automation/.env

⚠ Back up your .env immediately after editing it. If you lose N8N_ENCRYPTION_KEY, every credential stored inside N8N workflows (Jira tokens, OAuth handshakes, integration passwords) becomes permanently unreadable — even if the database is intact.

-- Version: 1.1 (lifetime counter + MMYYYY format)

-- HOW TO RUN:

-- v1.1 CHANGES:
--   - Replaced incident_month_seq with incident_seq (lifetime counter,
--     no monthly reset)
--   - Trigger now formats date as MMYYYY instead of YYYYMM
--     Old: CLIENT-A-IR-202607-001 (resets monthly)
--     New: CLIENT-A-IR-072026-510 (continues forever per customer)
-- ═══════════════════════════════════════════════════════════════════════

-- TABLE 1: users — dashboard login accounts
CREATE TABLE IF NOT EXISTS users (
  id                    SERIAL PRIMARY KEY,
  username              TEXT NOT NULL UNIQUE,
  full_name             TEXT NOT NULL,
  email                 TEXT NOT NULL,
  password_hash         TEXT NOT NULL,
  role                  TEXT NOT NULL DEFAULT 'analyst'
                        CHECK (role IN ('admin','analyst','readonly')),
  active                BOOLEAN NOT NULL DEFAULT TRUE,
  last_login            TIMESTAMPTZ,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  username_change_count INTEGER NOT NULL DEFAULT 0,
  username_changed_at   TIMESTAMPTZ,
  session_hours         INTEGER DEFAULT NULL,
  permissions           JSONB NOT NULL DEFAULT '{}'::jsonb
);
CREATE UNIQUE INDEX IF NOT EXISTS users_email_unique ON users(email);

-- TABLE 2: sessions — one row per active login session
CREATE TABLE IF NOT EXISTS sessions (
  id         SERIAL PRIMARY KEY,
  token      TEXT NOT NULL UNIQUE,
  user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  expires_at TIMESTAMPTZ NOT NULL,
  user_agent TEXT DEFAULT ''
);

-- TABLE 3: incidents — the core table (one row per security incident)
CREATE TABLE IF NOT EXISTS incidents (
  seq                    SERIAL PRIMARY KEY,
  incident_id            TEXT,
  customer               TEXT NOT NULL DEFAULT 'CLIENT-A',
  alert_name             TEXT NOT NULL,
  severity               TEXT NOT NULL DEFAULT 'MEDIUM',
  category               TEXT NOT NULL DEFAULT 'Alert',
  log_source             TEXT DEFAULT '',
  status                 TEXT NOT NULL DEFAULT 'open'
                         CHECK (status IN ('open','closed','reopened','resolved')),
  assigned_to            TEXT DEFAULT '',
  logged_at              TEXT DEFAULT '',
  email_sent             BOOLEAN NOT NULL DEFAULT FALSE,
  email_sent_at          TIMESTAMPTZ,

  jira_key               TEXT DEFAULT '',

  resolved_at            TIMESTAMPTZ,
  assigned_at            TIMESTAMPTZ,
  alert_generated_at     TIMESTAMPTZ,
  wazuh_received_at      TIMESTAMPTZ,
  n8n_received_at        TIMESTAMPTZ,
  mttd_seconds           INTEGER,
  mtta_seconds           INTEGER,
  mttr_seconds           INTEGER,
  remarks                TEXT DEFAULT '',
  is_manual              BOOLEAN NOT NULL DEFAULT FALSE,
  deleted                BOOLEAN NOT NULL DEFAULT FALSE,
  deleted_at             TIMESTAMPTZ,
  deleted_by             TEXT,
  deletion_requested     BOOLEAN NOT NULL DEFAULT FALSE,
  deletion_reason        TEXT,
  deletion_requested_by  TEXT,
  deletion_requested_at  TIMESTAMPTZ,
  created_by             TEXT DEFAULT ''
);

-- TABLE 4: incident_seq — lifetime counter per customer (never resets)
-- Example:
--   Last incident of June  -> CLIENT-A-IR-062026-509
--   First incident of July -> CLIENT-A-IR-072026-510  (NOT reset to 001)
CREATE TABLE IF NOT EXISTS incident_seq (
  customer  TEXT NOT NULL PRIMARY KEY,
  last_seq  INTEGER NOT NULL DEFAULT 0
);

-- TRIGGER: set_incident_id — auto-generates incident_id on every INSERT
-- Format: {customer}-IR-{MMYYYY}-{NNN}
CREATE OR REPLACE FUNCTION set_incident_id()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_my   TEXT;
  v_seq  INTEGER;
  v_cust TEXT;
BEGIN
  v_my   := TO_CHAR(NEW.created_at AT TIME ZONE 'Asia/Kolkata', 'MMYYYY');
  v_cust := COALESCE(NEW.customer, 'CLIENT-A');

  INSERT INTO incident_seq (customer, last_seq)
    VALUES (v_cust, 1)
  ON CONFLICT (customer)
    DO UPDATE SET last_seq = incident_seq.last_seq + 1
  RETURNING last_seq INTO v_seq;

  NEW.incident_id := v_cust || '-IR-' || v_my || '-' || LPAD(v_seq::TEXT, 3, '0');
  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_set_incident_id ON incidents;
CREATE TRIGGER trg_set_incident_id
  BEFORE INSERT ON incidents
  FOR EACH ROW EXECUTE FUNCTION set_incident_id();

-- TABLE 5: incident_comments — investigation notes attached to incidents
CREATE TABLE IF NOT EXISTS incident_comments (
  id           SERIAL PRIMARY KEY,
  incident_seq INTEGER NOT NULL REFERENCES incidents(seq) ON DELETE CASCADE,
  author       TEXT NOT NULL,
  user_id      INTEGER REFERENCES users(id),
  comment      TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),

9

(Optional) Set up the auto-scaler

The platform works fine with the two default worker replicas in docker-compose.yml. But if you expect alert bursts — a misconfigured firewall rule, a vulnerability scanner kicking off, anything that fires hundreds of alerts in a few minutes — you'll want workers to scale up automatically and scale back down when things are quiet.

This is a small bash daemon that polls Redis queue depth every 5 seconds and adjusts the worker count between MIN_WORKERS and MAX_WORKERS. Managed by systemd so it survives reboots.

Save this as /opt/security-automation/autoscale.sh:

📄 /opt/security-automation/autoscale.sh 
#!/bin/bash
# -----------------------------------------------------------
# N8N Worker Auto-Scaler
# Polls Redis queue depth and scales workers between MIN/MAX.
# -----------------------------------------------------------

MIN_WORKERS=10
MAX_WORKERS=50
SCALE_UP_THRESHOLD=1        # queued jobs >= this -> scale up
SCALE_DOWN_THRESHOLD=2      # queued jobs <= this -> scale down
COOLDOWN=10                 # seconds between scale actions
COMPOSE_DIR="/opt/security-automation"

REDIS_PASS=$(grep ^REDIS_PASSWORD "$COMPOSE_DIR/.env" | cut -d= -f2)
LAST_SCALE=0

get_queue_depth() {
    local wait active
    wait=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN bull:jobs:wait 2>/dev/null | tr -cd '0-9')
    active=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN bull:jobs:active 2>/dev/null | tr -cd '0-9')
    echo $(( ${wait:-0} + ${active:-0} ))
}

get_current_workers() {
    local count
    count=$(docker compose -f "$COMPOSE_DIR/docker-compose.yml" \
        ps --format json 2>/dev/null | grep -c "n8nq-worker" || echo 2)
    echo "${count:-2}"
}

scale_workers() {
    local target=$1
    local now current elapsed
    now=$(date +%s)
    elapsed=$((now - LAST_SCALE))

    [ "$elapsed" -lt "$COOLDOWN" ] && return     # still cooling down

    current=$(get_current_workers)
    [ "$current" -eq "$target" ] && return       # already at target

    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Scaling workers: $current -> $target"
    cd "$COMPOSE_DIR"
    docker compose up -d --scale n8nq-worker="$target" --no-recreate 2>/dev/null
    LAST_SCALE=$now
}

echo "[$(date '+%Y-%m-%d %H:%M:%S')] Auto-scaler started | Min=$MIN_WORKERS Max=$MAX_WORKERS"

while true; do
    QUEUE=$(get_queue_depth)
    CURRENT=$(get_current_workers)
    echo "[$(date '+%H:%M:%S')] Queue: $QUEUE | Workers: $CURRENT"

    if [ "$QUEUE" -ge "$SCALE_UP_THRESHOLD" ] && [ "$CURRENT" -lt "$MAX_WORKERS" ]; then
        NEW=$((CURRENT + 1))
        [ "$NEW" -gt "$MAX_WORKERS" ] && NEW=$MAX_WORKERS
        scale_workers "$NEW"
    elif [ "$QUEUE" -le "$SCALE_DOWN_THRESHOLD" ] && [ "$CURRENT" -gt "$MIN_WORKERS" ]; then
        NEW=$((CURRENT - 1))
        [ "$NEW" -lt "$MIN_WORKERS" ] && NEW=$MIN_WORKERS
        scale_workers "$NEW"
    fi

    sleep 5
done

Make it executable:

▸ make executable 
sudo chmod +x /opt/security-automation/autoscale.sh

# Watch it work
tail -f /opt/security-automation/autoscale.log

You should see a line every 5 seconds reporting the current queue depth and worker count. When the queue piles up, the script scales workers up; when things go quiet, it scales them back down to MIN_WORKERS.

Tuning for your environment: The defaults (MIN=10, MAX=50) are sized for a busy SOC. For lower-volume setups, drop MIN_WORKERS to 2 and MAX_WORKERS to 10. Edit the variables at the top of the script and restart the service: sudo systemctl restart n8n-autoscale.

Quick Troubleshooting

The most common failures during a first-time deploy:

n8nq-main stuck in "unhealthy" state.

Usually a missing or wrong N8N_ENCRYPTION_KEY, or the certs aren't readable. Run docker compose logs n8nq-main --tail 100 and look for "Could not decrypt" or "ENOENT" errors. Fix the .env or cert permissions and restart.

Workers won't connect to Redis.

The two Redis-password variables in .env must match. QUEUE_BULL_REDIS_PASSWORD (used by N8N) and REDIS_PASSWORD (used by the Redis container itself) need identical values. If they drift, workers will keep restarting with auth errors.

Schema apply fails with "permission denied" or "role does not exist".

This usually means the Postgres container didn't initialize with the right user. Check POSTGRES_USER and DB_POSTGRESDB_USER in .env match. If they don't, the easiest fix is to wipe the volume and start fresh: docker compose down -v then rerun step 6.

Port 5678 / 5433 / 6380 already in use.

Something else is bound to one of those ports. Either stop the conflicting service or change the host-side port in docker-compose.yml (e.g. "5679:5678" instead of "5678:5678"). Workers connect internally over the Docker network, so internal ports don't change.
