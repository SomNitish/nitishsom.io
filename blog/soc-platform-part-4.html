📙 PART 4 / 6 — The Backend API

The Backend That Makes the Database a Product

A FastAPI service that wraps the data layer, handles login, enforces permissions, syncs status to Jira, and pushes notifications. About 1,300 lines of Python. The goal: every action an analyst takes should go through one well-tested code path — never directly to the database.

Where We Left Off

End of Part 3: the database is solid. IDs are atomic. The audit log captures every state change. But the only way to interact with it is to docker exec -it into the Postgres container and write SQL by hand. That's fine for me at 2 AM debugging an outage. It's unusable for everyone else on the team.

Part 4 builds the layer that turns the database into a product. A REST API serving the dashboard, with proper login, sessions, permission checks, and integrations to the systems that already exist around it — primarily Jira.

Why FastAPI

I considered three options: Flask (familiar, mature), Django REST (too heavy for ~30 endpoints), and FastAPI. FastAPI won on four specific things:

  • Pydantic-style request validation built in. No serializer boilerplate. The function signature is the schema.
  • Dependency injection. The Depends() pattern is exactly the shape of "extract the current user from the session token, fail if invalid, otherwise pass it to the endpoint." More on this below.
  • Automatic OpenAPI docs. Visit /docs and you get an interactive API browser. Saved me writing a Postman collection.
  • Native async support. Useful when I needed background threads polling Redis without blocking the request loop.

Performance was a tie-breaker, not a decider. At our request volume (~5 RPS peak), Flask would work fine. The reason was developer ergonomics — the code reads cleanly, and that matters when I'm debugging an endpoint at midnight.

The File That Holds Everything

I chose to keep the entire backend in a single api.py file. About 1,300 lines. Some people would split this into routes/, services/, models/, auth/. I didn't, and I'd make the same choice again:

  • For a single-developer project, the cognitive overhead of jumping between 12 files outweighs the "clean architecture" benefit.
  • Ctrl-F across one file is fast. Searching across folders is slow.
  • When this codebase needs to be handed off, "open api.py, read top to bottom" is a friendlier instruction than "here's the project structure diagram."

The file is laid out top-down: imports → config → DB helpers → auth → permissions → endpoints → background threads → app startup. You can read it linearly and it tells you the whole story.

A few details worth pointing out:

  • role has a CHECK constraint. Database-level enforcement means you can't insert a typo like 'admin ' with a trailing space, or invent a phantom role like 'superadmin'. The constraint catches it before the row is written.
  • permissions JSONB — holds per-user overrides. We'll come back to this when we cover the permission model.
  • session_hours is nullable — when NULL the system uses the global default (1 hour). For specific accounts (a TV-monitor account that lives on a wall display) we set it to 8 or 12 hours so it doesn't log out every hour.
  • sessions.user_agent — the User-Agent header captured at login. This is what powers the single-session logic we'll see in a moment.

Login: bcrypt and a Session Token

Standard pattern. Receive username + password. Look up the user. Compare the password to the stored bcrypt hash. If it matches, create a session and return a token. The interesting part isn't the bcrypt — it's what happens with the session.

📄 api.py — login endpoint (core logic) 
from fastapi import FastAPI, HTTPException, Depends, Request
from datetime import datetime, timedelta, timezone
import bcrypt, secrets, json

app = FastAPI()
SESSION_HOURS_DEFAULT = 1

@app.post("/api/login")
def login(payload: dict, request: Request):
    username = (payload.get("username") or "").strip().lower()
    password = payload.get("password") or ""
    if not username or not password:
        raise HTTPException(400, "Username and password required")

    user = query(
        "SELECT id, password_hash, role, active, session_hours, full_name "
        "FROM users WHERE LOWER(username)=%s",
        [username], fetch="one"
    )
    if not user or not user["active"]:
        raise HTTPException(401, "Invalid credentials")

    if not bcrypt.checkpw(password.encode(), user["password_hash"].encode()):
        raise HTTPException(401, "Invalid credentials")

    user_id   = user["id"]
    hours     = user["session_hours"] or SESSION_HOURS_DEFAULT
    ua        = (request.headers.get("user-agent") or "")[:500]
    expires   = datetime.now(timezone.utc) + timedelta(hours=hours)

    token = handle_session(user_id, ua, expires)
    update("UPDATE users SET last_login=NOW() WHERE id=%s", [user_id])

    return {
        "token":     token,
        "user_id":   user_id,
        "name":      user["full_name"],
        "role":      user["role"],
        "expires":   expires.isoformat(),
    }

Standard up to the call to handle_session(). That function is where the interesting behavior lives.

Single Session, Same Browser Refresh

Most apps have one of two session models:

  • Multiple sessions allowed. Simple to implement. If someone steals a token, they use it concurrently with the legitimate user, and nobody notices.
  • One session per user, ever. Better security. But: every browser tab refresh on a fresh login kicks the previous tab out. Annoying.

I wanted a middle ground: one session per user, but logging in again from the same browser refreshes the existing session instead of replacing it. This is what enterprise SSO products do. About 30 lines of Python.

When the user logs in: Look for any existing session for this user Case 1: user_agent matches existing session → extend expires_at, keep the existing token → user stays logged in across tabs/refreshes Case 2: user_agent differs (or no existing session) → delete any existing session for this user → create a new session with a fresh token → the old tab gets booted on its next API call
📄 api.py — handle_session() 
def handle_session(user_id: int, user_agent: str, expires) -> str:
    """One session per user. Same browser = refresh existing. Different = replace."""
    existing = query(
        "SELECT token, user_agent FROM sessions WHERE user_id=%s",
        [user_id], fetch="one"
    )

    if existing and existing["user_agent"] == user_agent:
        # Same browser — extend the existing session
        update(
            "UPDATE sessions SET expires_at=%s WHERE user_id=%s",
            [expires, user_id]
        )
        return existing["token"]

    # Different browser (or no existing session) — replace it
    if existing:
        update("DELETE FROM sessions WHERE user_id=%s", [user_id])

    token = secrets.token_urlsafe(32)
    update(
        "INSERT INTO sessions (token, user_id, expires_at, user_agent) "
        "VALUES (%s, %s, %s, %s)",
        [token, user_id, expires, user_agent]
    )
    return token

The user-agent string isn't a security boundary — anyone who can steal a token can also spoof a User-Agent header. What it is: a usability boundary that makes the single-session rule survive normal browser behavior (refreshes, tab duplicates, idle-then-reactivate). Tokens are still rotated when the credentials are reused from a different machine.

The Dependency That Validates Every Request

This is where FastAPI's dependency injection earns its keep. I want every protected endpoint to start with: "extract the bearer token, find the session, verify it hasn't expired, return the user." Without DI, that's five lines copied into every endpoint. With DI, it's user=Depends(get_current_user) in the function signature, and everything else is automatic.

📄 api.py — get_current_user dependency 
from fastapi import Header

def get_current_user(authorization: str = Header(None)) -> dict:
    if not authorization or not authorization.startswith("Bearer "):
        raise HTTPException(401, "Missing or invalid Authorization header")

    token = authorization[7:].strip()
    row = query(
        "SELECT s.user_id, s.expires_at, u.username, u.full_name, "
        "       u.role, u.active "
        "FROM sessions s "
        "JOIN users u ON u.id = s.user_id "
        "WHERE s.token=%s",
        [token], fetch="one"
    )
    if not row:
        raise HTTPException(401, "Invalid session")

    if not row["active"]:
        raise HTTPException(403, "Account deactivated")

    if row["expires_at"] < datetime.now(timezone.utc):
        update("DELETE FROM sessions WHERE token=%s", [token])
        raise HTTPException(401, "Session expired")

    return {
        "id":        row["user_id"],
        "username":  row["username"],
        "full_name": row["full_name"],
        "role":      row["role"],
    }

Now every endpoint that needs an authenticated user looks like:

▸ how it's used in an endpoint 
@app.get("/api/me")
def me(user: dict = Depends(get_current_user)):
    return user

One word — Depends(get_current_user) — and the entire auth pipeline runs before the endpoint body. If anything fails, the endpoint never executes. The body becomes the business logic, not the plumbing.

The Permission Model

Now we're at the part of the system that has the most security responsibility. Sessions get you "is this a real user?" Permissions answer "is this user allowed to do this action?"

I built a three-role model with per-user overrides that can only restrict, never elevate. The defaults live in the application code (they change occasionally), the overrides live in the users.permissions JSONB column.

The "deny-only" override pattern

This is the bit that makes the model safe against misconfiguration. The function that resolves a user's effective permissions:

📄 api.py — get_user_permissions() 
def get_user_permissions(user_id: int, role: str) -> dict:
    """Effective permissions = role defaults, with overrides applied.
    Overrides can only RESTRICT (set to False), never elevate."""
    defaults = ROLE_DEFAULTS.get(role, ROLE_DEFAULTS["readonly"]).copy()

    row = query(
        "SELECT permissions FROM users WHERE id=%s",
        [user_id], fetch="one"
    )
    if not row or not row.get("permissions"):
        return defaults

    overrides = row["permissions"]
    if isinstance(overrides, str):       # in case psycopg2 returned it as JSON text
        overrides = json.loads(overrides)

    for key, value in overrides.items():
        if key in defaults and not value:
            defaults[key] = False          # the only valid override is False
    return defaults

Read the loop carefully:

for key, value in overrides.items():
    if key in defaults and not value:
        defaults[key] = False
  • if key in defaults — Ignore any override key that isn't already in the role defaults. You can't invent a new permission by writing it into the JSONB column.
  • and not value — Apply the override only when it's False. True overrides are silently dropped.

Why this matters: if a bug in the admin UI ever let someone write {"flush_queue": true} onto an analyst account, that analyst still cannot flush the queue. The role default is False, and the not value check rejects the True override. The worst a tampered permissions column can do is take permissions away — which is recoverable by removing the override. Privilege escalation through misconfiguration is structurally impossible.

Real use case: restricting a new analyst

Enforcing Permissions on Endpoints

Each protected endpoint needs to check whether the current user has a specific permission. Rather than calling get_user_permissions() inside every handler, I built a small dependency factory:

📄 api.py — require_permission factory 
def require_permission(perm: str):
    """Returns a dependency that enforces the named permission."""
    def _checker(user: dict = Depends(get_current_user)) -> dict:
        perms = get_user_permissions(user["id"], user["role"])
        if not perms.get(perm, False):
            raise HTTPException(403, f"Missing permission: {perm}")
        return user
    return _checker

Now any endpoint adds permission enforcement with one decorator-like declaration:

▸ usage 
@app.post("/api/incidents/{seq}/close")
def close_incident(
    seq: int,
    user: dict = Depends(require_permission("close_ticket"))
):
    # If we get here, the user is authenticated AND has 'close_ticket'.
    ...

The auth check, the permission check, and the user object — all set up before the function body runs. Inside the body, I never have to worry about whether the user is allowed. That's enforced at the door.

A Real Endpoint: Adding a Comment

To show how all the pieces fit, here's a complete endpoint. Adds a comment to an incident, writes to the audit log, pushes a notification to whoever is assigned.

📄 api.py — add comment endpoint 
@app.post("/api/incidents/{seq}/comments")
def add_comment(
    seq: int,
    payload: dict,
    user: dict = Depends(require_permission("add_comment"))
):
    body = (payload.get("body") or "").strip()
    if not body:
        raise HTTPException(400, "Comment body required")

    # Verify the incident exists and grab its assigned analyst + incident_id
    incident = query(
        "SELECT incident_id, assigned_to FROM incidents WHERE seq=%s",
        [seq], fetch="one"
    )
    if not incident:
        raise HTTPException(404, "Incident not found")

    # Write the comment
    update(
        "INSERT INTO incident_comments (incident_seq, author, body) "
        "VALUES (%s, %s, %s)",
        [seq, user["full_name"], body]
    )

    # Audit log
    audit(seq, "comment_added", user["full_name"], "", body[:200])

    # Notify the assignee, if they're a registered user and not the author
    if incident["assigned_to"] and incident["assigned_to"] != user["full_name"]:
        push_notif(
            assignee_username=incident["assigned_to"],
            title=f"New comment on {incident['incident_id']}",
            message=f"{user['full_name']}: {body[:120]}",
            incident_seq=seq,
        )

    return {"ok": True}

Twenty lines. One endpoint. Five things it does: validates input, looks up the incident, writes the comment, audit-logs the action, notifies the assignee. Auth and permission are handled by the dependency — they don't show up in the body at all.

The Jira Sync

When an analyst closes a ticket in our dashboard, the corresponding Jira issue (the one created in the original automation flow from Part 1) needs to follow. Same for status changes, assignee changes, and added comments.

I considered three patterns:

  1. Synchronous Jira calls inside the endpoint. Simple, but a slow Jira API response would block the user.
  2. A background queue that picks up sync jobs. Most robust, but overkill for ~50 syncs/day.
  3. Fire-and-forget via Python's threading. The endpoint returns immediately; sync happens in the background. If Jira fails, we log it and an admin investigates.

I chose option 3. Here's the sync function and how it's called:

📄 api.py — Jira sync 
import httpx, threading

JIRA_BASE = os.environ.get("JIRA_BASE")           # https://acme.atlassian.net
JIRA_AUTH = (os.environ.get("JIRA_EMAIL"),
             os.environ.get("JIRA_API_TOKEN"))

# Map our internal status -> Jira transition ID (configured per project)
JIRA_TRANSITIONS = {"resolved": "31", "closed": "41", "reopened": "11"}

def jira_sync_status(jira_key: str, new_status: str, actor: str, note: str = ""):
    """Push a status change to Jira. Runs in a background thread."""
    if not jira_key:
        return
    transition_id = JIRA_TRANSITIONS.get(new_status)
    if not transition_id:
        return

    try:
        with httpx.Client(auth=JIRA_AUTH, timeout=15.0) as c:
            # 1. Transition the issue
            r = c.post(
                f"{JIRA_BASE}/rest/api/3/issue/{jira_key}/transitions",
                json={"transition": {"id": transition_id}}
            )
            r.raise_for_status()

            # 2. Add a comment recording who did it on our side
            if note or actor:
                c.post(
                    f"{JIRA_BASE}/rest/api/3/issue/{jira_key}/comment",
                    json={"body": {
                        "type": "doc", "version": 1,
                        "content": [{"type": "paragraph", "content": [
                            {"type": "text",
                             "text": f"[Dashboard] {actor} -> {new_status}. {note}"}
                        ]}]
                    }}
                )
    except Exception as e:
        log_error(f"jira_sync_status failed for {jira_key}: {e}")

def jira_sync_status_bg(jira_key, new_status, actor, note=""):
    """Fire-and-forget wrapper. Always use this from endpoints."""
    t = threading.Thread(
        target=jira_sync_status,
        args=(jira_key, new_status, actor, note),
        daemon=True,
    )
    t.start()

And from the close-incident endpoint:

▸ kick off the sync after closing locally 
@app.post("/api/incidents/{seq}/close")
def close_incident(
    seq: int,
    payload: dict,
    user: dict = Depends(require_permission("close_ticket"))
):
    note = (payload.get("note") or "").strip()
    incident = query("SELECT jira_key FROM incidents WHERE seq=%s", [seq], fetch="one")
    if not incident:
        raise HTTPException(404, "Incident not found")

    # Local close + audit
    update("UPDATE incidents SET status='resolved', resolved_at=NOW() WHERE seq=%s", [seq])
    audit(seq, "status_change", user["full_name"], "open", "resolved")

    # Sync to Jira in the background — response returns immediately
    jira_sync_status_bg(incident["jira_key"], "resolved", user["full_name"], note)

    return {"ok": True}

The endpoint returns in ~50 ms. The Jira call happens in the background. If Jira is slow, the analyst doesn't feel it.

What about Jira failures?

If the Jira API call fails (network blip, auth issue, Atlassian outage), we log the error and the local state and Jira state are now out of sync. That's a real risk. The mitigation:

  • Every action is logged to audit_log before the Jira call. We have a complete record locally.
  • A reconciliation script runs nightly, finds incidents where the local status doesn't match Jira, and either retries the sync or flags them in the admin UI for manual review.
  • The audit log captures who made the change locally, so even if Jira is out of date for a few hours, we know what should be there.

I've had two Jira outages since launch. Both times, the reconciliation script caught up on its own once Atlassian was back. No data lost, no analyst manually fixing things.

The Notifications Table and Helper

In-app notifications power the bell icon in the dashboard. When something happens that a specific user (or all users) should know about, we write a row.

▸ notifications table 
docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue <<'EOF'
CREATE TABLE IF NOT EXISTS notifications (
  id           SERIAL PRIMARY KEY,
  user_id      INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type         TEXT NOT NULL DEFAULT 'system',
  incident_seq INTEGER,
  incident_id  TEXT DEFAULT '',
  title        TEXT NOT NULL,
  message      TEXT NOT NULL DEFAULT '',
  from_user    TEXT DEFAULT '',
  read         BOOLEAN NOT NULL DEFAULT FALSE,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()

  ON notifications(user_id) WHERE read = FALSE;
EOF

The partial index on the unread predicate is a small but meaningful optimization. The most common notification query is SELECT COUNT(*) FROM notifications WHERE user_id=? AND read=FALSE, polled every 10 seconds per logged-in user. The partial index keeps only the unread rows — typically a few per user — so the count is constant-time.

The helper that writes notifications:

📄 api.py — push_notif() 
def push_notif(*, assignee_username: str = None, title: str, message: str,
               incident_seq: int = None, ntype: str = "assignment",
               from_user: str = ""):
    """Send a notification to one specific user (by username)."""
    if not assignee_username:
        return
    target = query(
        "SELECT id FROM users WHERE LOWER(full_name)=LOWER(%s) OR username=%s",
        [assignee_username, assignee_username], fetch="one"
    )
    if not target:
        return
    update(
        "INSERT INTO notifications (user_id, type, incident_seq, title, message, from_user) "
        "VALUES (%s, %s, %s, %s, %s, %s)",
        [target["id"], ntype, incident_seq, title, message[:500], from_user]
    )

def broadcast_notif(*, title: str, message: str, ntype: str = "system"):
    """Send to every active user. Used for worker-health alerts."""
    update(
        "INSERT INTO notifications (user_id, type, title, message) "
        "SELECT id, %s, %s, %s FROM users WHERE active=TRUE",
        [ntype, title, message[:500]]
    )

The dashboard polls GET /api/notifications/unread_count every 10 seconds. When the number is non-zero, the bell icon turns red. Click the bell, the user sees a dropdown list of unread notifications. Click one, it marks read.

I considered WebSockets for "real" real-time push. I didn't go that route — polling is simpler, the dashboard already polls for queue metrics every 5 seconds anyway, and a 10-second delay on "you have a new notification" is invisible to the user. Operational simplicity won.

Background Threads

Two background tasks run inside the FastAPI process — not as separate cron jobs, not as Celery workers, just threading.Thread(daemon=True) started on app boot.

Worker health watcher

Every 30 seconds, count how many N8N worker containers are running. If the count hits zero for more than 5 minutes, broadcast a notification to every active user. When workers come back, broadcast a recovery notification.

📄 api.py — worker_health_watcher 
import subprocess

def count_running_workers() -> int:
    try:
        out = subprocess.check_output(
            ["docker", "compose", "-f", COMPOSE_FILE, "ps", "--format", "json"],
            text=True, timeout=10,
        )
        return out.count('"Service":"n8nq-worker"')
    except Exception:
        return -1   # unknown

def worker_health_watcher():
    """Background thread. Notifies on full outage (5+ min at 0 workers)."""
    last_state = "healthy"
    zero_since = None

    while True:
        try:
            n = count_running_workers()
            now = datetime.now(timezone.utc)

            if n == 0 and zero_since is None:
                zero_since = now
            elif n > 0:
                zero_since = None
                if last_state == "down":
                    broadcast_notif(
                        title="Workers recovered",
                        message=f"{n} N8N worker(s) back online.",
                        ntype="worker_recovery",
                    )
                    last_state = "healthy"

            if zero_since and (now - zero_since).total_seconds() > 300:
                if last_state != "down":
                    broadcast_notif(
                        title="Workers DOWN",
                        message="No N8N workers have been running for 5 minutes. "
                                "Alerts are not being processed.",
                        ntype="worker_down",
                    )
                    last_state = "down"
        except Exception as e:
            log_error(f"worker_health_watcher tick: {e}")

        time.sleep(30)

@app.on_event("startup")
def start_background_threads():
    threading.Thread(target=worker_health_watcher, daemon=True).start()
    threading.Thread(target=queue_snapshot_writer,  daemon=True).start()

Queue snapshot writer

Same pattern. Every 30 seconds, ask Redis how deep the queue is, write a row to a queue_snapshots table. This becomes the historical chart in Part 5's dashboard — "here's the queue depth over the last 24 hours."

Beyond the SOC: Where This Pattern Applies

Same backend shape — different domain

  • Internal tools at small companies. Five to fifty users, custom workflows, no budget for off-the-shelf SaaS. Single-file FastAPI + Postgres + bcrypt is enough.
  • Customer admin portals for B2B products. The "deny-only override" permission pattern is a good fit when you have customer-tier roles plus optional restrictions on individual customer users.
  • SaaS dashboards layered over an existing database. When the data already lives somewhere, an API that wraps it with auth + audit + integrations is often what's actually needed.
  • Operations dashboards for infra/ML teams. Background-thread health watchers and queue snapshot writers are the exact pattern for "alert when the GPU cluster goes down" or "chart the inference queue depth."
  • Any system that needs to sync state to a third-party tool. The fire-and-forget threading + reconciliation script pattern works for Slack, Linear, Notion, GitHub — anywhere you can't afford to block on slow external calls.

The backend's job isn't to be clever. It's to be the single, well-tested path between humans and data.
