📓 PART 6 / 6 — Keeping It Alive

The Boring Part That Turns a Project Into a System

Building the platform was the fun half. Keeping it alive — backups that actually restore, monitoring that catches things before users do, postmortems that change behaviour — is the half that matters more. Six months in production, two outages, every lesson written down.

Where We Left Off

Parts 1 through 5 built the platform. Workflows running, IDs atomic, API locked down, dashboard live. Everything works on a good day.

This part is about the bad days. The Saturday morning when the workers crashed and nobody noticed for three hours. The migration that locked the database for four minutes. The Wednesday afternoon when the Postgres container hit its connection limit. The disk that quietly filled with old execution logs until N8N couldn't write anymore.

Every one of those things happened. Every one of them taught me something. This part is the cumulative version of those lessons — the routine I run now, the tooling I added afterwards, and the two postmortems that taught me the most.

The Morning Routine

Every morning at 9 AM IST I run one command. It takes about three seconds and tells me whether anything is wrong before I open the dashboard.

📄 /opt/security-automation/morning_check.sh 
#!/bin/bash
# 9 AM sanity check. Output should fit on one screen.

set -e
cd /opt/security-automation
source .env

echo "=== Containers ==="
docker compose ps --format "table {{.Name}}\t{{.Status}}\t{{.Ports}}"

echo
echo "=== Disk (/opt) ==="
df -h /opt | awk 'NR==1 || /\/opt/'

echo
echo "=== Postgres ==="
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -t -c \
  "SELECT
     (SELECT COUNT(*) FROM incidents WHERE created_at > NOW() - INTERVAL '24 hours')
       AS incidents_24h,
     (SELECT COUNT(*) FROM pg_stat_activity WHERE datname='n8n_queue')
       AS connections,
     (SELECT pg_size_pretty(pg_database_size('n8n_queue')))
       AS db_size;"

echo
echo "=== Redis ==="
docker exec n8nq-redis redis-cli -a "$REDIS_PASSWORD" --no-auth-warning \
  INFO memory | grep -E "used_memory_human|maxmemory_human"
docker exec n8nq-redis redis-cli -a "$REDIS_PASSWORD" --no-auth-warning \
  LLEN bull:jobs:wait | awk '{print "waiting jobs: " $1}'

echo
echo "=== Latest backup ==="
ls -lh /var/backups/soc-platform/postgres-*.sql.gz 2>/dev/null | tail -1

echo
echo "=== SSL cert expiry ==="
openssl x509 -in certs/cert.pem -enddate -noout

Six checks, three seconds, one screen of output. If everything is normal I know in five seconds and move on. If something is off — disk over 80%, connections climbing, no recent backup — I see it before I've even started the workday.

This is more valuable than any dashboard. Dashboards make you look. Scripts make you check.

Why I Don't Run Prometheus

The default advice for monitoring would be: install Prometheus, install Grafana, scrape metrics from the host, build alert rules, set up Alertmanager. That's the proper way. I considered it. I didn't do it.

Reasons:

  • Prometheus + Grafana is more code than the platform. For a single-host setup with ~20 users, the monitoring stack would have a bigger operational surface than the thing being monitored. Wrong ratio.
  • The dashboard already does most of it. Queue depth chart, worker count, recent incidents — all rendered live from data the API already serves. Adding Grafana would be a second window showing the same numbers.
  • Alert fatigue. Every monitoring system I've worked with eventually fires too many low-signal alerts. The 4-condition queue-stuck overlay from Part 5 fires only when something is actually wrong. That's better than 20 Grafana panels that I've muted because they cried wolf.
  • If the platform grows past this size, Prometheus becomes the right answer. We're not there yet.

What I have instead: the morning script, the dashboard, the queue-stuck overlay, and a few cron jobs that catch the boring stuff before it bites.

The Cron Jobs That Catch Boring Things

Three cron jobs run on the host. None of them are clever. All of them have saved me at least once.

📄 /etc/cron.d/soc-platform 
# 1. Backup every 6 hours
0 */6 * * * root /opt/security-automation/backup.sh >> /var/log/soc-backup.log 2>&1

# 2. Disk usage alert (email if /opt > 80%)
*/15 * * * * root /opt/security-automation/disk_alert.sh

# 3. SSL cert expiry alert (email 14 days before expiry)
0 9 * * * root /opt/security-automation/cert_alert.sh

The disk check is the simplest possible thing:

📄 disk_alert.sh 
#!/bin/bash
USED=$(df /opt | awk 'NR==2 {gsub(/%/,"",$5); print $5}')
if [ "$USED" -gt 80 ]; then
  echo "Disk usage on /opt is at ${USED}%" | \
    mail -s "[SOC-PLATFORM] Disk warning" admin@company.com
fi

Eight lines. The first time it fired, I had three weeks before /opt would have filled. That's three weeks to clean up old execution data, archive old logs, or expand the disk — instead of three minutes between "everything is fine" and "the platform is down."

Backups (And Why I Test Them)

Every six hours a backup script runs. It dumps Postgres, copies the encryption key and .env, and rotates old backups. Standard stuff. The detail that matters is the restore drill.

📄 /opt/security-automation/backup.sh 
#!/bin/bash
set -euo pipefail
cd /opt/security-automation
source .env

TS=$(date +%Y%m%d-%H%M%S)
DEST=/var/backups/soc-platform
mkdir -p "$DEST"

# 1. Postgres dump
docker exec -e PGPASSWORD="$POSTGRES_PASSWORD" n8nq-postgres \
  pg_dump -U n8n_queue -d n8n_queue --no-owner --clean --if-exists \
  | gzip > "$DEST/postgres-$TS.sql.gz"

# 2. Critical config (encryption key lives in here)
cp .env "$DEST/env-$TS"
cp -r certs "$DEST/certs-$TS"

# 3. Rotate: keep daily for 30 days, weekly older
find "$DEST" -name "postgres-*.sql.gz" -mtime +30 -delete
find "$DEST" -name "env-*" -mtime +30 -delete
find "$DEST" -name "certs-*" -mtime +30 -type d -exec rm -rf {} +

# 4. Verify size sanity (catches dumps that produced empty output)
SIZE=$(stat -c%s "$DEST/postgres-$TS.sql.gz")
if [ "$SIZE" -lt 100000 ]; then
  echo "Backup is suspiciously small: $SIZE bytes" \
    | mail -s "[SOC-PLATFORM] Backup small" admin@company.com
fi

echo "[$(date)] Backup OK: $DEST/postgres-$TS.sql.gz ($SIZE bytes)"

The restore drill

Every month I run a restore against a throwaway container. Backup files that haven't been tested aren't backups — they're hope.

▸ monthly restore drill 
# Pick the most recent backup
BACKUP=$(ls -t /var/backups/soc-platform/postgres-*.sql.gz | head -1)

# Spin up a throwaway Postgres
docker run --rm -d --name pg-restore-test \
  -e POSTGRES_PASSWORD=test -e POSTGRES_DB=restore_test \
  postgres:16-alpine

sleep 5  # let it boot

# Restore into it
gunzip -c "$BACKUP" | \
  docker exec -i pg-restore-test psql -U postgres -d restore_test

# Verify counts
docker exec pg-restore-test psql -U postgres -d restore_test -c "
  SELECT 'incidents:' AS t, COUNT(*) FROM incidents
  UNION ALL
  SELECT 'audit_log:' , COUNT(*) FROM audit_log
  UNION ALL
  SELECT 'users:'     , COUNT(*) FROM users;
"

# Clean up
docker stop pg-restore-test

If the row counts look right, I have a restore in my pocket. If they don't, I know before the day I actually need it.

The one file the backup script copies that's more important than the database itself: .env.

It contains N8N_ENCRYPTION_KEY. If you restore Postgres without that key, every credential stored inside N8N workflows is unrecoverable — Jira tokens, OAuth refresh tokens, integration passwords. The database dump is the table data; the .env is the key that decrypts it. I learned this the hard way once. Now they always travel together.

The Troubleshooting Cookbook

Six months of running this platform produced a recurring set of failures. Each one has a familiar shape. I keep these as a runbook — symptom on top, command underneath.

Workers won't start after a deploy

Almost always a mismatch in N8N_ENCRYPTION_KEY between main and workers, or a credential file from a previous install lingering in the data volume.

▸ check 
docker compose logs n8nq-worker --tail 100 | grep -i "decrypt\|encryption\|credential"
# Look for "Could not decrypt" — that's the encryption-key mismatch signature
Queue depth is climbing but no errors visible

Workers are alive but stuck on something — usually a slow external API (Jira during one of Atlassian's bad days). Check active job count.

▸ check 
docker exec n8nq-redis redis-cli -a "$REDIS_PASSWORD" --no-auth-warning LLEN bull:jobs:active
# If active > 0 but waiting climbing, workers are busy on long-running jobs
# If active == 0 and waiting climbing, workers are dead -> restart them
"Postgres connection refused" from the API logs

Either the container is down, or it's hit max connections. The second one happens when a code path forgot to release a connection — usually a recent change.

▸ check 
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue -c "
  SELECT state, COUNT(*) FROM pg_stat_activity
  WHERE datname='n8n_queue'
  GROUP BY state;
"
# If 'idle' is huge, something is holding connections without using them.
# Restart the dashboard API first; if that fixes it, look at recent code changes.
Dashboard shows old data, refresh isn't working

Polling loop probably hit an unhandled exception. Open browser console — there will be an error from the api() wrapper. If the console is clean, the API itself is wedged.

▸ check 
systemctl status dashboard
tail -50 /opt/security-automation/dashboard/api.log
# If the log shows fastapi exceptions, the API is broken but running.
# If it's silent, restart: systemctl restart dashboard
Jira sync silently failing

Background thread errors don't surface to the dashboard. Have to check the API log directly.

▸ check 
grep "jira_sync_status failed" /opt/security-automation/dashboard/api.log | tail -20
# Most common: API token expired. Rotate in app_config table and restart.
# Second most common: Atlassian Cloud rate limit (429). Wait and let the
# reconciliation script catch up overnight.
Containers won't pull on update — "no space left on device"

Old Docker images and dangling volumes. Happens every few weeks if you update images regularly.

▸ fix 
docker system df                              # see what's eating space
docker image prune -f                         # untagged images
docker container prune -f                     # stopped containers
docker volume prune -f                        # NOT n8nq_pgdata etc -- those are still in use

Postmortem: The 4-Minute Migration Outage

I referenced this one in Part 3. Here's the full version, because the lesson generalizes well beyond schema migrations.

🔥 PM-001: ALTER TABLE blocks all inserts for 4 minutes

Date:March 2026
Duration:4 min, 12 sec
Impact:~62 alerts queued, 0 lost
Severity:Medium
T+0:00 Ran ALTER TABLE incidents ADD COLUMN email_sent BOOLEAN NOT NULL DEFAULT FALSE T+0:15 N8N worker logs start showing "relation is locked" T+0:30 Queue depth begins climbing rapidly T+1:10 Autoscaler ramps from 10 -> 18 workers (new workers also block) T+2:00 Dashboard API connections to Postgres time out T+2:05 Queue-stuck overlay fires on the SOC TV monitor T+4:12 ALTER TABLE completes. Locks release. T+4:45 Workers drain the queue T+5:30 System back to normal
Root cause
Postgres treats ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT <value> as a full table rewrite on older versions and on certain default expressions. With ~4,000 incident rows it took 4 minutes of exclusive lock. During that window every INSERT to the table queued behind the lock.
What I changed
  • Every migration now splits into three statements: nullable add, batched backfill, then tighten constraints. None of those hold a long lock.
  • Migrations now run only during the SOC's quiet window (3-5 AM IST), monitored manually.
  • The morning script's row-count checks would have caught the column shape before; now they also sanity-check schema version against a known list.
  • The queue-stuck overlay was already in place — it correctly told the analyst something was wrong. The fix shifted left: stop letting bad migrations reach prod.
What I'd do differently next time
Test the migration against a copy of production data. The local test database had 4 rows in incidents — the ALTER ran instantly there. Production had 4,000. I should have known to size-up before running.

Postmortem: The Saturday Worker Crash

The one that gave us the queue-stuck overlay. Worth telling in full now.

🔥 PM-002: Workers crashed at 6 AM, found at 9:30 AM

Date:April 2026
Duration:3 hours, 28 minutes
Impact:347 alerts delayed, 0 lost
Severity:High
06:02 Host kernel hit an OOM condition affecting cgroup memory accounting 06:02 All 10 n8nq-worker containers terminated by the kernel 06:03 Docker tried to restart them. Restart failed for the same kernel reason. 06:03 Autoscaler script tried to scale up; got back failed-restart errors 06:05 Queue starts climbing. No one watching. 09:32 Analyst pings me on Slack about missed alerts 09:34 I find docker daemon stuck. systemctl restart docker. 09:36 Workers come up. Queue drains in 4 minutes. 09:40 All 347 queued alerts have tickets. Customer notifications go out.
Root cause
A kernel bug in the host's cgroup memory subsystem corrupted memory accounting under specific load patterns. When that fired at 6 AM, every container N8N spawned was killed instantly. Docker daemon kept retrying but the kernel state was bad. A reboot would have fixed it; a daemon restart eventually did.
What I changed
  • Built the queue-stuck overlay (Part 5). Same incident today would alert me within 2 minutes — not 3.5 hours.
  • Added a worker-health watcher to the API process (Part 4). When workers are at zero for >5 minutes, every active user gets a broadcast notification. The SOC TV monitor in particular makes this very visible.
  • Wrote a runbook entry for "Docker daemon stuck." The fix is systemctl restart docker — but only if you know that's an option.
  • Upgraded the host kernel. The bug had a fix in a newer kernel. We hadn't taken it.
What this taught me about monitoring
The autoscaler was technically working — it tried to scale up. The container healthchecks were technically running — they reported failure. Nothing told a human. Every monitoring decision after this incident has been pointed at the same question: when the platform breaks, how does a human find out within minutes, not hours?

Six Months In: What I'd Tell Myself on Day One

If I could send notes back to the version of me starting this project, here's what they'd say.

1. Start with the boring stuff first

I built the workflow, then the database, then the API, then the dashboard. Then I started thinking about backups, monitoring, and runbooks. That ordering is backwards. The first thing you should build after "it runs" is "it survives." Two days I spent on a Saturday-night restore drill would have saved a week of anxiety the first time something broke.

2. Test migrations against production-shaped data

The 4-minute outage was entirely avoidable. A test database with one realistic-volume table would have shown the lock-up immediately. Now I run a docker compose stack with a copy of last night's backup whenever I'm about to ship something schema-touching.

3. Logs are the single most important thing

Every postmortem I wrote was reconstructed from logs. autoscale.log, api.log, audit_log in Postgres, docker compose logs. The instinct to "clean up logs to save disk" is wrong. Buy more disk. Keep logs longer.

4. Polling is fine. Almost always.

I spent a few hours early on prototyping WebSocket support for the dashboard. After running the polling version in production for six months, I haven't once wished for WebSockets. The bandwidth is negligible, the simplicity is invaluable. Reach for fancy infrastructure only when fancy infrastructure actually solves a problem.

5. The most-used feature is the one you almost didn't build

The queue-stuck overlay took me half a Sunday to build. It has fired meaningfully twice. Both times, it cut response time from "hours" to "minutes." It's the highest-leverage code in the entire system. The features that seem optional during the build are usually the ones that matter most when the platform breaks.

6. Write the runbook the first time you fix the thing

Not the third time. Not next week. The minute you fix something at 2 AM, type the command into a file. The cookbook section above didn't exist for the first three months. Every entry is "the third time I had to figure this out from scratch."

Beyond the SOC: Where This Pattern Applies

Same operational shape — any small-to-medium production system

  • Self-hosted SaaS replacements. Single-host or small-cluster deployments of self-hosted tools (Gitea, Plausible, Bitwarden, Outline) — the morning-check + cron + backup-drill model fits all of them.
  • Internal microservices for small teams. When your "production environment" is a handful of servers, lightweight ops practices beat enterprise observability stacks every time.
  • Side projects that grew up. The hobby thing that's now in production at work. This series is essentially a template for "the platform grew. now what?"
  • Bootstrapped startups. Before the team is big enough to justify a dedicated SRE, this is roughly the operational footprint that keeps a small product alive at low cost.

The fun part is building it. The important part is making sure that when it breaks at 6 AM, a human knows about it by 6:02.

One Last Thought

When I started this project, I wasn't trying to build a SOAR platform. I was trying to stop typing the same Jira ticket fifty times a day.

The lesson I take from six months of running this thing isn't about N8N, or Postgres triggers, or FastAPI, or any of the technical pieces. It's that the gap between "annoying manual workflow" and "automated system that the team uses every day" is bridgeable — and the path across that gap is shorter than it looks if you start with the work, not the tools.

Pick the most painful manual task you do every week. Sketch what the automated version would look like. Build the smallest possible version of it. Run it. Find what breaks. Fix it. Repeat. Six months from now, you'll have a platform.

That's the whole series, compressed into a paragraph. Thanks for reading.
