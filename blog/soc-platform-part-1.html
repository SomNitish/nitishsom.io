Building a Production SOC Automation Platform

From a SQLite-based N8N hobby setup to a multi-tenant SOAR platform processing 5,000+ incidents/month — with auto-scaling workers, custom FastAPI backend, real-time dashboard, and zero data loss.

The Project

Over the past several months, I designed and built a complete SOC automation platform from the ground up. Not a tutorial, not a proof-of-concept — a real production system that handles multi-tenant security operations for two separate client environments, processes thousands of alerts per month, and runs 24/7 without manual intervention.

This 6-part series documents how I built it, the architectural decisions I made, the problems I encountered, and the lessons I learned. It covers every layer of the stack: infrastructure, database, backend, frontend, monitoring, and operations.

The goal wasn't just to make alerts flow from Wazuh into Jira. The goal was to build a platform an analyst would actually want to use — fast, reliable, transparent, and self-healing.

The Problem I Was Solving

The starting point was a simple N8N instance running in a Docker container with SQLite. It worked for low-volume automation — receiving Wazuh alerts, formatting them, creating Jira tickets, sending emails. About 20-30 workflows per day.

Then alert volume started growing. We added a second client environment. We added SentinelOne EDR alerts on top of Wazuh. We added scheduled compliance workflows. We added monthly reporting. Within weeks, the single-threaded N8N instance started showing cracks:

❌ Race conditions in incident ID generation
When 3 alerts hit the system within the same second, they would all generate the same incident ID. We'd see IR-202604-007 appear three times in the database.

❌ Workflow backlog
Single-threaded execution meant a 90-second compliance report would block 50 incoming alerts. By the time the report finished, those alerts were 2 minutes stale.

❌ No visibility
N8N's built-in UI shows executions, but not incidents. Our analysts needed a dashboard that showed: which alerts are open, who's working on them, what's the MTTR trend, which IPs are repeating. We were running queries against Postgres manually.

❌ No RBAC
Anyone with the N8N URL could see all workflows and credentials. We needed proper role-based access control, audit logging, session management — none of which N8N provides out of the box.

❌ Manual incident tracking
Every time an analyst closed a ticket, they had to remember to update Jira separately. Comments lived in three places (N8N execution data, Jira, Slack). Audit trails were nonexistent.

❌ No failure detection
When a worker crashed, we'd find out hours later when someone noticed alerts piling up. There was no monitoring, no alerting, no self-recovery.

I could have bought a commercial SOAR platform. The cheapest options started at $40,000/year for a single tenant. The features I needed — atomic incident IDs, per-customer monthly resets, custom dashboards, multi-tenant isolation — required enterprise tiers running $150K+ per year.

So I built it.

Why Vanilla JS for the Dashboard?

I made a deliberate decision to build the frontend without React, Vue, or any framework. Here's why:

  • Zero build step. I can SSH into the server, edit HTML directly, reload the browser. For a security dashboard that needs hotfixes, this matters.
  • One file, one page. Each dashboard page is self-contained. I can hand off queue.html to a colleague and they can understand the entire page in 15 minutes.
  • No dependency hell. When npm audit flags a vulnerability in a transitive dependency three months from now, I don't care. There are no transitive dependencies.
  • Performance. The whole dashboard loads in under 200ms. No virtual DOM, no hydration, no bundle.
  • Skills demonstration. Modern JS is powerful. Fetch, async/await, template literals, CSS Grid — you can build a sophisticated dashboard without React. I wanted to prove that.

The tradeoff: when I need a complex interactive component, I have to write it myself. The queue stuck detection overlay, the notification system, the modal stack — all hand-written. But the resulting code is small, fast, and maintainable.

Key Features I Built

1. Atomic Incident ID Generation

A PostgreSQL trigger that generates incident IDs at the database level — guaranteed unique, guaranteed sequential, automatically resets monthly per customer. No race conditions possible.

2. Auto-Scaling N8N Workers

A bash daemon monitors Redis queue depth every 5 seconds and scales worker containers between 10 and 50 instances based on load. Cooldown prevents thrashing. The system literally heals itself during alert storms.

3. Role-Based Access Control with Per-User Overrides

Three roles (admin, analyst, readonly) define the ceiling of what users can do. Admins can restrict individual users below their role's defaults but never elevate them. This pattern prevents privilege escalation through misconfiguration.

4. Single-Session Enforcement

When the same user logs in from a different browser, the old session dies. When they log in from the same browser, the existing session refreshes. Same security as enterprise SSO, but built into 50 lines of Python.

5. Soft Delete with Approval Workflow

Non-admins can request ticket deletion with a reason. Admins get notified, review, and approve or reject. Approved deletions are soft — the row is hidden but preserved in the database. The audit trail is immutable.

6. Real-Time Queue Stuck Detection

The dashboard polls Redis every 30 seconds. If jobs are waiting but no workers are active for more than 2 minutes, a full-screen alert appears with: live stats, the affected workflow, an action checklist, and a pre-drafted email ready to copy and send to engineering.

7. Worker Health Monitoring

A background thread inside the FastAPI process checks worker count every 30 seconds. If all workers are down for 5+ minutes, every active user gets an in-app notification. When workers recover, a recovery notification fires.

8. Comprehensive Audit Trail

Every state change, every login, every comment, every permission change is logged with: who did it, when, from which IP, what changed. The audit log is append-only — even admins can't modify history.

9. Queue Flush with Forensic Logging

When the queue gets corrupted (rare, but happens), admins can flush it. But every flush is logged with: who flushed it, what execution IDs were affected, how many Wazuh alerts were dropped. The audit log lets us go back to Wazuh and recheck those alerts manually.

10. MTTD / MTTA / MTTR Calculations

Mean Time To Detect, Acknowledge, Resolve — calculated on every ticket from real timestamps. Analyst performance metrics are derived from these. The numbers are honest; nobody can fudge them because they come from the database itself.

What Makes This Different

There are plenty of "I built a SOAR with N8N" projects on GitHub. Most of them stop at "I made a webhook receiver." This one goes further:

This is production-grade. It runs 24/7 in a real SOC, handling real alerts, used by real analysts. Every decision was forced by reality, not by a tutorial.

This is multi-layered. The N8N piece is maybe 30% of the work. The other 70% — database design, RBAC, dashboard, monitoring, operations — is where real production systems differ from demos.

This is multi-tenant. Most "build your own SOAR" tutorials handle one tenant. Real MSSP work means complete isolation between clients, separate ID sequences, separate dashboards, separate Jira projects.

This is honest about failures. I'll show you the bugs I shipped, the late-night incidents, the database queries I had to write to recover from my own mistakes. This isn't a victory lap; it's a real engineering story.

The 12-Table Schema (Preview)

I'll go deep on the database in Part 3, but here's the table list to give you a sense of scope:

users                  -- dashboard login accounts (admin/analyst/readonly)
sessions               -- active login sessions with browser fingerprinting
incidents              -- the core table (4,800+ rows and growing)
incident_month_seq     -- per-tenant monthly counter for ID generation
incident_comments      -- investigation notes attached to incidents
audit_log              -- immutable record of every action
notifications          -- in-app notification system
analysts               -- ticket assignment dropdown (separate from users)
app_config             -- platform settings (Jira credentials etc.)
user_activity_log      -- per-user action audit (admin-visible)
queue_snapshots        -- Redis queue depth history (every 30s)
queue_flush_log        -- record of every manual queue flush

Each table has a specific purpose. There's no kitchen-sink "everything" table. The schema took several iterations to get right, and Part 3 walks through every column and the reasoning behind it.
