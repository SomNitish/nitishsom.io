📘 PART 1 / 6 — The Problem

Why I Built a SOC Automation Platform

Most engineering write-ups open with the architecture. This one opens with the work. Here's what a SOC analyst actually does all day, the part of that day I wanted to disappear, and the workflow I set out to build.

A Day in My Job

I work as a SOC analyst — Security Operations Center. The short version of the job: I watch what's happening across our customers' networks (laptops, servers, cloud accounts, identity providers) and respond to anything that looks suspicious.

The data flows into a SIEM — a Security Information and Event Management platform. We use Wazuh. The SIEM runs detection rules against everything it sees and turns the suspicious stuff into alerts:

  • "User j.smith just logged into the bastion from an IP we've never seen"
  • "500 failed SSH attempts in 60 seconds on a production server"
  • "PowerShell launched with arguments that match known malware"
  • "A file with a known-bad hash just touched disk on an endpoint"

An alert isn't an answer. It's a question. The SIEM is saying: this might be bad — please look at it. My job is to figure out whether it really is bad, and if so, do something about it.

What "Doing Something" Looks Like — Manually

Here's what happens, step by step, when one of those alerts hits the queue. This is the manual workflow we used to follow:

1

Read the alert

Open the SIEM, find it in the queue. Note source IP, target server, username, timestamp.

~30 seconds
2

Check if it's expected

Is the source IP on the whitelist? Is it a Jenkins runner? A known bastion? Cross-reference against a wiki page or spreadsheet.

~1-2 minutes
3

Decide: legitimate, or real?

If whitelisted with the right owner, mark benign and move on. If not, this is a real incident — keep going.

~30 seconds
4

Create the ticket in Jira

Our team tracks all incident work in Jira (the customer's project, our team's tickets). Open Jira → new ticket → pick the right project → copy the alert details over → set priority → format the title to match the team's convention → assign yourself.

~2-3 minutes
5

Generate and assign a unique incident ID

Each ticket needs a SOC incident ID following the format CLIENT-IR-MMYYYY-NNN. Look up the last number used, increment it, type it into Jira.

~30 seconds
6

Email the customer

Write the notification. Include incident ID, severity, source IP, affected server, recommended actions. Format professionally. Send to the customer's security contact list.

~3-5 minutes
7

Actually investigate

Pull related logs. Check if the same IP appeared elsewhere. Look at what the suspicious account did after logging in. This is the real analyst work — what justifies my paycheck.

~10-30 minutes
8

Document and close

Update Jira with findings, update customer, close the ticket. Move on to the next alert.

~3-5 minutes

Now Multiply by 100

One alert: about 20 minutes of work. Of that, maybe 10-12 minutes is actual investigation — the part only a human can do. The other 8-10 minutes is moving information between systems. SIEM to wiki to Jira to email and back.

Now imagine you're handling 80-150 alerts per day. Across two customer environments. As one analyst on a small team.

Here's what actually happens:

  • You spend most of your day copy-pasting between the SIEM, the wiki, Jira, and your email client
  • You typo incident IDs because you've typed them a hundred times that day
  • You forget to send the customer email on one ticket out of fifty because you got pulled into a real incident
  • You spend the last hour of your shift triaging alerts that piled up during lunch
  • By Friday afternoon, the backlog is two days deep and growing

You're not doing security work. You're doing data entry with security context.

The Workflow I Wanted to Build

Before I touched any code, I sketched out what the automated version would look like. Stripped down to its essence, here's what needed to happen every time an alert fires:

▸ The automation, end to end
STEP 1
Wazuh sends the alert to the platform
SIEM hits a webhook — payload contains everything we need: source IP, user, server, rule that triggered, timestamp.
▼
STEP 2
Check the whitelist
Is this source IP on our known-good list? If yes, mark it benign and stop here. If no, continue.
▼
STEP 3
Generate a unique incident ID
Format CLIENT-IR-MMYYYY-NNN. The counter is continuous per customer — never resets — only the month in the ID changes.
▼
STEP 4
Create the Jira ticket
Open the customer's Jira project, create a ticket with the right title, priority, and all alert details. Return the Jira link.
▼
STEP 5
Email the customer
Compose the notification (incident ID, severity, source, recommended actions), send it to the customer's security contact list via Microsoft Graph.

That's the goal. Five steps a machine does in about 3 seconds. The analyst's first interaction with an alert should be opening an already-created Jira ticket and going straight into investigation. No typing. No copy-paste. No remembering to email anyone.

The First Version (And Why It Wasn't Enough)

I picked N8N as the workflow engine. N8N is an open-source automation tool — think self-hosted Zapier. Visual editor that an analyst can read, a library of integrations, and the ability to drop into JavaScript when the built-in nodes aren't flexible enough.

I built the five-step flow above as my first version. Wazuh webhook → JS node for whitelist check → Postgres insert for the ID → Jira API call → Microsoft Graph email. Wired up in an afternoon.

It worked. For about three weeks.

Then we added a second customer. Alert volume doubled. And the cracks appeared:

  • N8N ran single-threaded. When alerts came in faster than they could be processed, the workflow queue silently backed up. Real-time alerts arriving with a one-minute delay are no longer real-time.
  • A 90-second compliance report would block alert processing for the entire 90 seconds. Anything running on a schedule starved out the live work.
  • Duplicate incident IDs. When two webhooks fired at the same millisecond, both workflows read the same "last number" from Postgres and both produced the same ID. Different alerts, same identifier, customer confusion.
  • No way to see if it was working. N8N's UI shows executions, not tickets. Analysts kept asking "did my alert make it through?" and the only way to answer was to SSH in and run Postgres queries.
  • No alerting when it broke. A workflow would fail silently at 3 AM. We'd find out the next morning at standup.

The first version proved the concept worked. It also proved that "a webhook into Jira" is not a platform — it's a demo.

What I Actually Needed

The fix wasn't "make the workflow better." The fix was building the layers around the workflow that turn it from a script into a system.

⚙️

A workflow engine that scales

Multiple workers, separate from the main process, that can chew through alerts in parallel without one slow job blocking the rest.

🗄️

A database I can trust

Incident IDs generated by the database itself, not application code. No race conditions, no duplicates, even when 20 workflows hit the same table at once.

📊

A dashboard for the work

One screen showing all open tickets, who's assigned, what's overdue, response-time metrics. Filterable by customer. Refreshes in real time.

❤️

Monitoring that catches breakage

If workers crash, if the queue stops draining, if the SIEM connection drops — I find out in seconds, not at the next standup.

📈

Automatic scaling for bursts

One misconfigured firewall = 800 alerts in 5 minutes. The system needs to add workers automatically, drain the queue, and scale back down — without me touching it.

📝

An audit trail

Every state change, every login, every ticket update — recorded forever. Security work has compliance implications and customer-trust implications.

That's what this series builds. Six layers, one per part, all working together to make the five-step flow above run reliably at any volume.

Beyond the SOC: Where This Pattern Applies

I built this for security operations, but the bones of the architecture — webhook in, parallel workers, atomic database operations, dashboard out, audit log — fit a lot of problems that aren't security at all. If any of these sound like your day, the next five parts are still relevant:

Same pattern, different domain

  • Alert-to-ticket pipelines for any monitoring stack — Datadog, Prometheus, Grafana, Sentry. Any tool that fires webhooks and any ticket system you can hit over HTTP.
  • Compliance evidence collection — scheduled jobs that pull data from N sources, transform it, store it in Postgres, and produce reports with full audit history.
  • Customer-facing automation for an MSP/MSSP — multi-tenant by design, with isolated ID sequences and per-tenant routing.
  • Internal IT workflows — provisioning, deprovisioning, access requests. Same shape as incident response: input event → checks → record → notification → human follow-up.
  • ETL with auditability — when "what changed in this row and who changed it" matters as much as the value itself.

The N8N + Postgres + FastAPI + dashboard pattern is general-purpose. The SOC use case is just the version of it I built.

The point of automation isn't to replace people. It's to let people do the work only they can do.
