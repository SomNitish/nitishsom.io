Building Auto-Scaling N8N with Queue Mode

From single-threaded bottlenecks to production-grade SOAR automation with Redis, PostgreSQL, and dynamic worker scaling

The Problem: When One Thread Isn't Enough

Our SOC automation was breaking. Not crashing — just slow. Alerts from Wazuh were piling up, SentinelOne webhooks were timing out, and the Incident ID generator was creating duplicates again despite our N8N_CONCURRENCY_PRODUCTION_LIMIT=1 fix.

The culprit? Our N8N instance was running in single-threaded mode with SQLite. One Docker container handling everything — the editor UI, webhooks, scheduled triggers, AND workflow execution. Every workflow that ran blocked the next one from starting. When three Wazuh alerts arrived simultaneously, they queued. When a client report took 2 minutes to generate, nothing else could execute.

We needed queue mode. Multiple workers. Parallel execution. But documentation was scattered, production deployments were trial-and-error, and nobody was writing about the real problems you hit when moving from dev to prod.

This is that guide. Everything I learned deploying N8N queue mode with auto-scaling workers for a production SOC environment.

Architecture: Understanding Queue Mode

Queue mode splits N8N into specialized components:

N8N Main Process
Handles the lightweight stuff — editor UI, receiving webhooks, firing scheduled triggers. When a workflow needs to execute, it pushes a job into Redis instead of running it locally.

Redis (BullMQ Broker)
Acts as the message queue. Pending workflow executions sit here until a worker picks them up. Think of it as a waiting room for jobs.

N8N Worker Processes
Do nothing but execute workflows. They pull jobs from Redis, run them, write results to PostgreSQL. You can scale from 2 to 20 workers based on load.

PostgreSQL Database
Replaces SQLite. All processes (main + workers) connect to the same Postgres instance for workflows, credentials, and execution logs. Required because SQLite can't handle concurrent writes.

Key difference from single-threaded mode: In default mode, everything runs in one process. In queue mode, the main process never executes workflows — it only queues them. Workers do all the heavy lifting, and you can scale workers independently.

Prerequisites

Before starting, you'll need:

  • Ubuntu/Debian server with Docker and Docker Compose installed
  • At least 4GB RAM (2GB for N8N, 1GB for Postgres, 512MB for Redis)
  • Basic understanding of Docker networking
  • Access to your existing N8N instance if migrating workflows

Create the project directory:

# ── Security ──
N8N_ENCRYPTION_KEY=CHANGE_ME_ENCRYPTION_KEY

# ── Execution Settings ──
EXECUTIONS_DATA_PRUNE=true
EXECUTIONS_DATA_MAX_AGE=168
N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0

# ── Webhook Configuration ──
WEBHOOK_URL=https://YOUR_SERVER_IP:5678

N8N_HOST=YOUR_SERVER_IP
N8N_PORT=5678

# ── SSL (Self-Signed) ──

N8N_SSL_CERT=/certs/fullchain.pem

# ── Timezone ──
GENERIC_TIMEZONE=Asia/Kolkata
TZ=Asia/Kolkata

⚠️ Critical Variables to Change:

  • DB_POSTGRESDB_PASSWORD — Use a strong password (20+ characters)
  • REDIS_PASSWORD — Different from Postgres password
  • N8N_ENCRYPTION_KEY — 32+ random characters. Back this up! Without it, all stored credentials become unrecoverable
  • WEBHOOK_URL — Your actual server IP or domain
  • N8N_HOST — Same as WEBHOOK_URL without protocol

Key Configuration Explained

EXECUTIONS_MODE=queue — The master switch. Without this, N8N runs in default mode and ignores Redis entirely. Must be set on both main and worker processes.

N8N_CONCURRENCY_PRODUCTION_LIMIT=5 — Each worker picks up 5 jobs concurrently. With 2 workers, you can run 10 workflows simultaneously. Start at 5, adjust based on server resources.

OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true — Even when you click "Execute" in the editor, the job goes to a worker. Keeps the UI responsive.

EXECUTIONS_DATA_MAX_AGE=168 — Auto-deletes execution logs older than 7 days. Without this, Postgres bloats quickly with high workflow volume.

N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0 — Forces the stable execution engine. The newer engine (version 1) has edge cases with sub-workflows.

Architecture Breakdown

PostgreSQL Service:

  • Port 5433 (not 5432) to avoid conflicts with existing Postgres instances
  • Health check ensures N8N doesn't start before database is ready
  • Data persists in postgres_data volume

Redis Service:

  • --requirepass — Authentication required (no open Redis)
  • --appendonly yes — Persistence enabled (queued jobs survive restarts)
  • --maxmemory 256mb — Memory cap prevents runaway growth
  • --maxmemory-policy noeviction — Rejects new jobs when full instead of silently dropping them

N8N Main Service:

  • Exposes port 5678 for web UI
  • depends_on with service_healthy waits for Postgres and Redis health checks
  • Mounts ./certs directory for SSL certificates (read-only)

N8N Worker Service:

  • command: worker — Starts as worker instead of main process
  • replicas: 2 — Starts two worker containers by default
  • Shares same n8nq_data volume as main (for file uploads)
  • No port exposure (workers don't serve HTTP)

setup.sh Deployment Script

Create the setup script to generate SSL certificates and start services:

#!/bin/bash

set -e

echo "=== N8N Queue Mode Setup ==="

# Check Docker
if ! command -v docker &> /dev/null; then
    echo "❌ Docker not found. Install Docker first."
    exit 1
fi

if ! command -v docker compose &> /dev/null; then
    echo "❌ Docker Compose not found. Install Docker Compose first."
    exit 1
fi

# Generate self-signed SSL certificates
echo "🔐 Generating self-signed SSL certificates..."
mkdir -p certs
openssl req -x509 -newkey rsa:4096 -keyout certs/privkey.pem \
    -out certs/fullchain.pem -days 365 -nodes \
    -subj "/CN=n8n-queue"
chmod 644 certs/*.pem

echo "✅ SSL certificates generated in ./certs/"

# Start services
echo "🚀 Starting N8N queue mode..."
docker compose up -d

echo ""
echo "=== Deployment Complete ==="
echo ""
echo "Services:"
echo "  - PostgreSQL: localhost:5433"
echo "  - Redis: localhost:6380"
echo "  - N8N UI: https://localhost:5678"
echo ""
echo "Check status:"
echo "  docker compose ps"
echo ""
echo "View logs:"
echo "  docker compose logs -f n8nq-main"
echo "  docker compose logs -f n8nq-worker"
echo ""

Make it executable:

chmod +x setup.sh

Step 5: Implementing Auto-Scaling

Docker Compose doesn't auto-scale by default. We'll create a custom auto-scaler that monitors Redis queue depth and scales workers dynamically.

Create the Auto-Scale Script

Create autoscale.sh:

#!/bin/bash

# Configuration
PROJECT_DIR="/opt/n8n-queue-mode"
MIN_WORKERS=2
MAX_WORKERS=6
SCALE_UP_THRESHOLD=10
SCALE_DOWN_THRESHOLD=2
CHECK_INTERVAL=15
COOLDOWN=120

cd "$PROJECT_DIR" || exit 1

REDIS_PASSWORD=$(grep ^REDIS_PASSWORD .env | cut -d= -f2)
LAST_SCALE_TIME=0

get_queue_depth() {
    docker exec n8nq-redis redis-cli -a "$REDIS_PASSWORD" --no-auth-warning \
        LLEN bull:jobs:wait 2>/dev/null || echo 0
}

get_active_jobs() {
    docker exec n8nq-redis redis-cli -a "$REDIS_PASSWORD" --no-auth-warning \
        SCARD bull:jobs:active 2>/dev/null || echo 0
}

get_current_workers() {
    docker compose ps n8nq-worker --format json 2>/dev/null | jq -s 'length'
}

scale_workers() {
    local target=$1
    docker compose up -d --scale n8nq-worker="$target" --no-recreate
}

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

while true; do
    PENDING=$(get_queue_depth)
    ACTIVE=$(get_active_jobs)
    CURRENT=$(get_current_workers)
    TOTAL_LOAD=$((PENDING + ACTIVE))
    CURRENT_TIME=$(date +%s)
    TIME_SINCE_LAST_SCALE=$((CURRENT_TIME - LAST_SCALE_TIME))

    log "Queue: $PENDING pending | $ACTIVE active | $CURRENT workers"

    # Cooldown check
    if [ $TIME_SINCE_LAST_SCALE -lt $COOLDOWN ]; then
        log "Cooldown active (${TIME_SINCE_LAST_SCALE}s / ${COOLDOWN}s)"
        sleep $CHECK_INTERVAL
        continue
    fi

    # Scale up logic
    if [ $TOTAL_LOAD -ge $SCALE_UP_THRESHOLD ] && [ $CURRENT -lt $MAX_WORKERS ]; then
        NEW_COUNT=$((CURRENT + 1))
        log "⬆ Scaling UP: $CURRENT → $NEW_COUNT workers (load: $TOTAL_LOAD)"
        scale_workers $NEW_COUNT
        LAST_SCALE_TIME=$CURRENT_TIME

    # Scale down logic
    elif [ $TOTAL_LOAD -le $SCALE_DOWN_THRESHOLD ] && [ $CURRENT -gt $MIN_WORKERS ]; then
        NEW_COUNT=$((CURRENT - 1))
        log "⬇ Scaling DOWN: $CURRENT → $NEW_COUNT workers (load: $TOTAL_LOAD)"
        scale_workers $NEW_COUNT
        LAST_SCALE_TIME=$CURRENT_TIME
    fi

    sleep $CHECK_INTERVAL
done

Make it executable:

chmod +x autoscale.sh

How the Auto-Scaler Works

  • Polls Redis every 15 seconds
  • Counts pending jobs (LLEN bull:jobs:wait) + active jobs (SCARD bull:jobs:active)
  • If total ≥ 10 jobs → adds 1 worker (up to max 6)
  • If total ≤ 2 jobs → removes 1 worker (down to min 2)
  • 2-minute cooldown between scaling actions prevents thrashing

Run as a Systemd Service

Create the service file:

sudo tee /etc/systemd/system/n8n-autoscale.service > /dev/null << 'EOF'
[Unit]
Description=N8N Worker Auto-Scaler
After=docker.service
Requires=docker.service

[Service]
Type=simple
ExecStart=/opt/n8n-queue-mode/autoscale.sh
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
EOF

Enable and start:

sudo systemctl daemon-reload
sudo systemctl enable n8n-autoscale
sudo systemctl start n8n-autoscale

Monitor the auto-scaler:

journalctl -u n8n-autoscale -f

Common Issues and Debugging

Issue 1: Workers Not Picking Up Jobs

Symptoms: Workflows queue in Redis but never execute.

Debug steps:

# Check if workers are running
docker compose ps n8nq-worker

# Check worker logs for errors
docker compose logs n8nq-worker

# Verify Redis connection
docker exec n8nq-redis redis-cli -a "YOUR_REDIS_PASSWORD" PING

# Check if jobs are in Redis
docker exec n8nq-redis redis-cli -a "YOUR_REDIS_PASSWORD" LLEN bull:jobs:wait

Common causes:

  • EXECUTIONS_MODE=queue not set in worker environment
  • Incorrect Redis password
  • Workers started before Redis health check passed

Issue 2: Database Connection Errors

Symptoms: Error: Connection to PostgreSQL failed

Debug steps:

# Check if Postgres is healthy
docker compose ps postgres

# Test connection from main container
docker exec n8nq-main pg_isready -h postgres -p 5432 -U n8n_queue

# Check Postgres logs
docker compose logs postgres | tail -50

# Verify credentials
grep DB_POSTGRESDB .env

Fix: Ensure DB_POSTGRESDB_HOST=postgres (not localhost). Inside Docker network, containers reference each other by service name.

Issue 3: Encryption Key Mismatch

Symptoms: Workflows fail silently when calling APIs. No obvious errors in logs.

Debug steps:

# Check if encryption key is identical across all processes
docker exec n8nq-main printenv N8N_ENCRYPTION_KEY
docker exec n8nq-worker-1 printenv N8N_ENCRYPTION_KEY
docker exec n8nq-worker-2 printenv N8N_ENCRYPTION_KEY

# All three must output the same value

Fix: The encryption key in .env is shared by all containers automatically via Docker Compose's environment: section. If keys mismatch, you manually set different keys somewhere.

Issue 4: SSL Certificate Errors

Symptoms: Browser shows "Your connection is not private" or N8N fails to start with certificate errors.

Debug steps:

# Check if certificates exist
ls -lh certs/

# Verify certificate permissions
docker exec n8nq-main ls -l /certs/

Fix: Certificates must be readable by the node user inside the container:

chmod 644 certs/privkey.pem certs/fullchain.pem

Issue 5: Auto-Scaler Not Scaling

Symptoms: Queue depth grows but worker count stays at 2.

Debug steps:

# Check if auto-scaler is running
systemctl status n8n-autoscale

# Check auto-scaler logs
journalctl -u n8n-autoscale -n 50

# Manually test scaling
cd /opt/n8n-queue-mode
docker compose up -d --scale n8nq-worker=4

Common causes:

  • Auto-scaler service not started
  • Incorrect PROJECT_DIR in autoscale.sh
  • Redis password in script doesn't match .env

Production Checklist

Before going live with this setup:

✅ Security

  • Replace self-signed certificate with proper SSL cert (Let's Encrypt or your domain cert)
  • Use strong passwords (20+ characters) for Postgres and Redis
  • Back up N8N_ENCRYPTION_KEY to a secure location (password manager, encrypted vault)
  • Restrict Postgres and Redis ports to localhost if not needed externally

✅ Performance

  • Adjust N8N_CONCURRENCY_PRODUCTION_LIMIT based on server CPU/RAM
  • Monitor worker CPU/memory usage under load
  • Set MAX_WORKERS in auto-scaler based on server capacity
  • Enable execution log pruning to prevent Postgres bloat

✅ Monitoring

  • Set up Prometheus metrics (N8N_METRICS=true) and Grafana dashboards
  • Monitor Redis memory usage
  • Track Postgres disk usage
  • Alert on auto-scaler failures

✅ Backup

  • Regular Postgres backups (pg_dump daily)
  • Back up n8nq_data Docker volume (contains binary files)
  • Export workflows periodically using N8N CLI

Key Takeaways

  1. Queue mode is essential for production SOC automation. Single-threaded N8N creates dangerous bottlenecks when handling alerts from multiple sources simultaneously.
  2. The encryption key is everything. Back it up immediately after generation. Without it, all stored credentials become unrecoverable.
  3. Redis is the orchestrator. Workers are stateless — all coordination happens through Redis. If Redis goes down, job execution stops until it's restored.
  4. Auto-scaling prevents resource waste. Idle workers consume RAM even when there's no work. Dynamic scaling keeps costs down while ensuring capacity during alert spikes.
  5. Monitor queue depth, not just CPU. Traditional metrics miss the queue buildup. A growing Redis queue is your early warning system for insufficient worker capacity.
  6. Per-workflow concurrency > global concurrency. Queue mode lets you set concurrency=1 on specific workflows (like Incident ID generators) while running everything else in parallel.
  7. Test before production. Create simple test workflows, generate artificial load, and verify scaling behavior before migrating critical SOC workflows.
