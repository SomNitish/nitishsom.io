📘 PART 1: Problem & Architecture

From Single-Threaded Bottlenecks to Multi-Worker Production SOAR

How we deployed N8N queue mode with Redis, PostgreSQL, and multiple workers to handle 100+ alerts/hour without data loss or ID collisions

The 3 AM Problem

It started at 3 AM on May 28th, 2026. Wazuh was throwing unauthorized SSH login alerts, and our old N8N setup was drowning. Here's what we were seeing:

❌ Problem 1: Single-threaded bottleneck
One N8N container. One process. When Wazuh fired 50 alerts simultaneously, they queued for 2+ minutes. By the time we got to alert #40, critical security events were already 2 minutes old.

❌ Problem 2: Missing data
Incidents were being created in Postgres, but data was inconsistent. Some alerts weren't making it to Jira. Some emails were missing complete details. We'd set up "pinned data" for testing and forgot to clear it — it was overriding real webhook data.

❌ Problem 3: No scalability
We'd hardcoded N8N_CONCURRENCY_PRODUCTION_LIMIT=1 to prevent duplicate Incident IDs. This made everything sequential — no parallelism, no speed, no efficiency.

❌ Problem 4: Integration chaos
We had two event sources (Wazuh JournalD events and SentinelOne alerts) running through the same workflow. One webhook endpoint. No routing logic. Events were getting mixed up or lost entirely.

On that May 28th morning, after clearing the pinned data and rebuilding the entire workflow architecture with proper queue mode, we deployed a solution. Within 24 hours, we'd processed 60 incidents without a single data loss or duplicate ID issue.

This is how we did it.

The Real Cost of Single-Threaded SOC Automation

Before queue mode, our architecture was simple: one Docker container, SQLite database, everything in one process. The problem wasn't the setup — it worked fine for 10-20 workflows per hour. But the moment we started handling real SOC load (100+ alerts/hour across multiple clients), the limitations became obvious:

The server had 8 CPU cores and 16GB RAM, but N8N was using maybe 200MB and 5% of one core. Everything else was blocked waiting for a single process to finish. Watching top showed the absurdity: 15 cores idle, one core saturated, 15GB RAM unused.

Architecture: From Single Process to Distributed Workers

Queue mode fundamentally changes how N8N works. Instead of one process doing everything, you split responsibilities:

N8N Main Process — Stateless. Handles the UI, receives webhooks, fires scheduled triggers. When a workflow needs to execute, it doesn't run it. It queues it.

Redis (Message Broker) — Holds the queue. Wazuh alert comes in → main receives it → pushes a job to Redis → immediately returns success. The alert is now safe in Redis, even if N8N crashes.

N8N Worker Processes — 2, 5, 10, or 20 independent processes. Each one pulls a job from Redis, executes it in isolation, writes results to Postgres. All workers run in parallel.

PostgreSQL Database — Shared state. All workers write execution results here. Main process reads from here. Supports concurrent writes from multiple workers (unlike SQLite).

Why This Matters for SOC Automation

Imagine Wazuh fires 5 unauthorized SSH alerts at the exact same second:

Old setup (single-threaded):
Alert 1 arrives → main executes it → whitelist check (2s) + Postgres insert (1s) + email (3s) + Jira ticket (2s) = 8s total. Meanwhile, alerts 2-5 wait. Time to complete all 5: 40 seconds.

New setup (queue mode, 4 workers):
Alerts 1-5 arrive → main queues all of them instantly → worker-1 takes alert-1, worker-2 takes alert-2, worker-3 takes alert-3, worker-4 takes alert-4 → all 4 execute in parallel. Alert-5 waits 8 seconds for a worker to free up. Time to complete all 5: 12 seconds instead of 40.

That's a 71% reduction in processing time and all it required was architecture change.

The Real Problem We Faced: Missing Data

Now here's the tricky part. We built this entire queue-mode setup, deployed it, started processing alerts... and nothing showed up in the database.

The logs showed green. N8N showed "Execution completed successfully." Redis showed jobs being consumed. But PostgreSQL had zero new incidents.

The debugging nightmare:

We spent 3 hours checking:

  • Is the Postgres connection string correct? ✓ (can connect manually)
  • Is the Postgres schema right? ✓ (table exists)
  • Are workers actually executing? ✓ (logs show execution)
  • Is N8N writing to Postgres? ✗ (no records)

Then we checked the N8N workflow nodes directly. On the "When Executed by Another Workflow" trigger node, we saw it: PINNED DATA.

During development, we'd clicked "pin" on a test payload (fake Wazuh data with user "pinned-data"). When we later sent real webhook data, N8N ignored it and used the pinned data instead. Same user "pinned-data" every time. The workflow ran fine, but on fake data.

The fix was one click: Right-click trigger node → Clear pinned data.

But the lesson was profound: in complex SOAR workflows, data can flow through 20 nodes and you'll never notice it's fake until you compare database output to expected results. This is why testing and validation matter.

TZ=Asia/Kolkata

# Use stable execution engine (newer one has edge cases with subworkflows)
N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0

Key Configuration Explained

EXECUTIONS_MODE=queue — This is the master switch. Without it, N8N ignores Redis and runs single-threaded. Set on both main and workers.

N8N_ENCRYPTION_KEY — All API keys, OAuth tokens, passwords stored in Postgres are encrypted with this key. Every process (main + all workers) must have the exact same key. If they differ, workers can't decrypt credentials and workflows fail silently.

N8N_CONCURRENCY_PRODUCTION_LIMIT=5 — Each worker picks up 5 jobs concurrently. With 2 workers → 10 parallel workflows. With 4 workers → 20 parallel workflows. Adjust based on server CPU and RAM.

OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true — When you click "Execute" in the editor, instead of running on main (blocking the UI), it gets queued to a worker. UI stays responsive.

EXECUTIONS_DATA_PRUNE=true + EXECUTIONS_DATA_MAX_AGE=168 — Auto-delete execution logs older than 7 days. Without this, Postgres grows 5-10GB per week if you're processing thousands of workflows.

The Multi-Worker Concept

This is the heart of what makes queue mode powerful. Instead of one N8N process doing everything, you have multiple independent workers:

Component Single-Threaded Queue Mode (2 workers) Queue Mode (4 workers)
Processes 1 3 (1 main + 2 workers) 5 (1 main + 4 workers)
Parallel Executions 1 workflow at a time Up to 10 workflows (5 per worker) Up to 20 workflows (5 per worker)
Queue Persistence None - in-process memory Redis - survives restarts Redis - survives restarts
Database SQLite - single writer PostgreSQL - multiple writers PostgreSQL - multiple writers
Scaling Vertical only (bigger server) Horizontal (add more workers) Horizontal (add more workers)

How Workers Coordinate (Without Talking)

One of the beautiful parts of this architecture: workers don't talk to each other. No synchronization, no locking, no coordination. All coordination happens through Redis and PostgreSQL:

  1. Main receives alert from Wazuh → Webhook hits main process
  2. Main queues the job → Pushes job to Redis with all alert data
  3. Any available worker picks it up → LRANGE bull:jobs:wait in Redis finds pending jobs
  4. Worker executes independently → Runs whitelist check, creates incident in Postgres, sends email, creates Jira ticket
  5. Worker marks job complete → Removes from Redis, updates execution in Postgres
  6. Next pending job gets picked up → If 3 alerts are waiting and you have 2 workers, worker-1 takes alert-A and worker-2 takes alert-B simultaneously

There's no "worker-1, you handle SSH events, worker-2 you handle SentinelOne events." There's no state sharing. Each worker is completely independent.

Use Cases This Architecture Enables

Our specific use case is Wazuh + SentinelOne alerts for a multi-tenant SOC, but this architecture works for:

  • Real-time threat detection — Wazuh alerts → whitelisting → Jira tickets → Slack notifications, all in parallel
  • Multi-source log normalization — Collect from 10 different security tools, normalize, enrich, alert
  • Compliance reporting — Daily/weekly/monthly reports processing in parallel without blocking the UI
  • Incident response automation — Block IPs, disable accounts, isolate endpoints — all simultaneously across multiple incidents
  • Integration hub — Webhook from 50 sources all hitting N8N, getting routed to appropriate destinations
  • Data pipeline at scale — Ingest 10,000 events per minute, transform, load to data warehouse

Anywhere you need to process multiple independent jobs without bottlenecks, this architecture is worth it.
