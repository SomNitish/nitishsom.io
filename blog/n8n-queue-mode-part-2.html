📗 PART 2: Workflows & Multi-Worker Execution

The Actual Workflows: Routing, Processing, and Multi-Worker Parallelism

Step-by-step breakdown of the parent router and JournalD processor workflows. Complete JSON exports. How multiple workers execute these in parallel without data loss.

How the Architecture Flows Data

Before diving into the workflows, let me show you the data journey from a Wazuh alert hitting our webhook all the way to a Jira ticket being created:

1. Webhook Wazuh sends JSON alert to N8N webhook endpoint
2. Main Process N8N main receives the alert, immediately queues it to Redis
3. Redis Queue Alert sits in bull:jobs:wait queue as a job object
4. Worker Pickup Any available worker (worker-1, worker-2, etc.) polls Redis and grabs the job
5. Router Switch Parent workflow runs, examines rule ID, routes to JournalD or SentinelOne workflow
6. Sub-Workflow Execution Whitelist check, Postgres insert, email, Jira ticket — all in one worker
7. Results Persist Worker writes to Postgres, marks job complete in Redis, returns to waiting

The beautiful part: steps 4-7 happen in parallel across multiple workers. While worker-1 is creating a Jira ticket for alert-A, worker-2 is simultaneously whitelisting alert-B, and worker-3 is sending emails for alert-C. All three write to the same Postgres instance without stepping on each other.

Workflow 1: The Parent Router (Dispatcher)

This workflow is the traffic cop. It has one responsibility: receive all alerts and route them to the appropriate sub-workflow based on alert type.

Design Philosophy

  • Single Entry Point — One webhook URL for all event sources
  • Stateless Routing — No stored state, just switch logic based on incoming data
  • Fast Dispatch — Minimal processing, immediately hand off to sub-workflows
  • Multi-Output — Can route to Jira, Slack, email, or custom endpoints

The Parent Router Workflow Structure

Node Type Purpose
Webhook Webhook Trigger Listen on POST /webhook/[uuid]. Receive Wazuh/SentinelOne JSON.
Switch Switch Node Examine rule ID. Route to "JournalD Event" or "Sentinel Flow" branch.
Call 'JournalD Events' Execute Workflow If rule ID = 5715, execute JournalD Events sub-workflow.
Call 'SentinelFlow' Execute Workflow If program = sentinelone, execute SentinelOne sub-workflow.

How the Switch Node Works

The Switch node is N8N's conditional router. It examines the incoming data and routes to different branches:

Route 1: JournalD Events
Condition: body.all_fields.rule.id == "5715"
This is the Wazuh rule ID for SSH authentication success. Any alert with this rule ID goes to the JournalD sub-workflow.

Route 2: SentinelOne Flow
Condition: body.all_fields.predecoder.program_name == "sentinelone"
Any alert from SentinelOne goes to the SentinelOne sub-workflow.

Default Route (no output)
If neither condition matches, the job is consumed but not forwarded. No error, no loop — just discarded. This is intentional for unknown alert types.

Workflow 2: The JournalD Events Processor

This is where the real work happens. This workflow processes SSH authentication alerts, checks whitelists, creates incidents, sends emails, and opens Jira tickets. It's called by the parent router multiple times in parallel (one call per worker).

Critical: The Postgres INSERT (Atomic Incident ID Generation)

This is the fix that solved our duplicate ID problem. The SQL function generates unique incident IDs atomically:

INSERT INTO incidents (logged_at, alert_name, event_timestamp, severity, category)
VALUES (
  '{{ $json.logged_at }}',
  '{{ $json.alert_name.replace(/'/g, "''") }}',
  '{{ $json.event_timestamp }}',
  '{{ $json.severity }}',
  '{{ $json.category }}'
)
RETURNING seq, incident_id, logged_at, alert_name, event_timestamp, severity, category;

The magic is RETURNING seq, incident_id. When the INSERT executes, Postgres atomically:

  1. Locks the incidents table
  2. Increments the sequence
  3. Generates incident_id as TSP-IR-202605-{seq}
  4. Inserts the row
  5. Unlocks and returns the generated ID

This happens even with multiple workers inserting simultaneously. Postgres queues the writes and processes them serially at the database level. No two incidents get the same ID, no matter how many workers hit Postgres at the same millisecond.

The Data Flow Through Multiple Workers

Each worker is a separate process, but they all coordinate through Redis and Postgres:

Worker-1 Processing Alert-1

1. Grabs Alert-1 from Redis
2. Runs Whitelist Check JS
3. Executes: INSERT INTO incidents (logged_at, alert_name, ...) RETURNING incident_id
    → Postgres: LOCK incidents table
    → seq = 1
    → incident_id = TSP-IR-202605-001
    → UNLOCK and return
4. Sends email via Microsoft Outlook
5. Creates Jira issue
6. Writes Jira ID back to Postgres: UPDATE incidents SET jira_id=... WHERE incident_id='TSP-IR-202605-001'
7. Marks job complete in Redis

Worker-2 Processing Alert-2 (Simultaneously)

1. Grabs Alert-2 from Redis
2. Runs Whitelist Check JS
3. Executes: INSERT INTO incidents (...) RETURNING incident_id
    → Postgres: LOCK incidents table (waits if Worker-1 has it)
    → seq = 2 (incremented from 1)
    → incident_id = TSP-IR-202605-002
    → UNLOCK and return
4. Sends email (different receiver, different data)
5. Creates Jira issue
6. Updates Postgres with Jira ID
7. Marks job complete

Even though both workers are trying to INSERT into the same Postgres table at the same time, Postgres's transaction isolation level ensures that:

  • Worker-1 gets seq=1, incident_id=TSP-IR-202605-001
  • Worker-2 gets seq=2, incident_id=TSP-IR-202605-002
  • Never a collision, no duplicate IDs

Where This Architecture Powers Different Use Cases

Real-Time Security Operations (Our Use Case)

Multi-source alerts (Wazuh, SentinelOne, Fortinet, etc.) → Router → Specialized processors → Incident database + Jira + Slack

Compliance Reporting at Scale

Generate 50 different compliance reports (PCI, ISO 27001, NIST RMF) in parallel. Each worker generates one report independently. One server with 4 workers finishes in 1/4 the time of single-threaded.

Multi-Tenant SaaS Log Normalization

Receive logs from 100 different customer environments. Router dispatches each to a customer-specific processor. Processors normalize, enrich, and load to that customer's data warehouse. 4 workers handle 400 logs/second instead of 100.

Integration Hub for Multiple Destinations

One webhook receives data. Router examines type and routes to: Salesforce, HubSpot, Slack, Email, Webhook, Custom API. Multiple workers hit all 6 destinations in parallel for different records.

Real-Time Data Pipeline

Kafka producer sends 10,000 events/minute to N8N via webhooks. Router categorizes (metrics, logs, traces). Workers transform and load to InfluxDB, Elasticsearch, and Datadog simultaneously.

🎯 Common Pattern Across All Use Cases:
Webhook → Router (switch on type) → Specialized workers (parallel execution) → Shared database/storage

This scales from 10 events/hour to 10,000 events/minute without code changes. Just add more workers.
