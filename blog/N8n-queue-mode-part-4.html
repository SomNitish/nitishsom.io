Backup Script

Automate daily Postgres backups with this script:

#!/bin/bash

# N8N Queue Mode Backup Script
# Run daily via cron

BACKUP_DIR="/backups/n8n-queue"
DB_CONTAINER="n8nq-postgres"
DB_NAME="n8n_queue"
DB_USER="n8n_queue"
RETENTION_DAYS=30

mkdir -p "$BACKUP_DIR"

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/n8n_backup_${DATE}.sql.gz"
ENCRYPTION_KEY_FILE="$BACKUP_DIR/.encryption_key"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

# Backup Postgres
log "Starting Postgres backup..."
docker exec $DB_CONTAINER pg_dump \
    -U $DB_USER \
    -d $DB_NAME \
    --no-password | gzip > "$BACKUP_FILE"

if [ $? -eq 0 ]; then
    SIZE=$(du -h "$BACKUP_FILE" | cut -f1)
    log "✅ Backup complete: $SIZE ($BACKUP_FILE)"
else
    log "❌ Backup failed"
    exit 1
fi

# Backup encryption key (if .env changed)
ENV_FILE="/opt/n8n-queue-mode/.env"
if [ -f "$ENV_FILE" ]; then
    KEY=$(grep ^N8N_ENCRYPTION_KEY "$ENV_FILE" | cut -d= -f2)
    echo "$KEY" > "$ENCRYPTION_KEY_FILE"
    chmod 600 "$ENCRYPTION_KEY_FILE"
fi

# Clean up old backups
log "Cleaning up backups older than ${RETENTION_DAYS} days..."
find "$BACKUP_DIR" -name "n8n_backup_*.sql.gz" -mtime +${RETENTION_DAYS} -delete

log "Backup job complete"
👉 Save as `/usr/local/bin/n8n-backup.sh`. Add to crontab: `0 2 * * * /usr/local/bin/n8n-backup.sh`

Disaster Recovery Procedure: Postgres Corruption

Scenario: Postgres container crashes. Database corrupted. Need to restore from backup.

Recovery Steps:

# 1. Stop N8N services (don't let workers write to corrupted DB)
cd /opt/n8n-queue-mode
docker compose down

# 2. Verify backup exists and is readable
ls -lh /backups/n8n-queue/n8n_backup_*.sql.gz

# 3. Start only Postgres (not main/workers yet)
docker compose up -d n8nq-postgres

# 4. Wait for Postgres to be ready
sleep 10

# 5. Drop the corrupted database
docker exec n8nq-postgres dropdb -U n8n_queue n8n_queue

# 6. Create a fresh database
docker exec n8nq-postgres createdb -U n8n_queue -O n8n_queue n8n_queue

# 7. Restore from backup (choose the LATEST backup)
zcat /backups/n8n-queue/n8n_backup_20260529_020000.sql.gz | \
    docker exec -i n8nq-postgres psql -U n8n_queue -d n8n_queue

# 8. Verify restoration
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue \
    -c "SELECT COUNT(*) as incidents FROM incidents;"

# Expected: Should show the incident count at backup time

# 9. Start the full stack
docker compose up -d

# 10. Monitor logs
docker compose logs -f n8nq-main | head -50

Troubleshooting: Common Issues & Solutions

Issue 1: Workers Not Executing Workflows

Symptom: Redis shows jobs in queue, but workers aren't processing them. Workflows sit pending for 10+ minutes.

Debug:

# Check if workers are running
docker compose ps n8nq-worker

# Check worker logs for errors
docker compose logs n8nq-worker | grep -i error | head -20

# Verify Redis connection from worker
docker exec n8nq-worker redis-cli -h n8nq-redis -a $REDIS_PASSWORD ping

# Check if EXECUTIONS_MODE=queue is set in workers
docker exec n8nq-worker printenv | grep EXECUTIONS_MODE

Likely Causes & Fixes:

  • EXECUTIONS_MODE not set on workers — Update .env and restart: docker compose restart n8nq-worker
  • Redis password mismatch — Verify QUEUE_BULL_REDIS_PASSWORD in .env matches REDIS_PASSWORD
  • Workers crashed on startup — Check logs: docker compose logs n8nq-worker --tail=100
  • Encryption key mismatch — Main and workers must have identical N8N_ENCRYPTION_KEY

Issue 2: Duplicate Incident IDs

Symptom: Two incidents with the same ID (e.g., both TSP-IR-202605-001). Should never happen with Postgres sequence, but indicates corruption.

Debug:

docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue << 'SQL'
SELECT incident_id, COUNT(*) as cnt 
FROM incidents 
GROUP BY incident_id 
HAVING COUNT(*) > 1
ORDER BY cnt DESC;
SQL

Fix:

# Check the sequence value
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue \
    -c "SELECT nextval('incidents_seq');"

# If sequence is behind (e.g., 500 but table has 600 rows):
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue << 'SQL'
SELECT setval('incidents_seq', (SELECT MAX(seq) FROM incidents));
SQL

# Verify
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue \
    -c "SELECT MAX(seq) FROM incidents;"

Issue 3: Postgres Connection Timeout

Symptom: Error: "connection timeout" in N8N logs. Workflows fail to insert into Postgres.

Debug:

# Check Postgres is running
docker compose ps n8nq-postgres

# Check active connections
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue \
    -c "SELECT count(*) FROM pg_stat_activity;"

# Check max connections setting
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue \
    -c "SHOW max_connections;"

Likely Causes:

  • Too many workers (6+ workers × 5 concurrency = 30 connections). Default Postgres max is 100, but you want <80 active.
  • Long-running queries holding connections. Check for slow INSERT/UPDATE statements.
  • Connection pool exhaustion. N8N isn't releasing connections properly.

Fix:

# Restart Postgres to reset connections
docker compose restart n8nq-postgres

# Or: reduce worker concurrency in .env
N8N_CONCURRENCY_PRODUCTION_LIMIT=3  # down from 5

# Or: increase Postgres max connections
docker exec n8nq-postgres psql -U postgres -c \
    "ALTER SYSTEM SET max_connections = 200;"
docker compose restart n8nq-postgres

Issue 4: Redis Memory Full

Symptom: Error: "OOM command not allowed when used memory > 'maxmemory'" in logs. Queue not accepting new jobs.

Debug:

docker exec n8nq-redis redis-cli -a $REDIS_PASSWORD info memory | grep used_memory

# Expected: <200MB (we set maxmemory=256MB)

Cause: Queue is full. Jobs aren't being consumed (workers are slow or down). Pending jobs accumulate in Redis.

Fix:

# Quick: Clear the queue (loses pending jobs — only if needed)
docker exec n8nq-redis redis-cli -a $REDIS_PASSWORD FLUSHALL

# Better: Scale up workers
docker compose up -d --scale n8nq-worker=6 --no-recreate

# Best: Investigate slow workflows and optimize them
# Check N8N execution times for > 5 second workflows

Performance Tuning for High-Volume Alerts

Workflow-Level Optimization

  • Remove unnecessary nodes — Each node = latency. If a node isn't needed, delete it.
  • Parallelize sub-workflows — Instead of email → Jira → Slack sequentially, execute them in parallel branches.
  • Cache lookups — If whitelist check hits the same IPs repeatedly, cache the result.
  • Batch database writes — Instead of 1 INSERT per workflow, accumulate 100 and bulk INSERT.

Infrastructure-Level Optimization

  • Postgres indexing — Add indexes on frequently queried columns (incident_id, alert_name, severity).
  • Postgres vacuum — Run VACUUM and ANALYZE nightly to defragment and update stats.
  • Redis persistence tuning — Adjust --appendfsync from "always" to "everysec" for better throughput.
  • Worker concurrency tuning — N8N_CONCURRENCY_PRODUCTION_LIMIT: 3 for CPU-heavy, 10 for I/O-heavy.

Monitoring Performance Trends

# Weekly: Check average execution time trend
docker exec n8nq-postgres psql -U n8n_queue -d n8n_queue << 'SQL'
SELECT DATE(startedAt) as date,
       ROUND(AVG(EXTRACT(EPOCH FROM (stoppedAt - startedAt))), 2) as avg_seconds,
       COUNT(*) as executions
FROM execution_entity
WHERE startedAt > NOW() - INTERVAL '30 days'
GROUP BY DATE(startedAt)
ORDER BY date DESC;
SQL

Scaling Beyond 500 Alerts/Hour

If you're processing >500 alerts/hour, the single-server setup becomes tight. Consider:

  • Separate Postgres onto its own server — Dedicate hardware to database. Current setup is colocated.
  • Separate Redis onto its own server — Allows independent Redis tuning and monitoring.
  • Run N8N main on one server, workers on 2-3 others — Distribute load. Main stays on one, workers scale horizontally.
  • Use managed cloud services — RDS for Postgres, ElastiCache for Redis. Reduces ops burden.
  • Multi-region setup — Primary region + standby. Failover on disaster.

What This Architecture Powers

This isn't just for Wazuh + SentinelOne. The same architecture works for:

  • ✅ Multi-source SIEM (Splunk, Elastic, Suricata)
  • ✅ Cloud provider alerts (AWS GuardDuty, Azure Sentinel, GCP Security Command Center)
  • ✅ Compliance automation (PCI, ISO 27001, SOC 2 reports)
  • ✅ Real-time data pipelines (Kafka → normalization → warehouse)
  • ✅ Multi-tenant integrations (Salesforce, HubSpot, Slack, custom APIs)
  • ✅ Incident response automation (IP blocking, user account disable, endpoint isolation)

The limiting factor is your server resources and workflow complexity. If you need to process 10,000+ events/hour, consider multi-server deployment or cloud-managed Postgres/Redis.

Final Recommendations

  • Start with 2 workers. Monitor for a week. Scale up only if needed.
  • Always test before production. Run the load test suite on a staging server.
  • Back up the encryption key. Losing it = all credentials unrecoverable.
  • Monitor daily. A single command checking queue depth takes 30 seconds.
  • Plan for failure. Postgres will crash. Redis will fill. A worker will die. Have recovery procedures.
  • Iterate on workflows. Your first workflows won't be perfect. Measure, optimize, repeat.
