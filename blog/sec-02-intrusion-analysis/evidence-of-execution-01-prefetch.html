Evidence of Execution — Part 1 of 3

What Prefetch Files Actually Remember

A workstation looks compromised. The tools are gone. The attacker was careful — except Windows was quietly taking notes the whole time, in a folder almost nobody thinks to check first.

📅 2026-07-13 ⏱ 14 min read DFIR Windows Forensics Wazuh Detection Engineering
I keep a running note on my desk that just says "the disk doesn't lie, it just forgets selectively." Prefetch is the first artifact that taught me that. It's also the first one I got embarrassingly wrong before I understood what it was actually for.

The Use Case

Say an endpoint alert fires — SentinelOne flags a process that's already gone by the time you pivot into the host. No live process, no file on disk, nothing in the recycle bin. The attacker cleaned up. This is the point where a lot of junior responders (me, a few months ago) shrug and write "insufficient evidence" in the ticket.

That's the wrong move. Windows has been silently building a performance-optimization index of everything you've run, since long before you knew you'd need it for an investigation — and it does this regardless of whether the attacker was careful about their actual payload.

What Prefetch Actually Is

Prefetch is not a security feature. It's a caching mechanism Windows built to make your applications launch faster. The OS watches which files and DLLs an application touches in its first ten seconds of execution, and the next time you launch that app, it pre-loads those same resources from disk into memory ahead of time — hence "pre-fetch."

Every time it does this, it writes a small metadata file to C:\Windows\Prefetch, named after the executable plus an 8-character hash of the full path it was run from, e.g. CMD.EXE-087B4001.pf. From Windows 8 onward, the system keeps up to 1024 of these; earlier versions capped at 128 — which matters, because on a busy or heavily-scripted host, old evidence gets rotated out faster than you'd expect.

Why this matters for IR

A prefetch file is proof the target executable ran — not just that it existed on disk. That distinction is the entire value of this artifact. A malicious binary sitting unexecuted in a temp folder is a finding. The same binary with a matching .pf file is a confirmed execution.

What's actually inside one

Each prefetch file tracks, per executable:

  • The run count — total number of times it's been executed
  • The original full path it was launched from
  • Up to 8 of the most recent execution timestamps (Windows 8–11)
  • The file's own creation time on disk, which — combined with the 8 embedded timestamps — gives you up to 9 data points for one binary's execution history

Checking Whether Prefetch Is Even On

Before you trust an absence of prefetch data as "this never ran," check whether prefetch collection itself was disabled — this happens more often than you'd think on server builds, and some hardening guides disable it as a side effect of trying to reduce disk writes.

Registry — verify prefetch is enabled
# Application launch + boot prefetching control
Path: HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters
Value: EnablePrefetcher (DWORD)

0 = Disabled
1 = Application launch prefetching enabled
2 = Boot prefetching enabled
3 = Both enabled (default on most workstation builds)

# SysMain (formerly Superfetch) service — governs whether the
# prefetch subsystem runs at all
Path: HKLM\SYSTEM\CurrentControlSet\Services\SysMain
Value: Start (DWORD)

0-2 = Automatic start
3   = Manual start
4   = Disabled
Don't skip this check

If a host you're investigating has zero prefetch files and you haven't confirmed these keys, you don't have "no evidence of execution" — you have an unanswered question. Audit the keys first, conclude second.

The Detail That Actually Changed How I Read a Case

Here's the thing that took me longest to internalize: the same executable name can legitimately have multiple prefetch files — but the reason matters enormously.

For most applications, a hash is computed from the executable's full path. So if you see two prefetch files for CMD.EXE with different hash suffixes, that usually means cmd.exe was run from two different locations — and if one of those locations isn't C:\Windows\System32, you may be looking at a renamed or planted binary impersonating a trusted process. That's a real finding, not a coincidence.

The exception: Windows "hosting" processes — svchost.exe, dllhost.exe, rundll32.exe, backgroundtaskhost.exe — get their hash calculated from the full command line, not just the path. So it's completely normal to see a dozen prefetch entries for svchost.exe with different arguments. Flagging every one of those as suspicious is how you burn a shift chasing noise instead of the one entry that matters.

Where I got this wrong initially

My first pass at "prefetch anomaly hunting" flagged multi-instance svchost prefetch files as suspicious duplicates. It wasn't wrong data — it was the wrong baseline. I hadn't yet separated "hosting processes hash on command line" from "everything else hashes on path." Know which bucket you're looking at before you draw a conclusion.

Getting the Data Out: PECmd

Prefetch files are binary and not human-readable on their own. Eric Zimmerman's PECmd.exe is the tool I actually reach for — it parses the .pf structure into readable CSV/JSON, including run counts, all embedded timestamps, and the list of files/directories referenced during that launch window (which is its own goldmine — it tells you what the process actually touched, not just that it ran).

PowerShell — parse a single host's entire prefetch folder
PECmd.exe -d "C:\Windows\Prefetch" --csv "C:\triage\output" --csvf prefetch_results.csv

# Key columns worth pivoting on once it's in the CSV:
# SourceFilename, ExecutableName, RunCount, LastRun,
# PreviousRun0-7, Volume0Name, FilesLoaded

Turning This Into a Wazuh Detection

Reading prefetch manually during triage is one thing. The more interesting engineering problem — the one I'm still actively working through — is: can you get signal on new prefetch activity in near-real-time instead of waiting for a manual pull?

The answer is yes, but it needs two separate pieces working together, because prefetch itself isn't a log source — it's a folder of files.

01

Sysmon Event ID 11 (File Create)

Configure Sysmon to watch C:\Windows\Prefetch\*.pf specifically. Every new prefetch file creation becomes a loggable event with a timestamp, the filename, and the process that triggered it — SysMain, in this case.

02

Forward to the SOC platform via eventchannel

The Wazuh agent picks up Sysmon's operational log through the standard eventchannel localfile block and ships it to the manager, same path used for every other Sysmon event type already flowing into the platform.

03

Custom rule layered on top of the base Sysmon ruleset

A child rule keyed off the existing sysmon_event11 group, filtered to the Prefetch path, so it inherits the decoder work already done rather than reinventing parsing.

04

Correlate, don't just alert

A single new .pf file is not an incident — it's telemetry. The actual value comes from correlating it against process creation (Event ID 1) for the same executable name arriving from a path outside its expected baseline.

local_rules.xml — new prefetch file creation, child of the base Sysmon ruleset
<group name="sysmon,evidence_of_execution,">

  <rule id="100310" level="5">
    <if_group>sysmon_event11</if_group>
    <field name="win.eventdata.targetFilename" type="pcre2">
      (?i)\\Windows\\Prefetch\\.*\.pf$
    </field>
    <description>New Prefetch file created: $(win.eventdata.targetFilename)</description>
    <group>evidence_of_execution,prefetch_creation,</group>
  </rule>

  <!-- Escalation: same executable name, new prefetch hash, seen from
       an unexpected volume/path within the correlation window -->
  <rule id="100311" level="10" frequency="2" timeframe="120">
    <if_matched_sid>100310</if_matched_sid>
    <same_field>win.eventdata.image</same_field>
    <different_field>win.eventdata.targetFilename</different_field>
    <description>Duplicate prefetch hash for same-named executable — possible path spoofing</description>
    <mitre><id>T1036</id></mitre>
    <group>evidence_of_execution,masquerading,</group>
  </rule>

</group>

Rule 100310 is deliberately low severity — level 5. It's telemetry, not a verdict, and it will fire constantly on a normal host. Rule 100311 is where the actual "same name, different hash" pattern from earlier in this post becomes an automated detection instead of something I have to notice manually during a manual PECmd review. I've mapped it to T1036 — Masquerading, though I'd honestly still call this rule a first draft — I haven't run it against a large enough baseline yet to know its false-positive rate on legitimate multi-path tools.

Open problem I haven't solved yet

The svchost/dllhost/rundll32 exception from earlier means rule 100311 needs an exclusion list for command-line-hashed processes, or it will fire constantly on completely normal svchost activity. I'm treating this as a tuning pass for a future revision rather than shipping it broadly yet — worth saying out loud rather than pretending the first draft is production-ready.

A Real Walkthrough

Pulling this together on one of my own lab hosts, running PECmd against a live Prefetch folder:

PowerShell — checking the AppCompatCache key exists before deeper triage
PS C:\Users\nsom> Get-ChildItem -Path "C:\Windows\Prefetch" -Filter "CMD.EXE*"

Directory: C:\Windows\Prefetch

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
-a----          7/13/2026  9:14 AM          28672 CMD.EXE-087B4001.pf
-a----          7/13/2026 11:02 AM          28912 CMD.EXE-3F27A9DE.pf

Two hashes for the same binary name, roughly two hours apart. That alone isn't an incident — it's a question. Running PECmd against both and comparing the embedded volume/path metadata is what turns "two files" into either "explainable" (a portable toolkit legitimately launching cmd from a USB volume) or "not explainable" (a renamed binary sitting in a user-writable temp path). This is exactly the kind of case where rule 100311, once tuned, would have surfaced the pair automatically instead of me finding it by manually browsing the folder.

Lessons Learned

  • Absence of a prefetch file is not absence of execution. Live response tooling running on a host can evict old .pf files once the cap (128 or 1024, depending on OS version) is hit — collect prefetch early in triage, before your own tools start overwriting the evidence you came to find.
  • The "duplicate name, different hash" pattern is only meaningful once you've separated path-hashed processes from command-line-hashed ones. Skipping that step turns a good detection idea into alert noise.
  • Prefetch tells you a program ran and roughly what it touched in its first seconds — nothing about what it did after that. It's a starting thread, not the whole fabric. The next artifact in this series exists specifically to cover cases where even the prefetch file itself has been deleted.
Beyond the SOC: Where This Pattern Applies

The core idea here — a system silently logging "what ran" as a side effect of an unrelated optimization feature — shows up outside security constantly. Application performance monitoring tools infer usage patterns from cache-warming logs the same way. Fraud detection pipelines lean on side-channel signals (session replay artifacts, CDN cache-hit logs) that were never designed for fraud detection but happen to prove something occurred. The transferable skill isn't "know what Prefetch is" — it's knowing to ask, for any system you're investigating: what does this thing log as an accidental byproduct of doing its actual job? Those byproducts are usually more honest than the logs a system was deliberately designed to produce, because nobody thought to clean them up.
