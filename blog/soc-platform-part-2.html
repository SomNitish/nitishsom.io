📗 PART 2 / 6 — The Workflow Engine

Fixing the Bottleneck: Queue Mode + Auto-Scaling

In Part 1, the first version of the platform choked when a second customer doubled our alert volume. This part is the rebuild — Docker Compose with main + workers + Redis + Postgres, and a small bash daemon that scales workers up when alerts surge.

Where We Left Off

End of Part 1: I had a working N8N flow. Wazuh fires an alert → check whitelist → generate an ID → create a Jira ticket → email the customer. Five nodes. Worked beautifully for one customer.

Then we added a second customer. Alert volume roughly doubled. And the cracks were obvious within a week:

N8N was running in default mode — a single process doing everything.

  • One workflow at a time. If a slow Jira API call took 8 seconds, the next webhook waited 8 seconds.
  • A 90-second scheduled compliance report would block all alert processing for 90 seconds straight.
  • A short burst of alerts during lunch built up a queue that took the whole afternoon to drain.

The bottleneck wasn't N8N's fault. It's just how N8N runs by default. The fix is a different mode entirely.

What "Queue Mode" Actually Means

N8N has a setting called queue mode. Flip it on and N8N stops being one process doing everything — it splits into three roles, with Redis as the glue between them.

N8N MAIN
Editor UI + webhook receiver. Pushes incoming jobs onto Redis. Never executes a workflow itself.
→
REDIS (queue)
Holds pending jobs. Workers pull from here. If a worker crashes mid-job, the job comes back.
→
N8N WORKERS × N
Pull jobs from Redis, execute the workflow, write results. Stateless — add or remove on the fly.

The magic property of this setup: workers are stateless and identical. Any worker can run any workflow. You can have 1 worker or 50. You can kill one mid-execution and another picks up the job. There's no coordination layer, no leader election, no shared filesystem. Just Redis holding the queue and workers grabbing jobs off it.

Now I need to actually wire this up. Four containers, one network, one .env.

1. Postgres — and the port collision I hit

Postgres holds N8N's internal state (workflows, executions, credentials) plus our incidents table (covered in Part 3). The container is straightforward, but note this line:

▸ snippet 
ports:
  - "5433:5432"     # host 5433 -> container 5432

The host machine already had a Postgres instance running on the default port 5432 (leftover from an earlier project). Rather than fight that, I mapped this Postgres to host port 5433. One character. Saved me hours.

The healthcheck uses pg_isready rather than nc -z. nc -z only confirms the port is open — which it is even while Postgres is still loading. pg_isready actually verifies the server is accepting authenticated connections.

2. Redis — and the eviction policy that almost cost me alerts

This is the queue. Workers will pull jobs from here. The settings that matter:

▸ snippet 
command: >
  redis-server
  --requirepass ${REDIS_PASSWORD}
  --appendonly yes
  --maxmemory 256mb
  --maxmemory-policy noeviction
  • --requirepass — Always. Even on a private network. Defense in depth.
  • --appendonly yes — Persistent queue. If Redis restarts, the queue survives.
  • --maxmemory-policy noeviction — This one is critical.

The eviction policy is where I almost lost data.

Redis's default policy in some configurations is allkeys-lru — when memory fills, evict the least recently used keys. Sounds fine for a cache. Catastrophic for a queue.

If Redis hits memory pressure with that policy, it silently drops the oldest queued alerts. You'd never know. Tickets just stop appearing.

noeviction means: if memory fills, refuse new writes and throw an error. Loud failure beats silent data loss every time.

3. N8N main — the entry point

This container runs the editor UI (port 5678) and ingests webhooks from Wazuh. It also handles scheduled triggers. But — and this is the important bit — it never executes a workflow itself. Every job gets pushed to Redis, picked up by a worker.

That behavior is controlled by one setting in .env: EXECUTIONS_MODE=queue. We'll see it in the next section.

4. N8N worker — the magic word

The worker container is identical to the main container in every way except one line:

▸ snippet 
command: worker

That single word — passed as the container's command — tells N8N: "don't be the main process, be a worker. Connect to Redis, pull jobs, execute them, write results, repeat." Same image. Same env. Completely different role. It's elegant.

A few subtler details:

  • No container_name — When you have multiple replicas, Docker needs to generate names like n8nq-worker-1, n8nq-worker-2. A fixed name would crash on the second replica.
  • No ports exposed — Workers don't serve HTTP. They only talk to Redis and Postgres.
  • deploy.replicas: 2 — The boot default. The autoscaler (below) overrides this at runtime.

The .env File

One config file shared by main and all workers. Same values everywhere — no drift, no surprises.

# --- Stability ---
N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0

Most of these are self-explanatory. A few earned their place the hard way:

EXECUTIONS_MODE=queue

The master switch. Without it, queue mode does nothing and N8N falls back to single-process. Main and workers both need this — that's why they share the same .env.

N8N_CONCURRENCY_PRODUCTION_LIMIT=5

Each worker handles up to 5 workflows in parallel before it stops claiming new jobs. With 10 workers, that's 50 concurrent workflows. With 50 workers, 250.

I tested values from 1 to 20. Below 5, workers were idle waiting for I/O. Above 8, individual workflows started timing out because the worker was thrashing on memory. Five is the sweet spot for our workload (Jira API calls + Postgres writes + email sends).

N8N_ENCRYPTION_KEY

The most important value in the entire .env. N8N encrypts every credential (Jira tokens, OAuth refresh tokens, DB passwords stored inside workflow nodes) with this key.

Lose this key → every credential is unrecoverable. You'll re-enter every API token, every OAuth handshake, every DB connection. The first day I had queue mode running, I regenerated this key without backing up the old one. Spent the rest of a Saturday rebuilding ~30 credentials. Don't do that.

OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true

When you click "Execute Workflow" in the N8N editor, by default that runs in the main process — which blocks the UI and ignores your shiny new workers. This setting routes manual runs through workers too. Always set it.

EXECUTIONS_DATA_MAX_AGE=168

168 hours = 7 days. N8N saves every execution's full payload by default. At 5,000 executions/month, that database grows fast. Seven days is enough for debugging without blowing up disk usage.

N8N_PARTIAL_EXECUTION_VERSION_DEFAULT=0

N8N has two execution engines. Version 1 is newer and faster — but has edge cases when sub-workflows pass data between each other, which our SOAR workflows do constantly. Pinning to 0 saved us from a class of weird intermittent bugs.

The Next Problem: Bursts

Two workers at 5-concurrency each = 10 parallel workflows. That handles the steady-state load with room to spare. But "steady-state" is a fiction.

What actually happens, monthly: someone pushes a misconfigured firewall rule, or an old user account gets re-enabled, or a vulnerability scanner kicks off — and we get 800 alerts in a 4-minute window. At 10 parallel workflows averaging 2 seconds each, that takes about 3 minutes to drain. Real-time alerts arriving 3 minutes late are not real-time.

The fix: detect the burst, add workers, let them drain the queue, then scale back down when things are quiet.

I could have reached for Kubernetes HPA or some cloud autoscaling gizmo. For a single-host setup, that's massive overkill. A 60-line bash script does the job perfectly.

The Autoscaler

This script runs in a loop. Every 5 seconds it asks Redis "how deep is the queue?" and counts how many worker containers are running. If the queue is busy and we're below max, add a worker. If the queue is quiet and we're above min, drop one. A short cooldown between actions prevents thrashing.

📄 /opt/security-automation/autoscale.sh 
#!/bin/bash
# -----------------------------------------------------------
# N8N Worker Auto-Scaler
# Polls Redis queue depth and scales workers between MIN/MAX.
# -----------------------------------------------------------

MIN_WORKERS=10
MAX_WORKERS=50
SCALE_UP_THRESHOLD=1        # queued jobs >= this -> scale up
SCALE_DOWN_THRESHOLD=2      # queued jobs <= this -> scale down
COOLDOWN=10                 # seconds between scale actions
COMPOSE_DIR="/opt/security-automation"

REDIS_PASS=$(grep ^REDIS_PASSWORD "$COMPOSE_DIR/.env" | cut -d= -f2)
LAST_SCALE=0

get_queue_depth() {
    local wait active
    wait=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN bull:jobs:wait 2>/dev/null | tr -cd '0-9')
    active=$(docker exec n8nq-redis redis-cli -a "$REDIS_PASS" \
        --no-auth-warning LLEN bull:jobs:active 2>/dev/null | tr -cd '0-9')
    echo $(( ${wait:-0} + ${active:-0} ))
}

get_current_workers() {
    local count
    count=$(docker compose -f "$COMPOSE_DIR/docker-compose.yml" \
        ps --format json 2>/dev/null | grep -c "n8nq-worker" || echo 2)
    echo "${count:-2}"
}

scale_workers() {
    local target=$1
    local now current elapsed
    now=$(date +%s)
    elapsed=$((now - LAST_SCALE))

    [ "$elapsed" -lt "$COOLDOWN" ] && return     # still cooling down

    current=$(get_current_workers)
    [ "$current" -eq "$target" ] && return       # already at target

    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Scaling workers: $current -> $target"
    cd "$COMPOSE_DIR"
    docker compose up -d --scale n8nq-worker="$target" --no-recreate 2>/dev/null
    LAST_SCALE=$now
}

echo "[$(date '+%Y-%m-%d %H:%M:%S')] Auto-scaler started | Min=$MIN_WORKERS Max=$MAX_WORKERS"

while true; do
    QUEUE=$(get_queue_depth)
    CURRENT=$(get_current_workers)
    echo "[$(date '+%H:%M:%S')] Queue: $QUEUE | Workers: $CURRENT"

    if [ "$QUEUE" -ge "$SCALE_UP_THRESHOLD" ] && [ "$CURRENT" -lt "$MAX_WORKERS" ]; then
        NEW=$((CURRENT + 1))
        [ "$NEW" -gt "$MAX_WORKERS" ] && NEW=$MAX_WORKERS
        scale_workers "$NEW"
    elif [ "$QUEUE" -le "$SCALE_DOWN_THRESHOLD" ] && [ "$CURRENT" -gt "$MIN_WORKERS" ]; then
        NEW=$((CURRENT - 1))
        [ "$NEW" -lt "$MIN_WORKERS" ] && NEW=$MIN_WORKERS
        scale_workers "$NEW"
    fi

    sleep 5
done

The decisions in this script that aren't obvious

Why MIN_WORKERS=10 and not 2? Baseline capacity. With 2 workers, a sudden burst means jobs queue up for ~30 seconds before the scaler reacts and a new worker becomes ready. With 10 workers at 5-concurrency, the system absorbs a 50-alert burst with zero queue depth. Eight extra workers cost about 200 MB of RAM at idle. Trivial price for never having a queue backlog under normal load.

Why scale up at queue depth = 1? Because by the time the queue is at 5, you're already late. The whole point is to react before the user notices.

Why scale down at queue ≤ 2 (not 0)? To avoid flapping. If we scaled down at 0, then a single arriving job would trigger scale-up. The system would oscillate. Threshold of 2 gives breathing room.

The --no-recreate flag is critical. Without it, every scale call would tear down and recreate all worker containers. With it, only the delta is applied — going from 10 to 11 starts one new container and leaves the existing 10 alone. This saved me from a class of subtle bugs where in-flight jobs got killed during scaling.

Asymmetric cooldown — what I'd improve

The current cooldown is 10 seconds for both directions. In practice, scaling up should be faster than scaling down. A 60-second down-cooldown would absorb transient quiet periods and avoid the system thrashing during borderline load. I haven't shipped that change yet — it's on my list.

Survive Reboots: systemd

The script can't run in a screen session. It needs to start on boot and restart on failure. That's exactly what systemd does.

📄 /etc/systemd/system/n8n-autoscale.service 
[Unit]
Description=N8N Worker Auto-Scaler
After=docker.service
Requires=docker.service

[Service]
Type=simple
WorkingDirectory=/opt/security-automation
ExecStart=/bin/bash /opt/security-automation/autoscale.sh
Restart=always
RestartSec=5
StandardOutput=append:/opt/security-automation/autoscale.log
StandardError=append:/opt/security-automation/autoscale.log
User=root
LimitNOFILE=4096

[Install]
WantedBy=multi-user.target
▸ enable and start 
sudo chmod +x /opt/security-automation/autoscale.sh
sudo systemctl daemon-reload
sudo systemctl enable n8n-autoscale.service
sudo systemctl start n8n-autoscale.service

Why Bash, Not Python

I write Python all day. I still chose bash for this. The reasons are short:

  • Zero dependencies. Bash and Docker are already on the host. No pip install, no virtualenv, no Python version to track.
  • Sixty lines, end-to-end readable. A junior team member can understand the whole script in five minutes.
  • Composes naturally with shell tools. docker compose, redis-cli, grep, cut, date — they're all already CLI. Wrapping them in Python adds friction with no payoff.

If the script grew to 200 lines or needed structured config, I'd port it. At 60 lines, Python would be over-engineering.

Beyond the SOC: Where This Pattern Applies

Same engine pattern, different use cases

  • Any bursty webhook → background-job pipeline. Stripe / GitHub / Slack webhooks where you want to take work off the request thread and process in parallel.
  • ETL with unpredictable load. Scheduled batch + ad-hoc one-off jobs sharing the same worker pool.
  • Notification fan-out. One event triggers N downstream sends (email, SMS, push) — let workers parallelize them.
  • Document/media processing. Image resize, PDF generation, video transcoding — variable runtime per job, perfect fit for stateless workers.
  • Cost-sensitive backend automation. Single-host Docker host + autoscaling bash is dramatically cheaper than Kubernetes for low-tens-of-RPS workloads.

The point of queue mode isn't "more N8N." It's decoupling the act of receiving work from the act of doing work. Once those are separate processes, scaling becomes a script.

What's Next

The engine is humming. Workers come and go automatically. Webhooks never queue for more than a second. The platform feels fast.

But there's a problem I haven't solved yet. When 20 workers are inserting incidents into Postgres at the same millisecond, some of them generate the same incident ID. Two different alerts, same ID, customer confusion. The race condition from Part 1's first version is still there — it's just hitting more often now that we have more parallel workers.

The fix isn't in application code. It's in the database itself. That's Part 3.
