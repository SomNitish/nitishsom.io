📔 PART 5 / 6 — The Dashboard

No Framework, No Build Step, No Excuses

Part 4 gave us a clean JSON API. Analysts don't drink JSON. This part is the screen they actually use — built in vanilla JavaScript, with a polling refresh loop, a queue-stuck detection overlay that drafts an emergency email for you, a notification bell, and Chart.js for the queue history.

Where We Left Off

End of Part 4: the backend is solid. About 30 REST endpoints, all auth-protected, all permission-checked, all audit-logged. Open /docs and there's a Swagger UI to poke around. Useful for me, useless for the team.

The team needs a screen. Open tickets visible at a glance, queue depth refreshing live, the bell icon turning red when something happens, a big red overlay that screams when workers stop processing. That's Part 5.

Why I Didn't Reach for React

This is the part where most blog posts would justify their framework choice with benchmark tables and TypeScript-vs-PropTypes arguments. I'll keep it short. I picked vanilla JavaScript on purpose, and here's why:

  • Zero build step. I can SSH into the server, edit index.html with vim, save, refresh the browser. For a tool that needs hotfixes at midnight, this matters more than people realize. No npm install on the production host. No webpack config. No "wait for the bundle."
  • One file per page. Each dashboard page is a single HTML file containing its own CSS, JavaScript, and markup. A junior analyst can open queue.html, read it top to bottom, and understand the entire page in 15 minutes.
  • No dependency hell. When npm audit flags a transitive dependency three months from now, it doesn't affect me. There are no transitive dependencies. The only external library is Chart.js, loaded from a CDN with SRI hashes.
  • Speed. The whole dashboard loads in <200ms. No virtual DOM, no hydration, no bundle to parse. Fetch, render, done.

The tradeoff is real — when I need something complex (the queue-stuck overlay, the column resizer, the notification dropdown), I write it myself. That's maybe 600 extra lines of JavaScript across the entire dashboard. In exchange, I never debug a build pipeline.

The Polling Loop

Every page is structured the same way: one async function that fetches all the data the page needs in parallel, then renders it. A setInterval calls that function on a fixed cadence. That's it — no observables, no reactive state, no virtual DOM diffing.

📄 dashboard/index.html — the refresh loop 
const REFRESH_MS = 5000;
let refreshTimer = null;
let currentTab = 'CLIENT-A';

async function api(path, opts={}) {
  const token = localStorage.getItem('token');
  const res = await fetch(path, {
    ...opts,
    headers: { 'Authorization': `Bearer ${token}`, ...(opts.headers||{}) },
  });
  if (res.status === 401) {                 // session died -> back to login
    localStorage.removeItem('token');
    location.href = '/login.html';
    return;
  }
  if (!res.ok) throw new Error(await res.text());
  return res.json();
}

async function refreshAll() {
  try {
    // Three calls fire in parallel
    const [incidents, queue, notifs] = await Promise.all([
      api(`/api/incidents?customer=${currentTab}&limit=200`),
      api('/api/queue/status'),
      api('/api/notifications/unread_count'),
    ]);

    renderIncidents(incidents);
    renderQueueIndicator(queue);
    renderNotifBadge(notifs.count);
    checkQueueHealth(queue);                 // the stuck-detector lives here
  } catch (err) {
    console.error('refresh failed:', err);
  }
}

function startRefreshLoop() {
  refreshAll();                              // immediate first call
  refreshTimer = setInterval(refreshAll, REFRESH_MS);
}

document.addEventListener('DOMContentLoaded', startRefreshLoop);

Three things to notice:

  • Three endpoints, one Promise.all. Calls fire in parallel, total wall time is the slowest call (usually 60-80ms for the incidents list). Sequential calls would triple that latency.
  • 401 → redirect to login. Centralized in the api() wrapper so every page doesn't have to handle expired sessions individually.
  • The checkQueueHealth() call after rendering. Every poll cycle also checks for queue-stuck conditions. That's the overlay we're about to build.

The Queue-Stuck Detection Overlay

This is the feature that's saved me twice now. Here's the problem it solves.

On a Saturday morning at 6 AM, all our N8N workers crashed simultaneously because of a kernel-level memory issue on the host. The autoscaler tried to start new ones, Docker refused, and the queue sat at 347 alerts waiting for workers that would never come back. Wazuh kept piling on more.

I didn't find out until 9:30 AM when an analyst messaged me. That's three and a half hours of alerts not getting tickets. Customers were calling.

After that incident I built the overlay. Every polling cycle, the dashboard checks: are jobs waiting in the queue, but no jobs being actively processed, for more than 2 minutes? If yes, the dashboard knows something is wrong, and it lights up the screen so loud that nobody can miss it.

The detection logic

The queue status endpoint returns the same numbers the autoscaler uses (waiting, active, worker count), plus the timestamp of the last successful execution. The dashboard checks all three on every refresh:

📄 dashboard/index.html — checkQueueHealth() 
const STUCK_THRESHOLD_MS = 120000;   // 2 minutes of no progress
let dismissUntil = 0;

function checkQueueHealth(q) {
  // q = { waiting, active, workers, last_executed_at_ms, earliest_queued }
  if (Date.now() < dismissUntil) return hideStuckOverlay();

  const hasBacklog   = q.waiting > 0;
  const nothingActive = q.active === 0;
  const noProgress   = q.last_executed_at_ms &&
                       (Date.now() - q.last_executed_at_ms) > STUCK_THRESHOLD_MS;

  if (hasBacklog && nothingActive && noProgress) {
    showStuckOverlay(q);
  } else {
    hideStuckOverlay();
  }
}

function showStuckOverlay(q) {
  const overlay = document.getElementById('queue-stuck-overlay');
  document.getElementById('stuck-waiting').textContent = q.waiting;
  document.getElementById('stuck-workers').textContent = `${q.workers} / 10`;
  document.getElementById('stuck-since')
          .textContent = formatAgo(q.last_executed_at_ms);
  document.getElementById('stuck-earliest').textContent = q.earliest_queued || '(none)';

  // Pre-build the emergency email so the analyst can copy and paste
  document.getElementById('stuck-email').textContent = buildStuckEmail(q);
  overlay.classList.add('visible');
}

function hideStuckOverlay() {
  document.getElementById('queue-stuck-overlay').classList.remove('visible');
}

Three conditions all have to be true. We don't want a one-second blip during a worker restart to fire this. We want it to fire when the queue is actually stuck.

The pre-drafted emergency email

This is the detail that earned the feature its place. When the overlay appears, the email is already written. The analyst clicks "Copy emergency email," pastes it into Outlook, hits send. The cognitive load of "what do I say, who do I send it to, what data do I include" — all gone.

📄 dashboard/index.html — buildStuckEmail() 
function buildStuckEmail(q) {
  const now = new Date().toISOString().replace('T', ' ').slice(0, 19);
  return [
    `Subject: [SOC PLATFORM] Queue stuck — workers not processing`,
    `To: oncall-engineer@<company>.com`,
    ``,
    `The SOC platform queue is stuck.`,
    ``,
    `Detected at: ${now} IST`,
    `Waiting jobs: ${q.waiting}`,
    `Active jobs: ${q.active}`,
    `Workers running: ${q.workers} / 10`,
    `Last execution: ${formatAgo(q.last_executed_at_ms)} ago`,
    `Earliest queued alert: ${q.earliest_queued || 'unknown'}`,
    ``,
    `Suggested checks:`,
    `  1. docker compose ps   (are containers actually up?)`,
    `  2. docker compose logs n8nq-worker --tail 200`,
    `  3. docker exec n8nq-redis redis-cli -a $REDIS_PASS LLEN bull:jobs:wait`,
    `  4. systemctl status n8n-autoscale`,
    ``,
    `Auto-generated from the SOC Platform dashboard. No action taken yet.`,
  ].join('\n');
}

document.getElementById('stuck-copy-btn').addEventListener('click', async () => {
  const text = document.getElementById('stuck-email').textContent;
  await navigator.clipboard.writeText(text);
  toast('Emergency email copied to clipboard');
});

The "Dismiss for 5 minutes" button is intentional. If I'm actively debugging the issue (running docker compose restart, watching logs), I don't want the overlay obscuring the dashboard. Five minutes of silence, then it comes back if the queue is still stuck.

Why this works: the dashboard already polls every 5 seconds for its normal data. Adding a queue-health check to that same loop costs us literally nothing. The detection logic is 10 lines of JavaScript. The pre-drafted email is the difference between "I see something is wrong" and "I have already started the response."

The Saturday-morning incident, on a re-run with the overlay in place, would have alerted the analyst within 2 minutes instead of 3.5 hours.

The Notification Bell

The bell icon in the header turns red when there are unread notifications. Click it, get a dropdown. The data comes from the /api/notifications endpoint we built in Part 4.

📄 dashboard/index.html — notification bell 
function renderNotifBadge(unread) {
  const bell  = document.getElementById('notif-bell');
  const badge = document.getElementById('notif-badge');
  if (unread > 0) {
    badge.textContent = unread > 99 ? '99+' : unread;
    badge.classList.add('visible');
    bell.classList.add('has-unread');
  } else {
    badge.classList.remove('visible');
    bell.classList.remove('has-unread');
  }
}

async function openNotifPanel() {
  const panel = document.getElementById('notif-panel');
  const list  = document.getElementById('notif-list');
  panel.classList.add('visible');
  list.innerHTML = '<div class="loading">Loading…</div>';

  const items = await api('/api/notifications?limit=20');
  list.innerHTML = items.length
    ? items.map(renderNotifItem).join('')
    : '<div class="empty">No notifications</div>';

  // Mark them as read on the server
  await api('/api/notifications/mark_all_read', { method: 'POST' });
  renderNotifBadge(0);
}

function renderNotifItem(n) {
  const ago = formatAgo(new Date(n.created_at).getTime());
  const link = n.incident_id
    ? `<a href="/incident.html?seq=${n.incident_seq}">${n.incident_id}</a>`
    : '';
  return `<div class="notif-item ${n.read ? '' : 'unread'}">
    <div class="t">${escapeHtml(n.title)}</div>
    <div class="m">${escapeHtml(n.message)}</div>
    <div class="x">${link} · ${ago} ago</div>
  </div>`;
}

Notice escapeHtml() wrapping the user-provided strings. The notification message can contain user input (a comment body, an assignee name) and it goes straight into innerHTML. Without escaping, you've got an XSS vector. With it, you've got a notification.

▸ the only HTML-escape helper I use 
function escapeHtml(s) {
  if (s == null) return '';
  return String(s)
    .replaceAll('&', '&amp;')
    .replaceAll('<', '&lt;')
    .replaceAll('>', '&gt;')
    .replaceAll('"', '&quot;')
    .replaceAll("'", '&#039;');
}

Twelve lines. Every string from the API goes through this before it hits the DOM. No template framework. No exotic library. Just five replacements.

Chart.js for the Queue History

The queue snapshot writer thread from Part 4 has been writing a row to queue_snapshots every 30 seconds. A separate page, queue.html, charts those snapshots over the last 24 hours.

I picked Chart.js because it's the most popular library that doesn't require a build step. Single script tag, draw a chart, done. No npm, no React-Chart-Wrapper-Bindings package, no compilation step that breaks every other release.

📄 dashboard/queue.html — chart setup

        integrity="sha384-..." crossorigin="anonymous"></script>

<canvas id="queue-chart" height="120"></canvas>

<script>
let chart = null;

async function loadQueueHistory() {
  const data = await api('/api/queue/history?hours=24');
  // data = [{ captured_at, waiting, active, workers }, ...]

  const labels  = data.map(p => new Date(p.captured_at)
                                .toLocaleTimeString('en-IN',
                                  { hour: '2-digit', minute: '2-digit' }));
  const waiting = data.map(p => p.waiting);
  const active  = data.map(p => p.active);
  const workers = data.map(p => p.workers);

  if (chart) chart.destroy();
  chart = new Chart(document.getElementById('queue-chart'), {
    type: 'line',
    data: { labels, datasets: [
      { label: 'Waiting', data: waiting, borderColor: '#ef4444',
        tension: .25, fill: false },
      { label: 'Active',  data: active,  borderColor: '#3b82f6',
        tension: .25, fill: false },
      { label: 'Workers', data: workers, borderColor: '#8b5cf6',
        tension: .25, fill: false, borderDash: [4,4] },
    ]},
    options: {
      responsive: true, maintainAspectRatio: false,
      interaction: { mode: 'index', intersect: false },
      scales: { y: { beginAtZero: true } },
      plugins: { legend: { position: 'top' } },
    },
  });
}

loadQueueHistory();
setInterval(loadQueueHistory, 30000);   // refresh every 30s
</script>

The chart shows three lines. The waiting count usually hugs zero. The active count tracks worker capacity. The workers line (dashed) shows the autoscaler at work — flat at 10 most of the time, jumping during bursts.

What this chart turned out to be useful for: incident postmortems. After any production hiccup, I can pull up queue.html, look at the last 24 hours, and see exactly when the platform started struggling. The waiting line spikes, the workers line ramps, then both come back down. Visual storytelling.

Smaller Details That Earned Their Keep

Column resizing on the incidents table

An analyst asked for this in week 2 — the "Alert Name" column was too narrow and important text got truncated. Rather than guessing better defaults, I added drag-resize handles to the column borders. Widths persist in localStorage.

📄 dashboard/index.html — column resize 
function attachColumnResize(table) {
  table.querySelectorAll('th').forEach((th, idx) => {
    const handle = document.createElement('div');
    handle.className = 'col-resize';
    th.appendChild(handle);

    handle.addEventListener('mousedown', e => {
      e.preventDefault();
      const startX = e.pageX;
      const startW = th.offsetWidth;

      function onMove(ev) {
        const w = Math.max(50, startW + (ev.pageX - startX));
        th.style.width = w + 'px';
      }
      function onUp() {
        document.removeEventListener('mousemove', onMove);
        document.removeEventListener('mouseup', onUp);
        // Persist
        const widths = JSON.parse(localStorage.getItem('colW') || '{}');
        widths[idx] = th.offsetWidth;
        localStorage.setItem('colW', JSON.stringify(widths));
      }
      document.addEventListener('mousemove', onMove);
      document.addEventListener('mouseup', onUp);
    });
  });

  // Restore on load
  const widths = JSON.parse(localStorage.getItem('colW') || '{}');
  table.querySelectorAll('th').forEach((th, idx) => {
    if (widths[idx]) th.style.width = widths[idx] + 'px';
  });
}

Thirty lines. The analyst's complaint was answered in the next deploy. That's the kind of agility a no-build-step frontend gives you.

Theme switcher (light / dark / auto)

Three CSS variable sets, a button in the header, persistence in localStorage. "Auto" follows the OS preference via prefers-color-scheme. Not exciting, but the SOC TV monitor account uses dark, my own account uses light, and that 15-line snippet keeps both happy.

The Login Page

One form, one POST. On success, save the token to localStorage and redirect. The whole file is ~80 lines including styles.

📄 dashboard/login.html — login.js 
document.getElementById('login-form').addEventListener('submit', async (e) => {
  e.preventDefault();
  const username = document.getElementById('username').value.trim();
  const password = document.getElementById('password').value;
  const errBox   = document.getElementById('error');
  errBox.textContent = '';

  try {
    const res = await fetch('/api/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ username, password }),
    });
    if (!res.ok) {
      const err = await res.json().catch(() => ({}));
      errBox.textContent = err.detail || 'Login failed';
      return;
    }
    const data = await res.json();
    localStorage.setItem('token',     data.token);
    localStorage.setItem('user_name', data.name);
    localStorage.setItem('user_role', data.role);
    location.href = '/index.html';
  } catch (err) {
    errBox.textContent = 'Network error: ' + err.message;
  }
});

localStorage for the token is a deliberate choice over cookies. The dashboard and the API live on the same origin, so I don't get the cross-site benefit of HttpOnly cookies. localStorage is simpler, plays nicely with the api() wrapper, and the only risk it adds (script-based exfiltration) is mitigated by escaping every string that hits the DOM.

Beyond the SOC: Where This Pattern Applies

Same frontend approach — different domains

  • Internal admin panels for small teams. Whenever the user count is in the dozens, not the millions, vanilla JS + a single HTML file per page is dramatically easier to maintain than a React app.
  • Operations dashboards. Anywhere you'd otherwise reach for Grafana but need custom interactions (action buttons, drill-down, modals), a polling-loop dashboard backed by your own API gives you full control.
  • Real-time monitoring screens. The polling-loop + alarm-overlay pattern works for any system where "something needs human attention right now" is a recurring scenario.
  • Customer-facing status pages for internal tools where you want the same kind of live updates without the operational cost of WebSockets.
  • Demo / showcase apps. When you want to share a tool that "just works" without asking the recipient to npm install, this is the only path.

Vanilla JS isn't a statement against React. It's a statement that the cheapest tool that does the job is usually the right one.
