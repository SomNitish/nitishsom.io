Linux Data Acquisition: A Forensic Analyst's Guide

Understanding volatile and non-volatile data collection techniques for incident response and digital forensics on Linux systems

The Wake-Up Call

It was 2 AM when the alert came through. A critical production server was exhibiting suspicious behavior — unusual network connections, unexplained CPU spikes, and processes that shouldn't be running. As I SSH'd into the system, my cursor blinking in the dark terminal, I realized something crucial: the moment I started investigating, I was destroying evidence.

Every command I ran, every directory I listed, every log I opened was overwriting timestamps, modifying access records, and altering the very data I needed to preserve. That night taught me a fundamental truth about incident response on Linux: data acquisition isn't just about copying files — it's about understanding what data exists, where it lives, and how to capture it without contamination.

While working with data collection on Linux systems over the past two years, I've realized that data acquisition is more than a technical checkbox. It's the foundation of every security investigation, system troubleshooting session, and performance analysis we conduct. Let me share what I've learned.

Why Linux Data Acquisition Matters

Linux data acquisition is critical for three core reasons:

1. Early Threat Detection
Advanced persistent threats (APTs) and sophisticated malware don't announce themselves. They hide in memory, masquerade as legitimate processes, and clean up their traces. By the time traditional monitoring catches them, the damage is done. Proper data acquisition captures the evidence while it still exists.

2. Root Cause Analysis
System crashes and performance issues can't be diagnosed from guesswork. You need the actual state of the system — what was running, what resources were consumed, what errors were logged — at the moment the problem occurred.

3. Legal Defensibility
In security incidents that escalate to legal proceedings, the integrity of your evidence determines whether it's admissible. Proper acquisition techniques with documented chain of custody and hash verification make the difference between evidence that holds up in court and evidence that gets dismissed.

The Two Faces of Data: Volatile vs Non-Volatile

When I first started in SOC work, I thought data collection meant copying files to an external drive. I was missing half the picture.

Volatile Data: Gone in 60 Seconds

Volatile data lives in RAM, CPU caches, and active system processes. The moment you power down the system, it's gone forever. This includes:

  • Running processes and their memory space — What was actually executing when the incident occurred
  • Active network connections — Who was the system talking to, and what data was being transmitted
  • Loaded kernel modules — Including potential rootkits hiding in memory
  • Clipboard contents and encryption keys — Evidence that exists only while the system is live

The Golden Rule: Volatile data must be collected while the system is still running. You get one shot at this. Once the power goes off, it's irretrievable.

Non-Volatile Data: The Persistent Evidence

Non-volatile data survives power loss. It lives on hard drives, SSDs, USB devices, and other persistent storage. This includes:

  • File systems and user data — Documents, downloads, application files
  • System configurations — Everything under /etc/ that defines how the system behaves
  • Log files — Historical records of system events and user actions
  • Deleted files — Data marked for deletion but still recoverable from unallocated disk space

Non-volatile data is typically collected while the system is powered down using forensic boot media. This prevents the operating system from modifying timestamps and metadata during the acquisition process.

Live Data Acquisition: Collecting Evidence on a Running System

Live acquisition is essential when you can't afford to power down the system — either because it's a critical production server or because you need to capture volatile data before it disappears.

RAM Capture: The Memory Dump

System RAM is a gold mine of forensic evidence. Here's what you can find in a memory dump:

  • Decrypted encryption keys and passwords
  • Malware that exists only in memory (fileless malware)
  • Command history and user activity
  • Network connections and their associated processes

Two approaches to memory capture:

1. Hardware-Based (Physical Memory Capture)
Creating a direct copy from memory chips using specialized hardware. Most accurate but requires physical access and specialized equipment.

2. Software-Based (Logical Memory Capture)
Using tools like memdump or LiME (Linux Memory Extractor) to dump RAM from within the OS. Easier to deploy but may miss kernel-protected memory regions.

Network Traffic Monitoring

Capturing network activity reveals what the compromised system is communicating with:

# Capture all traffic on interface eth0
tcpdump -i eth0 -w /evidence/network_capture.pcap

# Monitor specific suspicious IP
tcpdump -i eth0 host 192.168.1.100 -w /evidence/suspicious_host.pcap

Critical Configuration Files to Preserve

These files contain system state and user activity that may change on reboot:

File What It Contains Forensic Value
/etc/passwd User account list Detect unauthorized accounts created by attackers
/etc/shadow Encrypted passwords Identify weak passwords or unauthorized changes
/etc/ssh/sshd_config SSH server configuration Find backdoor SSH keys or config changes
/var/log/auth.log Authentication attempts Track successful and failed login attempts
/var/log/syslog System-wide events Timeline of system activity
~/.bash_history Command history per user See exactly what commands the attacker ran

Disk Imaging: Creating Forensically Sound Copies

When you need to preserve an entire disk for analysis, bit-level imaging is the gold standard. This creates an exact copy of every sector on the disk — including deleted files and unallocated space.

The dd Command: Simple But Powerful

The dd command is the Swiss Army knife of disk imaging. Despite its simplicity, it's one of the most trusted tools in digital forensics.

# First, identify the disk structure
lsblk -tf

# Image an entire disk
dd if=/dev/sda of=/evidence/disk_image.raw bs=4M

# Image a single partition
dd if=/dev/sda3 of=/evidence/partition_image.raw bs=4M

# Verify the image with hash
md5sum /dev/sda > /evidence/original_hash.txt
md5sum /evidence/disk_image.raw > /evidence/image_hash.txt

Key Parameters:

  • if= (input file) — The source disk/partition
  • of= (output file) — The destination image file
  • bs= (block size) — How much data to read/write at once (4M is optimal for modern hardware)

⚠️ Critical Warning: The dd command is extremely powerful and unforgiving. Get the if= and of= parameters backwards, and you'll overwrite your evidence disk with zeros. Always double-check your syntax before pressing Enter.

dcfldd: dd with Forensic Superpowers

The dcfldd command is an enhanced version of dd designed specifically for digital forensics. It adds critical features:

  • Real-time hashing — Calculate MD5/SHA256 during the copy process
  • Progress indicator — See how long large copies will take
  • Multiple outputs — Write to several destinations simultaneously
  • Built-in verification — Confirm the copy matches the source
# Image disk with hash verification
dcfldd if=/dev/sda of=/evidence/disk_image.raw \
       hash=sha256 hashwindow=1G \
       hashlog=/evidence/hash.log \
       bs=4M

The main drawback? It's not installed by default on most systems, which can be problematic during live acquisitions where installing new tools might alter evidence.

Clonezilla: GUI-Driven Disk Cloning

For those who prefer a graphical interface, Clonezilla provides enterprise-grade disk cloning capabilities:

  • Boots from live USB/CD without touching the target system
  • Supports multiple file systems (ext4, NTFS, FAT32, etc.)
  • Built-in compression to save storage space
  • Hash verification (MD5, SHA256) for integrity checking

Clonezilla is particularly useful for creating forensic images of systems you need to return to service quickly — image the disk, restore it to new hardware, and continue analysis offline.

Data Integrity: Proving Your Evidence Is Untainted

In digital forensics, maintaining data integrity isn't optional — it's the difference between evidence that's admissible in court and evidence that gets thrown out.

Hash Verification

Every forensic image should be accompanied by hash values:

# Generate hash before acquisition
sha256sum /dev/sda > original_hash.txt

# Generate hash after acquisition
sha256sum disk_image.raw > image_hash.txt

# Verify they match
diff original_hash.txt image_hash.txt

If the hashes match, you have cryptographic proof that the image is an exact copy of the original.

Write-Blockers: Hardware Protection

Hardware write-blockers prevent any writes to the evidence disk during acquisition. This ensures that your investigation tools can't accidentally modify timestamps or metadata on the original evidence.

Legal and Ethical Considerations

Data acquisition carries serious legal and ethical responsibilities:

Authorization: Only authorized personnel should perform data acquisition. Unauthorized access, even for "security purposes," can have severe legal consequences.

Privacy: Collected data may contain personal or sensitive information. Handle it according to data protection regulations (GDPR, CCPA, etc.).

Chain of Custody: Document every person who handles the evidence, when they had access, and what they did with it. Gaps in chain of custody can invalidate forensic evidence.

Transparency: In corporate environments, data collection should be disclosed to users. Surprise forensic acquisitions can create legal and HR issues.

Lessons Learned from the Field

After two years of SOC work and dozens of incident response cases, here's what I wish someone had told me on day one:

  1. Order of volatility matters. Collect the most volatile data first (RAM, network connections) before moving to persistent storage.
  2. Never work directly on original evidence. Always work on copies. Your analysis tools will modify timestamps and access records.
  3. Document everything. Timestamps, commands executed, hash values, who was present — all of it. Your notes may end up in court.
  4. Test your tools before the incident. Don't learn how dd works during a live breach. Practice on test systems.
  5. Automate where possible. Write scripts for common acquisition tasks to reduce human error under pressure.

Conclusion: Data Acquisition as a Craft

Data acquisition on Linux is both an art and a science. The technical commands are straightforward — dd, tcpdump, memdump — but knowing when to use them, in what order, and how to maintain evidence integrity requires experience and judgment.

What started as a confusing 2 AM incident has become a fundamental part of my security operations toolkit. Every log file I preserve, every memory dump I analyze, and every disk image I create contributes to a larger understanding of how systems fail, how attackers operate, and how defenders can respond effectively.

The next time you SSH into a suspicious system, remember: you're not just an analyst investigating a problem. You're a digital archaeologist, carefully excavating evidence that may be the only record of what really happened.

Collect it carefully. Document it thoroughly. Preserve it properly.

The truth is in the data.
